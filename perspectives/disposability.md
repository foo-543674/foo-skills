# Disposability

捨てやすさ (影響範囲のコントロール可能性) に関する品質レンズ。philosophy/core-principles.md の「捨てやすさ」をレビュー観点として具体化したもの。

## この観点で重視すること

- **シグネチャ露出の防止**: ドメイン層/アプリケーション層の public シグネチャに外部ライブラリ固有の型を出さない。今の 1 箇所だけでなく、将来の呼び出し元すべてが巻き込まれる
- **使用ガイドラインの存在と遵守**: 主要な外部ライブラリに「どこから import 可、ラッパー必須か」が明文化されていること
- **隔離戦略の妥当性**: 対象の性質 (技術基盤/UI系/IO/大型依存) に応じた適切な戦略 (A/B/C/D) が採られていること
- **影響範囲の実測**: 「明日この技術がなくなったら何箇所修正が必要か」が予測可能であること

## 判断基準

- ドメイン層/アプリケーション層に外部型が露出 → **Critical**
- ガイドライン不在 or ガイドライン違反 → **Important**
- 隔離戦略のミスマッチ (UI を無理にラッパー化等) → **Important**
- 影響範囲が散らばっている (lint 強制なし) → **Minor**
- 言語標準への不要なラッパー、薄い素通しラッパー → **Minor** (過剰設計)

## アンチパターン

- **Signature Leak**: Express の Request 型、Prisma の生成型がアプリケーション層に露出
- **Scattered Direct Import**: ガイドラインなしで 47 ファイルから dayjs を直接 import
- **Bypassed Wrapper**: ラッパーがあるのに直接 import で迂回
- **Strategy Mismatch**: UI ライブラリを無理にラッパー化 (戦略 B が適切なのに A)
- **Framework-Coupled Domain**: ドメイン層がフレームワークのデコレータ/DI コンテナに依存
- **Premature Disposability**: 言語標準への不要な抽象化

## 品質チェックポイント

- [ ] ドメイン層/アプリケーション層の public シグネチャに外部ライブラリ固有の型が出ていないか
- [ ] 主要な外部ライブラリに使用ガイドラインが明文化されているか
- [ ] ガイドラインに違反した直接 import がないか
- [ ] 各ライブラリに性質に合った隔離戦略 (A/B/C/D) が採られているか
- [ ] 「明日この技術がなくなったら何箇所修正が必要か」に答えられるか
- [ ] 言語標準や安定した依存への過剰なラッパーがないか
