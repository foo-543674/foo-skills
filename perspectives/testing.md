# Testing

テストに関する品質レンズ。

## この観点で重視すること

- **テスト名は実行可能な仕様**: 動作の説明であって実装の詳細ではない。テスト名だけ読んで仕様が分かること
- **境界値の網羅**: happy path だけでなく、0, 1, N-1, N, N+1 の境界と異常系を必ずテストする
- **テスト独立性**: 実行順序に依存しない、共有状態を持たない。各テストが自分のデータを作る
- **モックは外部依存のみ**: 純粋なロジックは実コードでテスト。内部実装のモックはテストの価値を下げる
- **テスト価値の見極め**: すべてにテストを書くのではなく、テストが最も価値を発揮する場所を選ぶ

## 判断基準

- テストの書きにくさは設計の問題のサイン。モックが大量に必要 → 設計を直す (逃げ E の禁止)
- テストピラミッド: 多くの unit → 適度な integration → 少数の E2E。逆ピラミッドは危険信号
- ユースケース本体の単体テストは書かない (ロジックがなくモック検証になるだけ)。ドメインロジックと統合テストに集中する

## アンチパターン

- **Vague Test Names**: `test1`, `should work`, `正常系テスト`
- **Missing Boundaries**: 境界値テストなし (off-by-one エラーの温床)
- **Shared State**: テスト間でデータベースや変数を共有
- **Implementation Mocking**: 内部メソッドをモックして結合を検証するだけのテスト
- **Inverted Pyramid**: E2E だらけで unit がない

## 品質チェックポイント

- [ ] テスト名が「何が起きるべきか」を記述しているか
- [ ] 境界値 (0, 1, N-1, N, N+1) がテストされているか
- [ ] テストが実行順序に依存していないか
- [ ] モックが外部依存 (DB, API, ファイル) に限定されているか
- [ ] テストピラミッドが健全か (unit > integration > E2E)
