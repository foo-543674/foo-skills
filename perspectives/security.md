# Security

セキュリティに関する品質レンズ。

## この観点で重視すること

- **入力は信頼しない**: 外部からの入力はすべて潜在的に悪意がある前提で検証する
- **最小権限**: 必要最小限のデータ公開、内部詳細の非露出
- **多層防御**: 入力検証、エンコーディング、パラメータ化クエリを組み合わせる

## 判断基準

- 認証: すべての保護エンドポイントで認証チェック
- 認可: リソースの所有者チェック (IDOR 防止)
- 秘密情報: 環境変数で管理。コードにハードコードしない
- パスワード: bcrypt / argon2 等の強いハッシュ。MD5, SHA1 は不可
- エラーメッセージ: スタックトレースや内部構造を漏らさない

## アンチパターン

- **Injection**: SQL/NoSQL/コマンドインジェクション (パラメータ化していない)
- **Missing Auth/Authz**: 認証・認可チェックの漏れ
- **Hardcoded Secrets**: コード内にAPIキーやパスワード
- **Weak Hashing**: MD5, SHA1, 平文でのパスワード保存
- **Information Leak**: エラーレスポンスにスタックトレースや内部パスを含む

## 品質チェックポイント

- [ ] 外部入力がパラメータ化クエリ等で安全に処理されているか
- [ ] 保護エンドポイントに認証チェックがあるか
- [ ] リソースアクセスに所有者チェックがあるか (IDOR)
- [ ] 秘密情報が環境変数で管理されているか
- [ ] パスワードが強いアルゴリズムでハッシュ化されているか
- [ ] エラーレスポンスが内部実装の詳細を漏らしていないか
