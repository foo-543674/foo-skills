# Error Handling

エラーハンドリングに関する品質レンズ。

## この観点で重視すること

- **予測可能 vs 予測不能の区別**: バリデーションエラー・ビジネスルール違反は Result/Either 型。ネットワーク障害・DB 接続失敗は例外/panic
- **エラーの握りつぶし禁止**: 空の catch、log だけして続行は禁止。回復不能なら伝搬させる
- **エラー原因の保全**: catch-rethrow 時に原因チェーンを維持する
- **境界での検証**: 外部入力 (ユーザーリクエスト、API レスポンス、ファイル) はシステム境界で検証する

## 判断基準

- try ブロックは狭く。異なるエラー源・異なる回復方法があるなら分割する
- catch は具体的な型で捕捉。`catch(Exception)` のような広い catch は避ける
- ドメイン層のエラーはドメインの言葉で表現する (HTTP ステータスコードではない)
- レイヤー境界でエラーを変換する (infrastructure の例外がドメイン層まで漏れない)

## アンチパターン

- **Exception for Control Flow**: 予測可能なビジネスエラーに例外を使う
- **Swallowed Error**: 空の catch ブロック
- **Lost Cause**: rethrow 時に原因チェーンを失う
- **Broad Catch**: `catch(Exception)` で全例外を一括処理
- **Unvalidated Boundary**: 外部データを検証せずに内部処理に渡す

## 品質チェックポイント

- [ ] 予測可能な失敗が Result/Either で表現されているか
- [ ] 空の catch ブロックがないか
- [ ] rethrow 時に原因チェーンが保たれているか
- [ ] try ブロックのスコープが適切か (狭すぎず広すぎず)
- [ ] 外部入力がシステム境界で検証されているか
