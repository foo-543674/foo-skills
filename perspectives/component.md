# Component

コンポーネント・モジュールの責務分割に関する品質レンズ。UI に限らずバックエンドのモジュール設計にも適用。

## この観点で重視すること

- **表示と状態管理の分離**: 表示は「状態をどう見せるか」のみ。状態管理は別の責務。混在するとデザイン変更のたびに全体をテストし直す必要がある
- **Props ベースの外部制御**: コンポーネントの状態は Props で外から制御可能であること。内部 state を持つと Storybook での全カタログ化が破綻する
- **4 区分の明確化** (フロントエンド): Page (URL 対応) / Container (状態管理の橋渡し) / Domain Presentational (ドメイン固有 UI) / Generic Presentational (汎用 UI)
- **シグネチャの境界設計**: 関数やメソッドのシグネチャに外部ライブラリ固有の型を露出させない (捨てやすさ)

## 判断基準

- **fully controlled 原則**: `<input>` の value/onChange のように、状態が外部から完全に制御できるか。CSS 擬似クラス (`:hover`, `:focus`) で済むものは state にしない
- **Rule of Three**: 3 回出るまで共通化しない。1-2 回のパターンを「将来のため」に抽象化するのは逃げ D
- **循環依存の回避**: モジュール A → B → A は設計のサイン。第三のモジュールに共通部分を抽出する
- **バックエンドモジュール**: ISP (Interface Segregation) を適用し、使わないメソッドへの依存を避ける

## アンチパターン

- **State + Rendering 混在**: 1 コンポーネントが API 呼び出し + 状態管理 + 表示を全部やる
- **Implicit Dependencies**: Context 経由の暗黙的依存 (Compound pattern 以外)
- **Presentational に副作用**: 表示専用コンポーネント内で API 呼び出しや useEffect
- **God Component / God Module**: 1 つのコンポーネント/モジュールが多数の責務を持つ

## 品質チェックポイント

- [ ] 状態管理と表示が分離されているか
- [ ] すべての状態値が Props で外部制御可能か
- [ ] Presentational コンポーネントに副作用がないか
- [ ] コンポーネント/モジュールの責務が 1 文で定義できるか (語彙定義義務)
- [ ] 共通化が Rule of Three を満たしているか
- [ ] 循環依存がないか
