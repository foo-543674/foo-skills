# API Design

API 設計に関する品質レンズ。設計時にもレビュー時にも使う。

## この観点で重視すること

### 契約の明確性

API はクライアントとの **契約** であり、一度公開すると後方互換性の維持が必須。曖昧な設計はクライアントの実装を複雑にし、修正が困難な技術的負債になる。

### 一貫性がクライアント開発効率を決める

エンドポイント間で命名規則、レスポンス構造、エラー形式が統一されていることが、クライアント側の実装コストを劇的に下げる。

### 内部実装を漏らさない

API 仕様に内部実装 (ORM の生成型、フレームワーク固有型、外部 SDK の DTO) が漏れると、内部を変えただけで API クライアント全員に影響が波及する。

## 判断基準

### HTTP ステータスコードのセマンティクス

HTTP ステータスは **プロトコル層の関心事に限定** する。ドメインバリデーションの細分類はボディのエラーコードで表現する。

| ステータス | 正しい使いどころ |
|---|---|
| **400** | ドメインバリデーション全般 (形式不正、ボディ参照 ID の不存在、ユニーク制約違反、ドメインルール違反、状態依存の拒否) |
| **401** | 認証トークンが無い / 無効 |
| **403** | 認証済みだが操作禁止 |
| **404** | **パスパラメータが指すリソースの不存在のみ** |
| **409** | **楽観的排他制御の並行更新コンフリクト専用** (ETag / version 不一致) |
| **500** | 予期しない内部例外 |

**典型的な誤用**: ユニーク制約違反やドメインルール違反を 409 にする、ボディ参照 ID の不存在を 404 にする → すべて 400。

### エラーレスポンスは常に配列ベース

- **単一エラー形式 (`error: {...}`) は禁止**。常に `errors: [...]` で設計する
- バリデーションは **fail-fast しない**。複数制約違反・複数フィールド違反を 1 レスポンスで返す
- `code` にはフィールド名を埋めない。`code: "too_long"` + `field: "name"` のように分離する
- バックエンドの Validator は収集型 (`Result<T, Vec<Error>>`) を返す契約にする

### REST と BFF の分離

SPA/PWA フロントエンドを持つアプリでは:
- `/api/v1/*` — リソース指向 (CRUD + 検索)。動詞パスを含めない
- `/web/v1/*` — Web UI BFF。動詞パス許容 (login, logout 等)

BFF のプレフィックスは「誰向けか」を明示する (`/web/v1`, `/mobile/v1`, `/admin/v1`)。`/bff/v1` は何の情報も伝わらない。

### 内部実装型の漏出防止

- ORM の生成型をレスポンスに使わない。ドメイン型と API DTO は別の型として定義する
- 外部 SDK の DTO をそのままレスポンスに使わない
- 内部のエラーコード (Prisma の `P2002` 等) をクライアントに露出しない
- エラーコード体系は API 独自の語彙で定義する

### バージョニング

- 公開 API にはバージョニング戦略を持つ (URL path / header / query)
- 破壊的変更の定義を明確にする (フィールド追加は非破壊、削除・型変更は破壊)
- 内部 API でクライアントとサーバーが同時デプロイなら不要

## アンチパターン

- **Chatty API**: 1 画面に大量のリクエストが必要
- **RPC in REST Clothing**: REST を名乗りつつ `POST /api/getUsers`
- **Inconsistent Error Shape**: エンドポイントごとにエラー構造が異なる
- **Fail-fast Validation**: 1 エラー見つけた時点で返し、他を隠す drip-feed UX
- **Status Code Abuse**: ドメインエラーの細分類に HTTP ステータスを使う
- **Mixed Verb and Resource Paths**: `/api/v1/users` と `/api/v1/auth/login` が同じ名前空間
- **ORM Type Leak / SDK DTO Leak**: 内部実装の型がレスポンスに露出
- **Domain = DTO Conflation**: ドメイン型と API DTO を兼用

## 品質チェックポイント

### 一貫性 (最優先)

- [ ] リソース名が複数形で統一されているか
- [ ] URL 命名規則が統一されているか (kebab-case or snake_case)
- [ ] REST でアクション動詞を含んでいないか
- [ ] レスポンス構造がエンドポイント間で一貫しているか (envelope pattern 等)
- [ ] 日時フォーマット、null の扱い、空配列 vs null が統一されているか

### エラー設計

- [ ] エラー構造が配列ベース (`errors: [...]`) か
- [ ] バリデーションが fail-fast していないか
- [ ] エラーコードが制約種別のみを表し、フィールド名を埋めていないか
- [ ] 内部実装のエラーコード (ORM 等) が漏れていないか
- [ ] HTTP ステータスコードのセマンティクスが正しいか (400/404/409 の使い分け)

### 漏出防止

- [ ] ORM 生成型がレスポンスに露出していないか
- [ ] 外部 SDK の DTO がレスポンスに露出していないか
- [ ] ドメイン型と API DTO が意図的に分離されているか
- [ ] 不要なフィールドを返していないか (over-fetching)

### ページネーション・冪等性

- [ ] リスト系エンドポイントにページネーションがあるか
- [ ] 方式 (offset vs cursor) が要件に合っているか
- [ ] 二重実行で実害がある操作に冪等性キーがあるか
