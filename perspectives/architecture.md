# Architecture

アーキテクチャに関する品質レンズ。設計時には「このレンズに照らして設計判断する」、レビュー時には「このレンズに照らして品質を評価する」ために使う。

## この観点で重視すること

### 営みの起源でレイヤーを分ける

レイヤーを「Clean Architecture だから」「DDD だから」のような既成の枠組みで決めない。各レイヤーが扱う関心事の **起源** で分ける。

| レイヤー | 起源 | 1 文定義 |
|---|---|---|
| domain | 紙の帳簿時代から存在した | アプリになる前から、現実でその営みをする上で必要だったロジック |
| application | アプリにしたから生まれた | 永続化、Tx 境界、ログ、認可等、アプリにしたときに初めて発生する文脈 |
| infrastructure | 外界への接続 | domain と application が外界に求める能力 (ポート) の具体実装 |
| interfaces | 外部プロトコル境界 | HTTP / gRPC / CLI 等のプロトコル境界。DTO 変換と認証ミドルウェア |

### この区分による帰結

- **リポジトリは domain ではなく application に置く** — 永続化は紙の帳簿の時代には存在しなかった。Clean Architecture の典型図を無批判に当てはめない
- **Clock や PasswordHasher のような「現実の概念のポート」は domain に置く** — 時刻も秘密の変換も紙の帳簿時代から存在した概念
- **infrastructure は domain にも依存する** — Clock 実装が domain の Clock を実装するため

### ドメインロジックの置き場所の優先順位

上から検討し、上で表現できないときだけ下に降りる。

1. **値オブジェクト** — 単一の値が満たすべき制約。`tryNew` / `parse` が Result を返す
2. **集約 / エンティティのメソッド** — 既存集約の状態に対する単純な操作。外部事実は引数で受け取る
3. **Input → Validator → Validated → Factory の型付きパイプライン** — 集約生成や複雑なミューテーション
4. **ドメインワークフロー** — 本当に複数集約にまたがる不可分な判断がある場合のみ。多くのプロジェクトでは不要

### アプリケーション層の設計

- 具体的責務クラスとして表現する (TransactionRunner, Authorizer, RetryPolicy 等)。「アプリケーションサービス」とまとめない
- ユースケースは薄いグルー: 入力 DTO 詰め替え → 責務クラス + domain 呼び出し → 出力 DTO 詰め替え
- ユースケース本体の単体テストは書かない (ロジックがなくモック検証になるだけ)

### エラー境界

- ユーザー入力・外界由来の失敗は Result/Either 型で表現する
- 例外/panic はプログラマのバグを表す中断専用
- ドメイン層のエラーはドメインの言葉で表現する (HTTP ステータスコードではない)
- レイヤー境界でエラーを変換する

## 判断基準

### いつレイヤーを分離するか

- プロジェクトの規模・複雑さに応じて判断する。単純な CRUD に過剰なレイヤーは不要
- 「ドメインロジックがあるか」が分水嶺。ロジックがあるなら分離する価値がある
- 既存プロジェクトでは既存構造を尊重し、段階的に改善する

### いつ CQRS を適用するか

- 書き込みと読み取りで要求が大きく異なる場合のみ
- Query 側に過度な抽象化 (Repository 等) をしない。ストレージの能力を活かす薄い構造
- 単純な CRUD に適用するのは過剰設計

### 外部事実の渡し方 (3 形式)

Validator が必要とする外部データは application 層から引数で渡す:

- **形 A: プレーンな値** — O(1)〜O(log n) で結論が出せる場合 (例: `nameTaken: Boolean`)
- **形 B: ドメイン語彙のコレクション抽象** — 「集合への問い合わせ」がドメインの語彙として独立している場合 (例: `ExerciseLedger`)
- **形 C: 高階関数** — 遅延的・条件付きでデータを読みたい場合 (例: `loadParent: (ExerciseId) -> Optional<Exercise>`)

## アンチパターン

- **Cargo Cult Architecture**: 有名企業の事例を自プロジェクトの文脈を無視して模倣
- **「ドメインサービス」最初置き**: 値オブジェクト・集約メソッド・Validator/Factory を検討せずに放り込む
- **貧血エンティティ**: 集約を getter/setter のみにして不変条件を守る責務をなくす
- **リポジトリ = domain**: Clean Architecture の典型図を雑に当てはめる
- **Framework Coupling**: ドメイン層にフレームワーク固有の依存を持ち込む
- **インスタンス化時例外**: コンストラクタで例外を投げて検証する
- **God Use Case**: 1 つのユースケースが複数のビジネス操作をまとめて実行
- **God Service**: 複数集約操作が XxxService に集約され、ドメインロジックの置き場所が曖昧

## 品質チェックポイント

### 依存の方向 (最優先)

- [ ] 上位モジュール (domain, application) が下位モジュール (infrastructure) の具象を import していないか
- [ ] 依存の方向が一方通行になっているか (循環依存がないか)
- [ ] 抽象 (インターフェース) を介して依存しているか

### レイヤーの責務

- [ ] domain に「アプリにして初めて必要になるロジック」(永続化, IO) が混入していないか
- [ ] domain にフレームワーク固有の依存がないか
- [ ] application に「アプリがなくても存在するビジネスルール」が埋まっていないか
- [ ] infrastructure にビジネスロジックが混入していないか

### ユースケース設計

- [ ] 1 ユースケース 1 クラスか
- [ ] 名前がビジネス操作を表しているか
- [ ] ユースケースが「手順の記述」に徹しているか (複雑なロジックがないか)

### エラー境界

- [ ] ドメイン層のエラーがドメインの言葉で表現されているか
- [ ] infrastructure 層の例外がドメイン層まで漏れていないか
- [ ] レイヤー境界でのエラー変換が行われているか
