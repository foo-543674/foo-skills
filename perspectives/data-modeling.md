# Data Modeling

データモデルに関する品質レンズ。

## この観点で重視すること

- **ストレージ非依存のドメインモデル**: ORM / DB 固有の型がドメインモデルに混入しない。ドメインモデルと永続化モデルは別物
- **数値的構造の文字列化禁止**: 日付は DATE/TIMESTAMPTZ、金額は DECIMAL、期間は INTERVAL。文字列で逃げない
- **直交する次元の独立表現**: 独立した属性の組み合わせを排他列挙で表現しない。各次元を独立カラムにする
- **正規化 vs 非正規化の判断**: Command 側は正規化、Query 側はパフォーマンス要件に応じて非正規化を許容

## 判断基準

- Nullable で制約から逃げない。「ある条件のときだけ必須」は CHECK 制約や別テーブルで表現する
- インデックスはクエリパターンから逆算して設計する
- マイグレーションは後方互換を意識する (カラム追加 → デプロイ → 旧カラム削除の段階的移行)
- 履歴・監査が必要なデータは設計段階で決める (後付けは困難)

## アンチパターン

- **String-Typed Everything**: 日付・金額・期間を VARCHAR で持つ
- **Enum Explosion**: 直交次元を排他列挙で表現し、組み合わせ追加のたびにマイグレーション
- **Nullable Escape**: すべて Nullable にして制約をアプリケーションコードに散らす
- **ORM Model = Domain Model**: ORM の生成型をそのままドメインモデルとして使う
- **Missing Index**: クエリパターンを考慮しないインデックス設計

## 品質チェックポイント

- [ ] 数値的構造のデータに適切な型が使われているか (DATE, DECIMAL, INTERVAL 等)
- [ ] 直交する次元が独立カラムとして表現されているか
- [ ] Nullable に必然性があるか (「面倒だから」ではないか)
- [ ] ORM の型がドメイン層に漏れていないか
- [ ] クエリパターンに対応するインデックスがあるか
- [ ] マイグレーションが後方互換を考慮しているか
