# Comments

コメントに関する品質レンズ。

## この観点で重視すること

- **コメントは真実であること**: コードが変わってコメントが古いままなのは、コメントがないより悪い
- **Why を書く、What は書かない**: コードが What を表現する。コメントはコードでは伝わらない理由 (Why) を補足する
- **prefix 規約**: NOTE, TODO, FIXME, HACK を使い分ける。prefix なしの非ドキュメントコメントは目的が不明
- **英語**: 検索性とツール互換性のため英語で記述する
- **public API にはドキュメントコメント**: パラメータ、戻り値、例外の契約を記述する

## 判断基準

- コメントが必要だと感じたら、まずコードで表現できないか考える (変数名の改善、関数抽出)
- TODO には Issue 番号か具体的なコンテキストを付ける。`// TODO: fix later` は意味がない
- ライセンスヘッダやコピーライトは別途のルールに従う

## アンチパターン

- **Stale Comment**: コード変更後にコメントが更新されていない
- **Redundant Comment**: `i++ // increment i` のように コードを繰り返すだけ
- **Prefix-less Comment**: NOTE/TODO/FIXME/HACK なしの非ドキュメントコメント
- **TODO Without Context**: `// TODO` だけで何をいつやるか不明

## 品質チェックポイント

- [ ] コメントの内容が実際のコードの動作と一致しているか
- [ ] 非ドキュメントコメントに prefix (NOTE/TODO/FIXME/HACK) があるか
- [ ] コメントが英語で書かれているか
- [ ] コメントが Why を説明しているか (What の繰り返しではないか)
- [ ] public API にドキュメントコメントがあるか
