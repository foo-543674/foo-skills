# Performance

パフォーマンスに関する品質レンズ。

## この観点で重視すること

- **N+1 問題の排除**: ループ内で DB/API を呼ぶ構造はスケール不可能
- **計算量の意識**: O(n²) のネストループは Map/Set で O(n) にできないか
- **大量データの段階処理**: 10k+ レコードの全件メモリロードを避ける。ストリーム/ページネーション
- **計測してから最適化**: 推測でなくプロファイリングに基づく。可読性を犠牲にする最適化は計測根拠が必須

## 判断基準

- データ量が増えたときの振る舞いを常に考える。「今は 100 件だから大丈夫」は答えにならない
- 独立した I/O は並列実行する (Promise.all, tokio::join 等)
- キャッシュは「何をキャッシュするか」より「いつ無効化するか」が難しい。安易に導入しない

## アンチパターン

- **N+1 Query**: `for user in users: orders = getOrders(user.id)`
- **Nested Loop Search**: O(n²) で検索。Map/Set で O(1) lookup が可能
- **All-in-Memory**: 10 万件を一度にメモリにロード
- **Sequential I/O**: 独立した非同期処理を直列に await
- **Premature Optimization**: 計測なしの最適化で可読性を犠牲にする

## 品質チェックポイント

- [ ] ループ内に DB/API 呼び出しがないか (N+1)
- [ ] ネストループが Map/Set で置き換え可能でないか
- [ ] 大量データのページネーション/ストリーム処理があるか
- [ ] 独立した I/O が並列実行されているか
- [ ] 最適化にプロファイリング根拠があるか
