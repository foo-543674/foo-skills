# Documentation

ドキュメントに関する品質レンズ。

## この観点で重視すること

- **設計判断の記録**: 「なぜこうしたか」を `.contexts/` や ADR に残す。コードは What を表現するが Why は残らない
- **依存ライブラリのガイドライン**: 新しいライブラリを追加したら、使い方ガイドライン (どこから import 可、隔離戦略) を `.contexts/dependencies/` に残す
- **AI 用コンテキスト**: CLAUDE.md を最新に保つ。AI が自走するための情報基盤

## 判断基準

- ドキュメントが必要なのは「コードから読み取れないこと」だけ
- ADR の作成基準: 複数の選択肢があり、選ばなかった理由も含めて記録する価値がある判断
- README は「git clone 直後に何をすればいいか」に答える

## アンチパターン

- **Undocumented Decision**: 非自明な設計判断が記録されていない
- **Ungoverned Library**: 外部ライブラリの使用ルールが明文化されていない
- **Stale Documentation**: ドキュメントが実装と乖離している
- **Missing Context**: 新しいレイヤー/モジュール/ディレクトリの追加に設計意図の記録がない

## 品質チェックポイント

- [ ] 非自明な設計判断が `.contexts/` または ADR に記録されているか
- [ ] 新しい外部ライブラリに使用ガイドラインがあるか
- [ ] 新しいレイヤー/モジュール/ディレクトリに設計意図の記録があるか
- [ ] CLAUDE.md がプロジェクトの現状と一致しているか
