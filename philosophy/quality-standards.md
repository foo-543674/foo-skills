# 品質の定義と基準

## 製品品質と技術品質

品質には 2 つの種類がある。これらを混同してはならない。

### 製品品質 — 「あったら嬉しい」

その製品のバリュー、セールスポイント。ユーザーが意識的に語るもの。

- アプリのレスポンスが速い
- UI が美しい
- 機能が豊富
- 使いやすい

### 技術品質 — 「なかったら問題」

その製品が当然担保されていると **暗黙的に認知されている** もの。誰も口に出さないのに必須。

- テストが書ける、通る
- リンタが効く、フォーマットが揃う
- CI が回る
- 依存が解決する
- 再現可能な環境で動く
- セキュリティ脆弱性がない
- アーキテクチャの秩序が保たれている

### 優先順位

**技術品質は製品品質に優先する。**

技術品質に Critical な問題が残っている状態で、製品品質の改善に時間を費やしてはならない。セキュリティ脆弱性やデータ損失リスクは、UI の美しさより先に対処する。

これは「技術品質のほうが大事」という意味ではない。技術品質は「当然担保されている前提」であり、その前提が崩れた状態で製品品質を語っても意味がないということ。

## AI 自走時の品質保証: 三層防御

AI に開発の大部分を任せる場合、「AI の判断を信頼する」のではなく「AI の判断を自動検証する」仕組みが必要。

```
Layer 1: CLAUDE.md (ガイダンス)
  AI が従うべきルールを言語で伝える
  → 破れる (AI が複雑な実装中に無意識に違反する可能性)

Layer 2: Architecture Tests (ガードレール)
  ルール違反を CI で自動検出する
  → 破れない (CI が壊れて止まる)
  → AI 自身が「壊れた → 直す」のループを回せる

Layer 3: Code Review Agent (評価)
  テストでは捉えきれない設計品質を評価する
  → 人間の最終判断を支援
```

### Architecture Tests で検証すべきルール

- **依存方向**: domain が infrastructure を import していないか
- **型の配置**: ORM 型がドメイン層に出現しないか
- **命名規則**: 禁止語 (Service, Manager 等) がクラス名に使われていないか
- **import 制限**: ガイドラインで定めた制限が守られているか

これらは perspectives/ の品質基準から導出し、プロジェクトの技術スタックに応じたツール (dependency-cruiser, ArchUnit, import-linter 等) で実装する。
