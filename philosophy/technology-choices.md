# 技術選定の判断基準

私が技術を選ぶとき、何を重視し、どういう傾向があるかを記述する。AI はこれを参照して技術提案を行う。

## 型システムへの信頼

**静的型付けを強く好む。** やろうと思えば動的型付けスクリプト言語でサクッと開発できるが、私の思想としてそうは思わない。

### なぜ静的型付けか

- **型は設計意図の表現**: 「未検証 / 検証済み」「ID / 生文字列」を型で区別することで、コンパイル時に設計意図の違反を検出できる
- **Result/Either でエラーを表現**: 例外に頼らず、型レベルで「失敗しうる操作」を明示する。呼び出し側がエラー処理を忘れることをコンパイラが防ぐ
- **リファクタリングの安全性**: 型があることで、大規模な構造変更をコンパイラに守られながら実行できる

### 選定傾向

- バックエンド: Rust, Scala, Kotlin, Go。TypeScript はバックエンドには使わない
- フロントエンド: TypeScript (strict mode 必須)、Flutter にも関心がある
- スクリプト/CLI: 言語そのものより **「どこで走らせるか」と「準備なしに実行できるか」** が重要。CLIツールを叩こうとしたら npm がインストールされていない、バージョンが違う — こういう状況を避けたい。GAS なら JS 一択、Lambda や GitHub Actions ならランタイムの制約に従う。実行環境が言語を決める
- 動的型付け言語を選ぶのは、実行環境の制約がある場合のみ

## プログラミングパラダイム

**関数型プログラミングを好む。** ただし純粋な関数型言語を使うという意味ではなく、実用的な言語の中で関数型の考え方を適用する。

### なぜ関数型か

高階関数と型システムによる **表現力** にある。手続き型言語では表現できない「英文法」のような構造を扱え、英文を書くように宣言的に読みやすいコードをシンプルに書ける。命令の羅列ではなく「何を求めているか」が読み取れるコードになる。

### 重視すること

- **Immutable ファースト**: データの変更ではなく新しいデータの生成
- **Pure functions の優先**: 副作用を最小限に抑え、テスタビリティを上げる
- **関数合成**: パイプライン的なデータ変換。英文のように左から右へ読める
- **型による状態遷移の表現**: Union types / sealed class でドメインの状態を表現
- **宣言的なスタイル**: 「どうやるか」ではなく「何であるか」を書く

### 避けること

- 最上部での変数一括宣言
- Mutable なコレクションの多用
- C 言語スタイルのループ (map/filter/reduce で書けるもの)
- 破壊的な操作の連続

## アーキテクチャの判断傾向

- **レイヤードアーキテクチャ**: 営みの起源でレイヤーを分ける (perspectives/architecture 参照)
- **テストファースト**: Red-Green-Refactor サイクル
- **CQRS**: 書き込みと読み取りで要求が異なる場合のみ。過剰適用しない
- **DDD**: ドメインモデルの語彙を大事にする。ただし戦術パターンの形式的な適用は避ける

## 選定プロセス

技術選定時に検討する軸:

1. **要件を満たせるか** (機能要件 + 非機能要件)
2. **型システムの強さ** (静的型付け + 型推論の質)
3. **コミュニティの成熟度** と長期サポート
4. **既存スタックとの整合性**
5. **捨てやすさ** (隔離戦略が適用できるか)

トレードオフが発生した場合は ADR に記録する。
