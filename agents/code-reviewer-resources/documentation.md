# ドキュメンテーションレビュー

コードレビュー時にドキュメンテーションの観点で指摘すべきポイントを定義する。

## Why: なぜドキュメンテーションレビューが重要か

ドキュメントは「読まれる場所」に集約され、AI を含む読み手が適切に文脈を取得できなければ、書かれていないのと同じ価値しか持たない。

**根本的な理由**:

1. **発見可能性（A）**: README / CONTRIBUTING.md / `.contexts/` 等、**読み手が探さなくても辿り着く場所** にドキュメントが集約されていなければ、せっかく書いた情報も誰にも読まれず腐る。独自命名の `ARCHITECTURE.md` や `DEVELOPMENT.md` を作っても、「そのファイルが存在することを知っている人」しか読まないため、新規参入者・将来の自分・AI から見ると存在しないのと同じ
2. **AI への文脈伝達（D）**: CLAUDE.md などの AI 向けドキュメントが整備されていないと、AI は文脈を持てず、生成物の品質が下がる。AI が前提を取り違えた生成物は、人間がレビューで弾かなければならず、レビューコストとして跳ね返る
3. **デファクト集約による保守性**: ドキュメントが散らばると、更新漏れと矛盾が発生する。「この情報はどこに書くべきか」の判断軸が「既存のデファクトに収まるか」で統一されていれば、置き場の議論が消え、更新箇所も1箇所に絞れる
4. **形骸化の回避**: 「ツール/機能/サービスが活用するファイル」という基準で必要性を判定することで、誰も読まない・誰のためにあるか分からないドキュメントの増殖を防げる

**ドキュメンテーションレビューの目的**:
- 独自命名のドキュメントを既存のデファクト（README / CONTRIBUTING.md / `.contexts/` 等）に統合させる
- 自動化で代替できるドキュメントを手書きさせない
- AI が文脈を取得できる状態（CLAUDE.md 等）を維持する
- 重要な意思決定の経緯を `.contexts/` に残す

## トリガー条件

この観点は以下の場合に発動する。`code-reviewer.md` 側で観点選択時にこの条件を確認すること。

1. **ドキュメント（.md ファイル）の追加・変更を含む変更**
2. **重要な意思決定を含む変更**:
   - 依存ライブラリの新規追加（必須）
   - 依存ライブラリのメジャーバージョンアップで破壊的変更を伴うもの（推奨）
   - 新しいレイヤー / モジュール / ディレクトリの追加（必須）
   - 新しい API エンドポイントの追加で、似た既存エンドポイントがあるのに分けている等の非自明な分割（必須）
   - データベーススキーマの変更で非自明な決定を含むもの（例: 物理削除レコードを退避するログテーブルの新設）（必須）
   - CI / インフラ構成の変更（新ワークフロー追加等）（必須）

上記に該当しない場合（純粋なロジック修正、リファクタリング等）は、この観点を適用しない。

## 判断プロセス（決定ツリー）

ドキュメンテーションのレビュー時は、以下の順序で判断する。重大度が高く、観点の核となる項目から確認する。

### ステップ1: 独自ドキュメントが追加されていないか（最優先）

**判断基準**:
新規に追加された .md ファイルが、**そのファイルを活用するツール / 機能 / サービスが実在する** ものかを確認する。実在しない場合は独自ドキュメントとみなし、既存のデファクトに統合させる。

**Why（なぜ独自ドキュメントが問題か）**:
- 独自命名のファイルは「存在を知っている人」しか読まないため、発見可能性がゼロに近い
- 「誰向けのドキュメントなのか」が不明瞭で、内容の妥当性を判断する基準が無い
- 既存のデファクト（README / CONTRIBUTING.md / `.contexts/` 等）に置けば、追加コスト無しで読み手に届く

**OK の判定基準**:
そのファイルを活用する **ツール / 機能 / サービス** が実在すること。

| ファイル | 活用する仕組み | 判定 |
|---------|---------------|------|
| `README.md` | GitHub のリポジトリトップ表示 | OK |
| `LICENSE` / `LICENSE.md` | GitHub 認識、npm/cargo 等のパッケージマネージャ | OK |
| `CONTRIBUTING.md` | GitHub の PR / Issue 作成画面に表示 | OK |
| `CODE_OF_CONDUCT.md` | GitHub Community Standards | OK |
| `SECURITY.md` | GitHub Security タブ | OK |
| `CHANGELOG.md` | release-please / changesets 等のリリースツール | OK |
| `CLAUDE.md` | Claude Code が読み込む | OK |
| GitHub テンプレート系（`.github/ISSUE_TEMPLATE/` 等） | GitHub の Issue / PR テンプレート機能 | OK |

**NG の例**:

| ファイル | NG の理由 | あるべき置き場 |
|---------|----------|---------------|
| `ARCHITECTURE.md` | アーキテクチャ構成を書きたいだけなら CONTRIBUTING.md で十分。意図や背景を書きたいなら `.contexts/` | CONTRIBUTING.md / `.contexts/` |
| `DEVELOPMENT.md` | CONTRIBUTING.md と何が違うのか不明 | CONTRIBUTING.md |
| `SECURITY_OVERRIDES.md` | 誰向けなのか不明。依存脆弱性の override 理由なら CONTRIBUTING.md か `.contexts/` | CONTRIBUTING.md / `.contexts/` |
| `backend/NOTES.md` | サブディレクトリ独自のメモは読み手不明。書きたい内容に応じて該当ディレクトリの README に統合するか、`.contexts/` に移す | サブディレクトリの README / `.contexts/` |
| `docs/setup-guide.md`（開発者向け） | 開発者向けなら CONTRIBUTING.md か `.contexts/` | CONTRIBUTING.md / `.contexts/` |

**例外**: `docs/` ディレクトリにエンドユーザー向けドキュメント（GitHub Pages や他のホスティングサービスで公開されるヘルプコンテンツ等）を置くのは OK。これは「コンテンツ」であって独自ドキュメントではない。

**判定フロー**:
```
新規 .md ファイルを発見
  ↓
そのファイルを活用するツール / 機能 / サービスが実在するか？
  ↓ Yes → OK
  ↓ No
  ↓
書きたい内容は既存のデファクト（README / CONTRIBUTING.md / .contexts/）に収まらないか？
  ↓ 収まる → [Warning] 該当のデファクトに統合すべき
  ↓ 収まらない → [Warning] なぜ収まらないかを説明し、それでも独自ファイルが必要か再検討
```

**判定**: 独自ドキュメントの追加は **必ず指摘する**（Warning）

### ステップ2: 自動化できるドキュメントを手書きしていないか（高優先度）

**判断基準**:
ツール / 仕組みで自動生成・自動同期できるドキュメントを手書きしている場合は、自動化への置き換えを求める。手書きドキュメントは陳腐化が早く、コードとの矛盾が放置されやすい。

**自動化すべきドキュメントの例**:

| 手書きしてはいけないもの | 自動化手段 |
|------------------------|-----------|
| API 仕様（手書き `API.md` 等） | OpenAPI / Protocol Buffers / GraphQL SDL から生成 |
| 型 / 関数リファレンス（手書き `TYPES.md` 等） | rustdoc / TypeDoc / JSDoc で生成 |
| 環境変数一覧（手書き `ENV.md` 等） | `.env.example` を単一ソースにする |
| CHANGELOG（手書き更新） | conventional commits + release-please / changesets |
| 依存関係一覧（手書き `DEPENDENCIES.md` 等） | `Cargo.toml` / `package.json` を単一ソースにする |

**判定フロー**:
```
新規ドキュメントの内容を確認
  ↓
コード / 設定ファイル / コミット履歴から自動生成可能か？
  ↓ Yes → [Warning] 自動化手段に置き換えるべき
  ↓ No → OK
```

**判定**: 自動化できる内容を手書きしている場合は **必ず指摘する**（Warning）

### ステップ3: CLAUDE.md（AI 用ドキュメント）が整備されているか（高優先度）

**判断基準**:
AI を活用する前提のリポジトリで CLAUDE.md（または同等の AI 向け文脈ファイル）が存在しない場合は指摘する。CLAUDE.md が無い状態で AI 補助を使うと、毎回プロジェクトの前提を伝え直す必要があり、生成物の品質も安定しない。

**チェック項目**:
- リポジトリルートに `CLAUDE.md` があるか
- ドキュメント追加・変更を含む PR で、その変更が CLAUDE.md にも反映されるべき内容（例: 新しい開発フロー、新しい依存、新しい規約）であれば、CLAUDE.md の追従漏れが無いか

**判定フロー**:
```
リポジトリに CLAUDE.md があるか？
  ↓ No → [Warning] CLAUDE.md を追加すべき
  ↓ Yes
  ↓
今回の変更で CLAUDE.md が追従すべき内容があるか？
  ↓ Yes かつ CLAUDE.md 未更新 → [Warning] CLAUDE.md にも反映すべき
```

**判定**: CLAUDE.md の不在・追従漏れは **必ず指摘する**（Warning）

### ステップ4: README の必須要素が揃っているか（中〜低優先度）

**判断基準**:
README は最低限「プロジェクトの目的・概要」が必要。ライセンスは `LICENSE` / `LICENSE.md` ファイルがあれば README に重複記載する必要は無い。それ以外の運用情報（セットアップ手順、開発フロー、ディレクトリ構成）は、CONTRIBUTING.md や `.contexts/` 等の他のデファクトに記載されていれば README には必須ではない。

**チェック項目**:

| 項目 | 必須度 | 補足 |
|------|--------|------|
| プロジェクトの目的・概要（What / Why） | **必須** | README に書く以外の選択肢が無い |
| ライセンス | **必須**（README 内ではない） | `LICENSE` / `LICENSE.md` ファイルがあれば README への記載は不要 |
| セットアップ手順（依存・起動方法） | 条件付き必須 | CONTRIBUTING.md / `.contexts/` に記載があれば README には不要 |
| 開発フロー（テスト実行・ビルド） | 条件付き必須 | 同上 |
| ディレクトリ構成 | 条件付き必須 | 同上 |

**判定フロー**:
```
README が存在するか？
  ↓ No → [Warning] README を追加すべき
  ↓ Yes
  ↓
プロジェクトの目的・概要が書かれているか？
  ↓ No → [Nit] 追加すべき
  ↓ Yes
  ↓
LICENSE / LICENSE.md ファイルが存在するか？
  ↓ No
    ↓
    README にライセンス記載があるか？
      ↓ No → [Nit] LICENSE ファイルまたは README にライセンスを追加すべき
  ↓ Yes → OK（README への重複記載は不要）
  ↓
セットアップ / 開発フロー / 構成のうち、CONTRIBUTING.md にも .contexts/ にも記載が無い項目があるか？
  ↓ Yes → [Nit] README または該当のデファクトに追加すべき
```

**判定**:
- README の **不在** は Warning
- README の **内容の薄さ**（必須要素の欠落）は Nit

### ステップ5: 重要な意思決定が `.contexts/` に記録されているか（条件付き発動）

**判断基準**:
「トリガー条件」の項目2 に該当する変更（依存追加・新レイヤー追加・非自明な API/DB 設計・CI/インフラ変更等）があった場合、その意思決定の経緯が `.contexts/` に記録されているか、かつ **ファイル名がデファクトな命名規約に従っているか** を確認する。

**Why（なぜ `.contexts/` への記録が重要か）**:
- 「なぜこの選択をしたか」が失われると、将来の変更時に同じ議論を繰り返すコストが発生する
- 自明でない選択（複数候補から1つを選んだ等）は、後から見ると「なぜこちらを選んだか」が分からなくなる
- AI が `.contexts/` を読むことで、選択の前提を共有できる

**Why（なぜファイル名にデファクト命名規約が必要か）**:
- 読み手が **ファイルを開かなくても** 「これは読むべきドキュメントか」を判断できるようにするため
- ファイル名が `notes.md` や `memo.md` のような曖昧なものだと、開くまで内容種別が分からず、結果として読まれない
- ADR / RFC / Design Doc 等のデファクトな型に従えば、読み手は型の前提（書き方・フォーマット・読みどころ）を共有しているので、読むコストが下がる

**`.contexts/` 内のファイル命名規約**:

| ドキュメント種別 | 命名規約 | 例 |
|----------------|---------|-----|
| Architecture Decision Record（アーキテクチャ意思決定記録） | `ADR-NNNN-<title>.md` | `ADR-0001-use-axum-for-backend.md` |
| Request for Comments（設計提案・議論） | `RFC-NNNN-<title>.md` | `RFC-0003-event-sourcing.md` |
| Design Doc（特定機能の設計書） | `design-<feature>.md` | `design-auth-flow.md` |
| 要件・コンセプト | `concept.md` / `requirements.md` | `concept.md` |
| その他の型がある場合 | プロジェクト内で決めたデファクトに従う | - |

**判定基準**:
- 上記のような **広く知られた型**（ADR、RFC、Design Doc 等）が当てはまる内容であれば、その型のファイル名・フォーマットに従う
- 当てはまる型が無い場合でも、**プロジェクト内で命名規約を決めて統一する**
- `notes.md` / `memo.md` / `tmp.md` のような **内容種別が伝わらないファイル名** は禁止

**指摘が必要なケース**:

| 変更内容 | 指摘要否 | 補足 |
|---------|---------|------|
| 依存ライブラリの新規追加 | **必須** | なぜこのライブラリを選んだか（候補比較・選定理由） |
| 依存メジャーバージョンアップで破壊的変更あり | 推奨 | 通常のバージョンアップは指摘不要 |
| 新しいレイヤー / モジュール / ディレクトリの追加 | **必須** | なぜそのレイヤー分割を選んだか |
| 新しい API エンドポイントの追加 | 条件付き | 似た既存エンドポイントがあるのに分けている等の非自明な分割の場合のみ。当初要件に含まれる機能や REST/gRPC の文脈で自然なエンドポイントなら不要 |
| データベーススキーマの変更 | 条件付き | 物理削除レコードを退避するログテーブルの新設等、非自明な決定の場合のみ |
| CI / インフラ構成の変更 | **必須** | なぜそのワークフロー / 構成を選んだか |

**判定フロー**:
```
今回の変更がトリガー条件2 に該当するか？
  ↓ No → このステップはスキップ
  ↓ Yes
  ↓
.contexts/ に該当する意思決定の記録があるか？
  ↓ No → [Warning] .contexts/ に記録を追加すべき
  ↓ Yes
  ↓
ファイル名がデファクトな命名規約（ADR-/RFC-/design- 等）に従っているか？
  ↓ No → [Warning] 命名規約に沿ったファイル名にリネームすべき
  ↓ Yes → OK
```

**判定**:
- 該当する変更で `.contexts/` への記録漏れがある場合は **必ず指摘する**（Warning）
- 記録があってもファイル名が曖昧（`notes.md` 等）な場合も **必ず指摘する**（Warning）

## レビュー観点（詳細チェックリスト）

上記の判断プロセスを実行するための具体的なチェックリスト。

### 1. 独自ドキュメントの検出

- [ ] 新規追加された .md ファイルが、ツール / 機能 / サービスに活用されるものか
- [ ] 独自命名のファイル（`ARCHITECTURE.md`, `DEVELOPMENT.md`, `SECURITY_OVERRIDES.md`, `backend/NOTES.md` 等）が無いか
- [ ] サブディレクトリ独自のメモが READMEや `.contexts/` に統合可能でないか
- [ ] `docs/` 配下に開発者向けドキュメントを置いていないか（エンドユーザー向けコンテンツは OK）

### 2. 自動化されるべきドキュメント

- [ ] API 仕様を OpenAPI / Protocol Buffers 等から生成しているか（手書き `API.md` でないか）
- [ ] 型 / 関数リファレンスを rustdoc / TypeDoc / JSDoc 等で生成しているか
- [ ] 環境変数一覧を `.env.example` で管理しているか
- [ ] CHANGELOG を conventional commits / release ツールで自動生成しているか
- [ ] 依存関係一覧を `Cargo.toml` / `package.json` で管理しているか

### 3. CLAUDE.md（AI 用ドキュメント）

- [ ] リポジトリルートに `CLAUDE.md` があるか
- [ ] 今回の変更で CLAUDE.md が追従すべき内容があれば反映されているか

### 4. README の必須要素

- [ ] README が存在するか
- [ ] プロジェクトの目的・概要（What / Why）が書かれているか
- [ ] `LICENSE` / `LICENSE.md` ファイルが存在するか（無ければ README にライセンス記載があるか）
- [ ] セットアップ / 開発フロー / 構成のいずれかが、README にも CONTRIBUTING.md にも `.contexts/` にも無い項目が無いか

### 5. `.contexts/` への意思決定記録

- [ ] 依存ライブラリの新規追加に対して `.contexts/` に記録があるか
- [ ] 新しいレイヤー / モジュール / ディレクトリの追加に対して `.contexts/` に記録があるか
- [ ] 非自明な API 設計（似たエンドポイントの分割等）に対して `.contexts/` に記録があるか
- [ ] 非自明な DB スキーマ変更に対して `.contexts/` に記録があるか
- [ ] CI / インフラ構成の変更に対して `.contexts/` に記録があるか
- [ ] `.contexts/` 内のファイル名がデファクトな命名規約（ADR-/RFC-/design- 等）に従っているか
- [ ] `notes.md` / `memo.md` のような内容種別が伝わらないファイル名が無いか

## 指摘の出し方

### 指摘の構造

```
【問題点】このドキュメントは○○の問題があります
【理由】△△だからです
【影響】この問題により××のリスクが発生します
【提案】□□に変更すると改善します
```

### 指摘の例

#### 独自ドキュメントの指摘

```
【問題点】新規追加された `ARCHITECTURE.md` は独自ドキュメントです
【理由】このファイルを活用するツール / 機能 / サービスが存在しません。アーキテクチャ構成を書きたいなら CONTRIBUTING.md で十分で、意図や背景を書きたいなら `.contexts/` に置くべきです
【影響】独自命名のファイルは「存在を知っている人」しか読まないため、新規参入者・将来の自分・AI から見ると存在しないのと同じになります
【提案】内容を以下のどちらかに統合してください
- 構成図・モジュール一覧 → `CONTRIBUTING.md`
- アーキテクチャ選定の意図・背景 → `.contexts/architecture.md`
```

#### 自動化されるべきドキュメントの指摘

```
【問題点】`API.md` に API 仕様が手書きされています
【理由】API 仕様は OpenAPI / Protocol Buffers 等から自動生成すべきで、手書きすると実装との乖離がすぐに発生します
【影響】コードと矛盾したドキュメントが残り続け、利用者を誤った理解に導きます
【提案】OpenAPI スキーマを単一ソースとし、ドキュメントは生成物として扱ってください。`API.md` は削除し、`openapi.yaml` を `.contexts/` または `api/` に配置してください
```

#### CLAUDE.md 不在の指摘

```
【問題点】リポジトリルートに `CLAUDE.md` がありません
【理由】AI を活用する前提のリポジトリでは、AI が文脈を取得できる場所が必要です
【影響】AI 補助を使うたびにプロジェクトの前提を伝え直す必要があり、生成物の品質も安定しません
【提案】`CLAUDE.md` を追加し、プロジェクトの目的・主要なディレクトリ構成・規約・よく使うコマンドを記載してください
```

#### README 不在の指摘

```
【問題点】リポジトリに README がありません
【理由】README は GitHub のリポジトリトップ表示に使われ、新規参入者が最初に読む場所です
【影響】プロジェクトの目的が分からず、利用 / コントリビュートのハードルが上がります
【提案】最低限「プロジェクトの目的・概要」を含む `README.md` を追加してください。ライセンスは `LICENSE` ファイルがあれば README への重複記載は不要です
```

#### `.contexts/` への記録漏れの指摘

```
【問題点】新しい依存ライブラリ `foo` が追加されていますが、`.contexts/` に選定理由の記録がありません
【理由】「なぜこのライブラリを選んだか」が失われると、将来「他の候補ではダメだったのか」を検討し直すコストが発生します
【影響】同じ議論を将来繰り返すことになります
【提案】ADR 形式で `.contexts/ADR-NNNN-use-foo-for-<purpose>.md` を追加し、候補比較と選定理由を記録してください
```

#### `.contexts/` 内のファイル命名規約違反の指摘

```
【問題点】`.contexts/notes.md` というファイル名では、内容種別が伝わりません
【理由】ファイル名から「これは読むべきドキュメントか」を読み手が判断できる必要があります。`notes.md` のような曖昧な名前は、ファイルを開かないと内容種別が分からず、結果として読まれなくなります
【影響】せっかく書いた意思決定の記録が「存在を知っている人」しか読まないドキュメントになり、発見可能性がゼロに近くなります
【提案】内容に応じてデファクトな命名規約に従ってリネームしてください
- アーキテクチャ意思決定 → `ADR-NNNN-<title>.md`（例: `ADR-0001-layered-architecture.md`）
- 設計提案・議論 → `RFC-NNNN-<title>.md`
- 特定機能の設計書 → `design-<feature>.md`
```

### 指摘の優先度ラベル

- **[Warning]**: 独自ドキュメントの追加、自動化できる内容の手書き、CLAUDE.md 不在、README 不在、`.contexts/` への記録漏れ、`.contexts/` 内のファイル命名規約違反
- **[Nit]**: README の必須要素の欠落（目的・ライセンス・セットアップ等）

## テストケース

レビュー判断の精度を検証するためのテストケース。

### Case 1: 独自ドキュメントの追加（Warning 指摘）

**変更**: リポジトリルートに `ARCHITECTURE.md` を新規追加

**期待される指摘**:
- [Warning] 独自ドキュメントの追加
- 内容を `CONTRIBUTING.md` または `.contexts/` に統合すべき

### Case 2: API 仕様の手書き（Warning 指摘）

**変更**: `docs/API.md` に REST API のエンドポイント一覧と仕様を手書き

**期待される指摘**:
- [Warning] 自動化できるドキュメントを手書きしている
- OpenAPI スキーマを単一ソースとし、`API.md` は削除すべき

### Case 3: CLAUDE.md 不在（Warning 指摘）

**変更**: `README.md` を新規追加（CLAUDE.md は無い）

**期待される指摘**:
- [Warning] CLAUDE.md が存在しない
- AI 用文脈ファイルを追加すべき

### Case 4: 依存追加で `.contexts/` 記録漏れ（Warning 指摘）

**変更**: `Cargo.toml` に新しいライブラリ `foo` を追加（`.contexts/` への記録なし）

**期待される指摘**:
- [Warning] 依存追加に対する `.contexts/` への記録が無い
- 候補比較と選定理由を `.contexts/` に追加すべき（例: `ADR-0002-use-foo-for-bar.md`）

### Case 4b: `.contexts/` 内のファイル名が曖昧（Warning 指摘）

**変更**: 新レイヤー追加と同時に `.contexts/notes.md` を新規追加し、レイヤー分割の理由を記載

**期待される指摘**:
- [Warning] ファイル名 `notes.md` では内容種別が伝わらない
- ADR 等のデファクトな命名規約に従い、`ADR-NNNN-<title>.md` 形式にリネームすべき

### Case 5: README が薄い（Nit 指摘）

**変更**: README にプロジェクト名のみ記載、目的の記述が無く、`LICENSE` ファイルも無い

**期待される指摘**:
- [Nit] README にプロジェクトの目的・概要が無い
- [Nit] `LICENSE` ファイルまたは README にライセンスを追加すべき

### Case 5b: LICENSE ファイルがある場合（指摘なし）

**変更**: README にプロジェクトの目的・概要は記載済み。ライセンスは README に書かれていないが、リポジトリルートに `LICENSE` ファイルがある

**期待される判断**:
- ライセンスは `LICENSE` ファイルが単一ソースとして機能しているので README への重複記載は不要
- 指摘なし

### Case 6: 許容されるケース（指摘なし）

**変更**:
- `docs/user-guide.md` を新規追加（GitHub Pages で公開されるエンドユーザー向けヘルプ）
- `CHANGELOG.md` を `release-please` で自動生成
- 新しいライブラリ追加と同時に `.contexts/dependencies.md` に選定理由を追記

**期待される判断**:
- `docs/user-guide.md` はエンドユーザー向けコンテンツで OK
- `CHANGELOG.md` は自動生成ツールが活用しているので OK
- 依存追加に対する `.contexts/` 記録が揃っているので指摘なし
