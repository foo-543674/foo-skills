# コメントレビュー

コードレビュー時にコメントの観点で指摘すべきポイントを定義する。

## Why: なぜコメントレビューが重要か

コメントは、コードだけでは伝わらない意図・背景・制約を伝える重要な手段であり、適切に管理されなければ負債となる。

**根本的な理由**:
1. **Why（なぜ）の記録**: コードは「何を」するかを示すが、「なぜそうするか」は示さない。設計判断の背景、選択肢の検討結果、将来の拡張意図は、コメントでしか残せない。これがなければ、将来の変更時に同じ議論を繰り返す
2. **コメントの劣化**: コメントはコードと異なり、コンパイラやテストでチェックされない。リファクタリング後に更新されず、嘘の情報として残り続けると、読み手を誤った理解に導き、バグの温床となる
3. **不要なコメントの害**: 自明な内容を繰り返すコメントは、コード読解時のノイズとなり、重要な情報を埋もれさせる。「何を」を説明するコメントは、コードの重複であり保守コストを増やす
4. **公開APIの責務の明確化**: ドキュメンテーションコメントは、APIの契約を定義する。責務の境界、パラメータの制約、エラー条件が明確でなければ、利用者は誤用し、提供者は意図しない使われ方に対応を迫られる
5. **技術的負債の可視化**: TODO/FIXMEは、技術的負債と将来のタスクを可視化する。適切に管理されなければ、負債が蓄積し続け、どこに問題があるか把握できなくなる

**コメントレビューの目的**:
- Why（なぜ）を記録し、将来の変更を支援する
- コメントとコードの整合性を保ち、嘘の情報を排除する
- 不要なコメントを削除し、重要な情報を際立たせる
- 公開APIの契約を明確にし、誤用を防ぐ
- 技術的負債を可視化し、計画的に返済する

## 判断プロセス（決定ツリー）

コメントのレビュー時は、以下の順序で判断する。バグや誤解に直結する問題から確認し、問題があれば指摘する。

### ステップ0: 対象コメントの列挙（必須・最優先）

**判定ツリー（ステップ1〜6）を実行する前に、変更ファイル内の非ドキュメントコメントを全件列挙する。** このステップを省略してはならない。レビュー対象が大規模でも省略禁止。

**対象（prefix ルール等の判定対象）**:
- 行コメント: `//`（Rust/JS/TS/Go/C系）, `#`（Python/Ruby/Shell）, `--`（SQL/Haskell/Lua）, `;`（Lisp系）
- ブロックコメント: `/* ... */`（C系）, `=begin ... =end`（Ruby）等

**対象外（prefix ルールの対象外。ステップ4 のドキュメンテーションコメントの観点でのみ評価する）**:
- ドキュメンテーションコメント: `///`, `//!`（Rust）, `/** ... */`（JSDoc/JavaDoc）, `"""docstring"""`（Python）, `#[doc = ...]`
- shebang 行: `#!/usr/bin/env ...`
- ライセンスヘッダ・著作権表示

**走査手順**:
1. git diff から変更ファイルを特定する
2. 各ファイルで「対象」のコメント行を **全件抽出する**（grep 等で機械的に）
3. 抽出した各コメントを **1件ずつ** ステップ1〜6の判定ツリーにかける
4. 列挙を省略して「目についたものだけ」を判定するのは禁止

**よくある誤判定**:
- `//!` や `///` の直後に続く `//` 行コメントを「ドキュメンテーションコメントの一部」と誤認する → `//` で始まる行は独立した非ドキュメントコメントとして扱う
- 関数本体内の `//` コメントを「実装の一部」として判定対象から外す → 関数本体内であろうと判定対象

**判定**: このステップを省略すると、prefix なしコメントの見落としが発生する。レビュー対象の規模を理由にスキップしてはならない。

### ステップ1: コメントと実装が乖離していないか（最優先）

**判断基準**:
コメントと実装が一致していないと、読み手を誤った理解に導き、バグの原因となる。

**チェック項目**:

1. **コメントの記述と実際の動作**:
   ```typescript
   // ❌ コメントと実装が乖離
   // ユーザーの年齢を返す
   function getUserName(user: User): string {
     return user.name;  // 実際は名前を返している
   }
   ```

2. **リファクタリング後の更新漏れ**:
   ```typescript
   // ❌ 古い実装のコメントが残っている
   // データベースから直接取得する
   function getUsers() {
     return cache.get('users');  // 実際はキャッシュから取得
   }
   ```

3. **対応済みのTODO/FIXME**:
   ```typescript
   // ❌ 対応済みなのにコメントが残っている
   // TODO: エラーハンドリングを追加する
   try {
     // エラーハンドリング実装済み
   } catch (e) {
     handleError(e);
   }
   ```

**判定フロー**:
```
コメントを発見
  ↓
コメントの記述と実装が一致しているか？
  ↓ No → [Critical] コメントを修正または削除すべき
  ↓ Yes → OK
```

**判定**: コメントと実装の乖離は**必ず指摘する**（Critical）

### ステップ2: コメントが英語で書かれているか（高優先度）

**判断基準**:
コメントは **英語で書く**。日本語など他言語で書かれたコメント、英語と他言語が混在したコメントは指摘する。ドキュメンテーションコメント（`///`, `//!`, `/** */`, docstring 等）も含めて全コメントが対象。

**Why（なぜ英語か）**:
- 検索性: grep / IDE 検索で英単語キーワードと一致させやすい
- ツール親和性: Lint、翻訳、AI 補助、コードレビュー bot 等の多くが英語前提
- 一貫性: 識別子（変数名・関数名）は英語なので、コメントも英語で揃えると視線移動の負荷が下がる
- OSS / 外部公開への耐性: 後から英語化するコストは大きい

**チェック項目**:

1. **日本語コメント**:
   ```rust
   // ❌ 日本語
   // Internal エラーは中身の message を外部に出さない方針。

   // ✅ 英語
   // NOTE: Internal errors must not expose their message to clients
   //       to avoid leaking implementation details.
   ```

2. **英語と他言語の混在**:
   ```rust
   // ❌ 英語と日本語の混在
   // NOTE: retry until success （最大3回）

   // ✅ 英語に統一
   // NOTE: retry until success (max 3 attempts)
   ```

3. **ドキュメンテーションコメントも対象**:
   ```rust
   // ❌ doc コメントが日本語
   /// ユーザーを取得する
   pub fn get_user(id: UserId) -> Result<User> { ... }

   // ✅ doc コメントも英語
   /// Fetches a user by ID.
   pub fn get_user(id: UserId) -> Result<User> { ... }
   ```

**判定フロー**:
```
コメントを発見（doc コメント含む）
  ↓
英語のみで書かれているか？
  ↓ No → [Important] 英語に書き換えるべき
  ↓ Yes → OK
```

**例外**:
- 固有名詞（製品名、人名、地名等）はそのまま使ってよい
- ライセンスヘッダで原文が他言語の場合はそのまま許容

**判定**: 非英語コメント・混在コメントは**必ず指摘する**（Important）

### ステップ3: prefix ルール（TODO/NOTE/FIXME/HACK）が守られているか（高優先度）

**判断基準**:
ドキュメンテーションコメント以外のコメントには、prefix を付けて目的を明示すべき。

**prefix の定義**:

| prefix | 用途 |
|--------|------|
| `NOTE` | 補足説明・背景・意図（Why コメント） |
| `TODO` | 将来対応すべきタスク |
| `FIXME` | 既知の問題・暫定対処 |
| `HACK` | ワークアラウンド・意図的な非正規手段 |

**チェック項目**:

1. **prefix のないコメント**:
   ```typescript
   // ❌ prefix なし
   // この処理は重い操作なので非同期にしている
   async function process() { ... }

   // ✅ NOTE prefix
   // NOTE: この処理は重い操作（平均3秒）なので非同期にしている
   async function process() { ... }
   ```

2. **HACK の理由の併記**:
   ```typescript
   // ❌ HACK の理由が不明
   // HACK: setTimeout を使う
   setTimeout(() => { ... }, 0);

   // ✅ HACK の理由を併記
   // HACK: 外部ライブラリのバグ（#1234）により、
   // イベントループの次のティックまで遅延させる必要がある
   setTimeout(() => { ... }, 0);
   ```

**判定フロー**:
```
ドキュメンテーションコメント以外のコメントを発見
  ↓
prefix（TODO/NOTE/FIXME/HACK）があるか？
  ↓ No → [Important] prefix を追加すべき
  ↓ Yes
  ↓
HACK の場合、理由が併記されているか？
  ↓ No → [Important] 理由を併記すべき
```

**判定**: prefix のないコメントは**必ず指摘する**（Important）

### ステップ4: 不要なコメントがないか（中優先度）

**判断基準**:
コード自体で十分明確な内容を繰り返すコメントは、ノイズとなり重要な情報を埋もれさせる。

**チェック項目**:

1. **What（何を）を繰り返すコメント**:
   ```typescript
   // ❌ コードを読めばわかる
   i++;  // increment i
   const user = getUser();  // ユーザーを取得する

   // ✅ コメントなし（コードで明らか）
   i++;
   const user = getUser();
   ```

2. **コメントアウトされたコード**:
   ```typescript
   // ❌ コメントアウトされたコードが残っている
   // const oldLogic = ...;
   // if (oldLogic) { ... }
   const newLogic = ...;

   // ✅ 削除（バージョン管理で復元可能）
   const newLogic = ...;
   ```

3. **自明な doc コメント**:
   ```typescript
   // ❌ 自明な内容
   /**
    * ユーザー名を取得する
    */
   getName(): string { ... }

   // ✅ コメントなし（メソッド名で明らか）
   getName(): string { ... }
   ```

**判定フロー**:
```
コメントを発見
  ↓
コードを読めば明らかな内容か？
  ↓ Yes → [Minor] コメントを削除すべき
  ↓ No → OK
```

**判定**: 不要なコメントは**指摘する**（Minor）

### ステップ5: ドキュメンテーションコメントが適切か（中優先度）

**判断基準**:
公開API・モジュールの入口には、責務の境界と契約を明確にするドキュメンテーションコメントが必要。

**チェック項目**:

1. **責務の境界**:
   ```typescript
   // ❌ 何をするかだけで、何をしないかが不明
   /**
    * ユーザーデータを処理する
    */
   class UserProcessor { ... }

   // ✅ 何をする・何をしないが明確
   /**
    * ユーザーデータを処理する
    *
    * 責務:
    * - ユーザー入力のバリデーション
    * - データ変換・正規化
    *
    * 責務外:
    * - データベースへの保存（Repository の責務）
    * - 外部API呼び出し（Gateway の責務）
    */
   class UserProcessor { ... }
   ```

2. **パラメータの制約**:
   ```typescript
   // ❌ 制約が不明
   /**
    * ユーザーを作成する
    */
   function createUser(email: string, age: number) { ... }

   // ✅ 制約が明確
   /**
    * ユーザーを作成する
    * @param email - メールアドレス（null 不可、形式チェックあり）
    * @param age - 年齢（0-150の範囲、負値不可）
    */
   function createUser(email: string, age: number) { ... }
   ```

3. **例外・エラーの発生条件**:
   ```typescript
   // ❌ エラー条件が不明
   /**
    * ユーザーを取得する
    */
   function getUser(id: string): User { ... }

   // ✅ エラー条件が明確
   /**
    * ユーザーを取得する
    * @throws UserNotFoundError - 指定されたIDのユーザーが存在しない場合
    * @throws InvalidIdError - IDの形式が不正な場合
    */
   function getUser(id: string): User { ... }
   ```

**判定フロー**:
```
公開API・モジュールの入口を発見
  ↓
ドキュメンテーションコメントがあるか？
  ↓ No → [Minor] 追加を検討すべき
  ↓ Yes
  ↓
責務の境界が明確か？
  ↓ No → [Minor] 明確化すべき
```

**判定**: ドキュメンテーションコメントの不足・不明瞭さは**指摘する**（Minor）

### ステップ6: TODO/FIXMEの管理が適切か（低優先度）

**判断基準**:
TODO/FIXMEは技術的負債の可視化であり、適切に管理されなければ蓄積し続ける。

**チェック項目**:

1. **Issue番号の紐付け**:
   ```typescript
   // ❌ Issue番号がない
   // TODO: パフォーマンス改善

   // ✅ Issue番号あり
   // TODO(#1234): パフォーマンス改善
   ```

2. **PRに含まれるTODOの意図確認**:
   - 意図的に後回しにするTODOか？
   - 今のPRで対応すべきではないか？

**判定**: TODO/FIXMEの管理が不適切な場合は**指摘する**（Nit）

## レビュー観点（詳細チェックリスト）

### 1. コメントと実装の整合性

- [ ] コメントの記述と実際の動作が一致しているか
- [ ] リファクタリング後にコメントが更新されているか
- [ ] 対応済みの TODO / FIXME / HACK が削除されているか

### 2. コメントの言語

- [ ] すべてのコメント（doc コメント含む）が英語で書かれているか
- [ ] 英語と他言語が混在していないか
- [ ] 例外（固有名詞・原文ライセンス）以外で他言語が使われていないか

### 3. Prefix ルール

- [ ] ドキュメンテーションコメント以外に prefix（TODO/NOTE/FIXME/HACK）があるか
- [ ] `NOTE` で Why（なぜ）を説明しているか
- [ ] `HACK` に理由が併記されているか

### 4. 不要なコメント

- [ ] コードを読めばわかる内容を繰り返していないか
- [ ] コメントアウトされたコードが残っていないか
- [ ] 自明な getter/setter に形式的な doc コメントがついていないか
- [ ] prefix 付きでも内容が自明なコメントがないか

### 5. ドキュメンテーションコメント

- [ ] 公開 API・モジュールの入口に説明があるか
- [ ] 責務の境界（何をする・何をしない）が明確か
- [ ] パラメータの制約（null 許容、範囲、単位）が記述されているか
- [ ] 戻り値の意味（特に null/empty の場合）が明確か
- [ ] 例外・エラーの発生条件が書かれているか

### 6. TODO / FIXME の管理

- [ ] TODO に Issue 番号やチケット番号が紐づいているか
- [ ] PR に含まれる TODO が意図的なものか
- [ ] 期限や担当が不明な TODO が新たに追加されていないか

## 指摘の出し方

### 指摘の構造

```
【問題点】このコメントは○○の問題があります
【理由】△△だからです
【影響】この問題により××のリスクが発生します
【提案】□□に変更すると改善します
```

### 指摘の例

#### コメントと実装の乖離の指摘

```
【問題点】このコメントと実装が一致していません
【理由】コメントは「ユーザーの年齢を返す」と書かれていますが、実際は名前を返しています
【影響】読み手を誤った理解に導き、バグの原因となります
【提案】コメントを修正するか、関数名を変更してください
  ```typescript
  // Before: 乖離
  // ユーザーの年齢を返す
  function getUserName(user: User): string {
    return user.name;
  }

  // After: 修正
  // ユーザーの名前を返す
  function getUserName(user: User): string {
    return user.name;
  }
  ```
```

#### コメント言語の指摘

```
【問題点】このコメントが英語で書かれていません
【理由】コメントは英語で統一する規約です。日本語コメントは grep / 検索ツール / AI 補助との親和性が低く、識別子（英語）との視線移動も増えます
【影響】保守性が低下し、後から英語化するコストも積み上がります
【提案】以下のように英語で書き直してください
  ```rust
  // Before: 日本語
  // Internal エラーは中身の message を外部に出さない方針。

  // After: 英語
  // NOTE: Internal errors must not expose their message to clients
  //       to avoid leaking implementation details.
  ```
```

#### prefix ルールの指摘

```
【問題点】このコメントに prefix がありません
【理由】ドキュメンテーションコメント以外のコメントには、目的を明示する prefix（TODO/NOTE/FIXME/HACK）が必要です
【影響】コメントの目的が不明瞭で、後から見たときに対応すべきかどうか判断できません
【提案】以下のように NOTE prefix を追加してください
  ```typescript
  // Before: prefix なし
  // この処理は重い操作なので非同期にしている

  // After: NOTE prefix
  // NOTE: この処理は重い操作（平均3秒）なので非同期にしている
  ```
```

#### 不要なコメントの指摘

```
【問題点】このコメントはコード自体で明らかな内容を繰り返しています
【理由】`i++` が何をするかは、コードを読めば明らかです
【影響】不要なコメントはノイズとなり、重要な情報を埋もれさせます
【提案】コメントを削除してください
  ```typescript
  // Before: 不要なコメント
  i++;  // increment i

  // After: コメント削除
  i++;
  ```
```

#### ドキュメンテーションコメントの不足の指摘

```
【問題点】この公開APIにドキュメンテーションコメントがありません
【理由】公開APIは、責務の境界と契約を明確にする必要があります
【影響】利用者がAPIの制約や例外条件を理解できず、誤用のリスクがあります
【提案】以下のようなドキュメンテーションコメントを追加してください
  ```typescript
  /**
   * ユーザーを取得する
   *
   * @param id - ユーザーID（null 不可、UUID 形式）
   * @returns ユーザーオブジェクト
   * @throws UserNotFoundError - 指定されたIDのユーザーが存在しない場合
   * @throws InvalidIdError - IDの形式が不正な場合
   */
  function getUser(id: string): User { ... }
  ```
```

#### TODO の管理の指摘

```
【問題点】この TODO に Issue 番号が紐づいていません
【理由】Issue 番号がないと、対応すべきタスクが追跡できず、技術的負債が蓄積します
【影響】TODO が放置され、いつ、誰が対応するか不明になります
【提案】Issue を作成して番号を紐づけてください
  ```typescript
  // Before: Issue 番号なし
  // TODO: パフォーマンス改善

  // After: Issue 番号あり
  // TODO(#1234): パフォーマンス改善
  ```
```

### 指摘の優先度ラベル

- **[Critical]**: コメントと実装の乖離（誤解やバグの原因）
- **[Important]**: コメント言語が英語でない・混在、prefix ルール違反、HACK の理由不足
- **[Minor]**: 不要なコメント、ドキュメンテーションコメントの不足
- **[Nit]**: TODO の Issue 番号不足

## テストケース

レビュー判断の精度を検証するためのテストケース。

### Case 1: コメントと実装の乖離（Critical指摘）

**コード**:
```typescript
// ユーザーの年齢を返す
function getUserName(user: User): string {
  return user.name;
}
```

**期待される指摘**:
- [Critical] コメントと実装が乖離
- コメントを修正するか、関数名を変更すべき

### Case 2: 非英語コメント（Important指摘）

**コード**:
```rust
// Internal エラーは中身の message を外部に出さない方針。
let (code, message) = match &self { ... };
```

**期待される指摘**:
- [Important] コメントが英語で書かれていない
- 英語に書き換えるべき（同時に prefix も付与）

### Case 2b: prefix ルール違反（Important指摘）

**コード**:
```typescript
// この処理は重い操作なので非同期にしている
async function process() { ... }
```

**期待される指摘**:
- [Important] prefix がない
- `NOTE` prefix を追加すべき

### Case 3: 不要なコメント（Minor指摘）

**コード**:
```typescript
i++;  // increment i
const user = getUser();  // ユーザーを取得する
```

**期待される指摘**:
- [Minor] コードで明らかな内容を繰り返している
- コメントを削除すべき

### Case 4: ドキュメンテーションコメント不足（Minor指摘）

**コード**:
```typescript
// 公開APIだがドキュメンテーションコメントなし
export function getUser(id: string): User { ... }
```

**期待される指摘**:
- [Minor] 公開APIにドキュメンテーションコメントがない
- パラメータの制約、例外条件を追加すべき

### Case 5: TODO の Issue 番号不足（Nit指摘）

**コード**:
```typescript
// TODO: パフォーマンス改善
function process() { ... }
```

**期待される指摘**:
- [Nit] TODO に Issue 番号が紐づいていない
- Issue を作成して番号を追加すべき

### Case 6: 許容されるケース（指摘なし）

**コード**:
```typescript
// NOTE: 外部ライブラリのバグ（#1234）により、
// イベントループの次のティックまで遅延させる必要がある
// 修正されたらこのHACKを削除する
setTimeout(() => { ... }, 0);

/**
 * ユーザーを取得する
 *
 * @param id - ユーザーID（UUID 形式、null 不可）
 * @returns ユーザーオブジェクト
 * @throws UserNotFoundError - ユーザーが存在しない場合
 */
export function getUser(id: string): User { ... }
```

**期待される判断**:
- NOTE で Why が明確
- ドキュメンテーションコメントが適切
- 指摘なし
