# SOLID 原則レビュー

コードレビュー時に SOLID 原則の観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. 単一責任の原則（SRP）

- クラス・モジュールが複数の変更理由を持っていないか
- 「このクラスは何をするものか」を一文で説明できるか
- 変更理由が異なるロジックが同じクラスに同居していないか
  - 例: ビジネスロジックと永続化処理が同じクラスにある
  - 例: バリデーションとフォーマット処理が混在している
- ただし分割のしすぎで「1クラス1メソッド」のような過剰な細分化に注意

### 2. 開放閉鎖の原則（OCP）

- 新しいケースを追加するときに既存コードの修正が必要な構造になっていないか
- 分岐の連鎖（if-else / switch の長い列）がパターンの追加ごとに膨らむ構造になっていないか
- 拡張ポイント（抽象クラス、インターフェース、ストラテジ等）で対処できるか
- ただし「将来の拡張に備えて」の過剰な抽象化は避ける。現時点で 2 つ以上の具象がなければ抽象化は不要

### 3. リスコフの置換原則（LSP）

- サブクラスが親クラスの契約（事前条件・事後条件・不変条件）を守っているか
- オーバーライドしたメソッドが親の期待を裏切る動作をしていないか
  - 例: 親が保証する戻り値の範囲をサブクラスが破る
  - 例: 親では発生しない例外をサブクラスが投げる
- 継承より委譲（コンポジション）のほうが適切なケースを見逃していないか

### 4. インターフェース分離の原則（ISP）

- 太いインターフェースを実装するために、不要なメソッドを空実装や例外で埋めていないか
- インターフェースの利用者が使わないメソッドに依存させられていないか
- 役割ごとに小さなインターフェースに分離できないか

### 5. 依存逆転の原則（DIP）

- 上位モジュールが下位モジュールの具象に直接依存していないか
- 抽象（インターフェース・トレイト）を介して依存しているか
- 依存の方向が「安定した方向」に向いているか（変更頻度が高いものに依存しない）
- DI コンテナや手動注入で外部から依存を注入できる構造か

## よくあるアンチパターン

- **God Class**: 何でもやる巨大クラス（SRP 違反）
- **Switch on Type**: 型による分岐の連鎖（OCP 違反）
- **Refused Bequest**: 継承したが大半のメソッドを使わない・上書きする（LSP/ISP 違反）
- **Service Locator**: グローバルなレジストリから依存を取得する（DIP 違反の一種）

## 指摘の出し方

- 原則名だけでなく「○○の変更と△△の変更が同時に発生しやすい。責務を分けると影響範囲が限定される」のように実害を説明する
- 過剰設計にならないよう、現時点のコードベースの複雑さに見合った提案をする
- 「SOLID 違反」と言い切るのではなく、トレードオフを含めて議論する
