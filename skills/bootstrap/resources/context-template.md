# AGENTS.md テンプレート

bootstrap が生成する **`AGENTS.md` (プロジェクト指示の原典)** の構成テンプレート。プロジェクトの性質に応じて取捨選択する。

## ファイルの役割

- **`AGENTS.md`** — このテンプレートが対象とする原典。Codex CLI の既定ファイルであり、Claude Code 公式 docs も `CLAUDE.md` から `@AGENTS.md` で import する形で連携を推奨している (`AGENTS.md` を共通原典として複数 AI ツールから参照する設計)
- **`CLAUDE.md`** — 先頭に `@AGENTS.md` 1 行のみ。Claude Code 固有の指示があれば下に追記する。本テンプレートとは別途用意する必要はない
- **`GEMINI.md`** — 先頭に `@AGENTS.md` 1 行のみ。Gemini CLI 固有の指示があれば下に追記する (Gemini CLI は `@file.md` 構文で他ファイルを import 可能)
- **`.cursorrules` / `.github/copilot-instructions.md`** — import 構文がないため、bootstrap 時に AGENTS.md の核心部分を派生生成する。冒頭に「派生元: AGENTS.md / 再 bootstrap で再同期」とコメントで明記する

## Quality gates / push 前ローカル検証

### push 前に必ずローカルで全部通す

**リモート CI や AI Review は防壁であって、1 次フィードバックループではない。** 手元で見つけられた指摘をリモートに発見させると、1 件あたり数分〜十数分の往復が指摘数だけ積み上がる。push 前に、そのタスク種別で意味のあるローカル検証を全部通す。

**同種の指摘を 2 ラウンド連続で受けたら全件スイープ。** 個別修正をやめて、その観点で全文を grep / 全箇所突き合わせし、発見した全件を一度に潰す。リモートに同じカテゴリの指摘を 2 度させない。

### タスク種別ごとの最小チェックセット

push 前に最低限通すべき検証。タスクが複数種別にまたがる場合は該当する全種別を実行する。具体的なコマンドは「開発コマンド」節を参照。

| タスク種別 | 最小チェックセット |
|---|---|
| backend 実装変更 | format + lint + 型チェック + 該当範囲のテスト |
| frontend 実装変更 | lint + 型チェック + 該当範囲のテスト + アーキテクチャテスト |
| docs / 設計ドキュメント変更 | 関連他文書との grep 突き合わせ (HTTP ステータス / フィールド名 / エラーコード等の用語統一) |
| OpenAPI / スキーマ変更 | スキーマ lint + 参照元ドキュメント・実装との整合確認 |
| DB マイグレーション変更 | マイグレーション適用 + ORM スキーマ再生成 + 既存テストの再実行 |

[プロジェクトの技術スタックに応じて上記表を具体化する。例: backend = `cargo fmt + cargo clippy + cargo nextest`, frontend = `pnpm lint + pnpm typecheck + pnpm test + pnpm arch:test`, OpenAPI = `redocly lint`, migration = `db-migrate(-test) + db-prepare`]

## AI の判断委任範囲

### 自律判断してよいこと

- 実装の詳細 (関数・変数名、アルゴリズム選択、ライブラリ内部の使い方)
- テストの追加・修正
- リファクタリング (既存テストが全パスする範囲)
- lint/format の修正
- ドキュメントの更新

### 自走すべきこと (ユーザーへの依頼禁止)

settings.json で許可されている読み取り系・調査系コマンドは AI が直接実行する。「実行してください」とユーザーに依頼してはならない。結果はユーザーが見て判断するのではなく、AI が結果から判断して次のステップに進む。

- ログ・実行結果の取得 (`gh run list`, `gh run view --log-failed`, `gh pr view`, `gh issue view` 等)
- リポジトリ状態の確認 (`git log`, `git diff`, `git status`, `git show` 等)
- コードベース調査 (`grep`, `find`, ファイル読み込み、ディレクトリ一覧 等)
- テスト・lint・型チェック等の検証コマンド (許可されているもの)

ユーザー依頼が許されるのは、対話的認証・本番環境への破壊的操作・settings.json で明示的に許可されていないコマンドなど、AI に権限が無い場合に限定する。

### 確認が必要なこと

- 新しい外部依存の追加
- レイヤー構成・アーキテクチャの変更
- API の破壊的変更
- セキュリティに関わる判断
- 要件の解釈に曖昧さがある場合

## コミット規約

### prefix

| prefix | 用途 |
|---|---|
| [feat] | 新機能の追加 |
| [fix] | バグ修正 |
| [update] | 既存機能の改善・拡張 |
| [improve] | 非機能改善 (パフォーマンス等) |
| [refactor] | 動作を変えないコード構造の改善 |
| [chore] | メンテナンスタスク (依存更新、バージョンアップ等) |
| [docs] | ドキュメントのみの変更 |
| [test] | テストのみの追加・修正 |
| [style] | フォーマット、セミコロン等のスタイル修正 |

### 粒度

- 目的単位でステージング。ファイル単位ではなく変更の意図単位
- 1 コミット 1 目的。複数の目的を混ぜない
- **レビュー指摘の一括対応中も目的単位で分割する**。Round や時系列でまとめない。修正→検証→コミット→次のカテゴリ、というサイクルを回す
