## 開発の原則

[philosophy/ から選定した原則。プロジェクト固有の適用方法を含む]

### 基礎原則

- **語彙定義義務**: 構造的な語彙は「動詞+目的語」で 1 文定義できること。禁止語: Service, Manager, Helper, Utility, Processor, Worker, Engine
- **逃げの禁止**: 曖昧な箱への退避、数値構造の文字列化、直交次元の排他列挙化、安易な抽象化を避ける。逃げる場合は ADR に Why を残す
- **捨てやすさ**: 外部依存の影響範囲がコントロール可能であること。隔離戦略 (ラッパー/import制限/レイヤー分離/並存) を対象に応じて適用

### テストとフィードバック

- テストファースト (Red-Green-Refactor)
- バーチカルスライス (end-to-end 貫通を優先、レイヤー積み上げ禁止)
- ユーザー入力の失敗は Result/Either 型で表現。例外はバグ専用

## AI の判断委任範囲

### 自律判断してよいこと

- 実装の詳細 (関数・変数名、アルゴリズム選択、ライブラリ内部の使い方)
- テストの追加・修正
- リファクタリング (既存テストが全パスする範囲)
- lint/format の修正
- ドキュメントの更新

### 自走すべきこと (ユーザーへの依頼禁止)

settings.json で許可されている読み取り系・調査系コマンドは AI が直接実行する。「実行してください」とユーザーに依頼してはならない。結果はユーザーが見て判断するのではなく、AI が結果から判断して次のステップに進む。

- ログ・実行結果の取得 (`gh run list`, `gh run view --log-failed`, `gh pr view`, `gh issue view` 等)
- リポジトリ状態の確認 (`git log`, `git diff`, `git status`, `git show` 等)
- コードベース調査 (`grep`, `find`, ファイル読み込み、ディレクトリ一覧 等)
- テスト・lint・型チェック等の検証コマンド (許可されているもの)

ユーザー依頼が許されるのは、対話的認証・本番環境への破壊的操作・settings.json で明示的に許可されていないコマンドなど、AI に権限が無い場合に限定する。

### 確認が必要なこと

- 新しい外部依存の追加
- レイヤー構成・アーキテクチャの変更
- API の破壊的変更
- セキュリティに関わる判断
- 要件の解釈に曖昧さがある場合

## コミット規約

### prefix

| prefix | 用途 |
|---|---|
| [feat] | 新機能の追加 |
| [fix] | バグ修正 |
| [update] | 既存機能の改善・拡張 |
| [improve] | 非機能改善 (パフォーマンス等) |
| [refactor] | 動作を変えないコード構造の改善 |
| [chore] | メンテナンスタスク (依存更新、バージョンアップ等) |
| [docs] | ドキュメントのみの変更 |
| [test] | テストのみの追加・修正 |
| [style] | フォーマット、セミコロン等のスタイル修正 |

### 粒度

- 目的単位でステージング。ファイル単位ではなく変更の意図単位
- 1 コミット 1 目的。複数の目的を混ぜない
- **レビュー指摘の一括対応中も目的単位で分割する**。Round や時系列でまとめない。修正→検証→コミット→次のカテゴリ、というサイクルを回す
