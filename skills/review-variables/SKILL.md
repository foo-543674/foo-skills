# 変数・定数レビュー

コードレビュー時に変数・定数の取り扱いの観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. マジックナンバー・マジックストリング

- 意味のある数値・文字列がリテラルのままコード中に埋め込まれていないか
- 定数として名前をつけて意図を明示すべきか
- ただし `0`, `1`, `""`, 配列インデックスなど文脈上自明なものは定数化不要

### 2. 不要な変数宣言

- 一度しか使われず、代入元の式がそのまま十分に読みやすい場合、中間変数が冗長になっていないか
- 逆に、複雑な式を分解して中間変数に名前をつけることで可読性が上がるケースを見逃していないか
- 判断基準: 変数名が式に **意味的な情報を追加している** なら残す、単なるエイリアスなら不要

### 3. スコープの適切さ

- 変数のスコープが必要以上に広くないか（使う場所のできるだけ近くで宣言する）
- ループの外で宣言しているが、ループ内でしか使わない変数はないか
- クラスフィールドにしているが、メソッドのローカル変数で十分なものはないか

### 4. ミュータビリティ

- 再代入されない変数に `const` / `final` / `val` / `let`（言語に応じた不変宣言）を使っているか
- ミュータブルなコレクションが本当に必要か（イミュータブルで済む場合はそちらを優先）

### 5. 変数の再利用・上書き

- 一つの変数を異なる目的で使い回していないか（意味が途中で変わる変数）
- ループ変数やアキュムレータを意図せず外側で参照していないか
- シャドウイングが読み手を混乱させていないか

### 6. 定数の管理

- 定数の定義場所は適切か（使用箇所の近く vs 共通定数ファイル）
- 複数箇所で同じリテラルが使われている場合、定数として一元管理すべきか
- 環境依存の値（URL、タイムアウト値等）がハードコードされていないか

## 指摘の出し方

- マジックナンバーは「この `30` は何を意味する？ `MAX_RETRY_COUNT` のような定数にすると意図が明確になる」のように提案する
- 不要な変数は「この変数は直接インラインで書いても可読性は変わらない」と伝える
- スコープの問題は「この変数はブロック内に移動できる」と具体的な移動先を示す
