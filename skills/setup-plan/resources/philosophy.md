# 開発環境構築の思想

インタビューで方針を引き出した後、計画書に具体物を導出する際に参照する判断基準。

## 技術選定の原則

- **要件の最適解を選ぶ**: 自分の既知の技術に縛られない。要件に最も適した技術を選ぶ
- **プロジェクト目的による判断基準の変動**: 仕事 vs 個人、勉強目的 vs 製品開発で選定基準が変わる
  - 製品開発: 安定性・メンテナンス性・チーム習熟度を重視
  - 勉強目的: 新しい技術の習得を重視、多少のリスクは許容
- **ツール化できるドキュメントを積極活用**: OpenAPI、Protocol Buffers 等、コード生成やバリデーションに活用できる形式を優先する

## 開発環境（devcontainer）

- **根本思想: ホストを汚さない**: 言語ランタイム・ツール・依存関係は全て devcontainer に閉じ込める。ホストに直接インストールしない
- **理由**: バージョン競合の回避（Node 20 で開発中に別リポジトリで 24 を使いたい等）、新しい PC でのセットアップコスト排除
- **devcontainer とローカルインフラは疎結合**: Docker ネットワーク共有で接続。devcontainer 起動時にインフラが一緒に立ち上がらない構成にする

## テスト戦略

### 単体テスト
- 細かい条件分岐・ビジネスロジックの検証は単体テストに任せる
- テスタビリティを上げること自体が目的のテストは作らない

### 統合テスト
- **言語非依存を原則とする**: Postman / Newman 等、アプリケーション言語に依存しないツールで書く
- 理由: テストコードがアプリケーション言語と密結合すると、言語移行時にテストも書き直しになる。言語非依存ならテスト資産が技術スタック変更を生き残る

### E2E テスト
- **ユーザーストーリーベースで最小限にする**: E2E は遅く壊れやすいので、ユーザーの主要な操作フローのみ検証する
- 条件分岐の網羅は単体テストに委ねる
- ツールは固定しない（現状は Playwright だが、より良い選択肢が出れば移行する）

## CI パイプライン

- **自動化できるものは全てやる**: lint、型チェック、テスト、セキュリティスキャン等
- **テスタビリティのためだけのテストは作らない**: CI を通すためだけのテストに価値はない
- **実行速度を計測可能にする**: パイプラインのボトルネックを把握し、継続的に改善できるようにする

## レビュー体験

- **成果物はブラウザで確認できる状態にデプロイする**: Storybook、テストレポート、API ドキュメント等
- 理由: レビュアーが pull してローカルで立ち上げないと確認できない成果物は、レビュー品質が下がる。面倒なことは人間はやらなくなる
- CI/CD で成果物を自動デプロイし、PR にリンクを貼る

## API・ドキュメント

- **スキーマファースト**: API 設計はスキーマ定義から始める（OpenAPI, Protocol Buffers 等）
- **ドキュメントは常に最新を保つ**: 自動生成 + CI で最新性を担保する仕組みを入れる
- **人間用 README**: プロジェクトのセットアップ手順・使い方を README に書く
- **AI 用コンテキスト**: `.contexts/` に判断の背景・設計意図を蓄積し、AI が参照できるようにする
