# 命名レビュー

コードレビュー時に命名の観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. 意図が伝わるか

- 名前だけで役割・目的が推測できるか
- 略語を使う場合、チーム内で共通認識がある略語か（`req`, `res`, `ctx` など広く通じるものは OK）
- 一文字変数はループカウンタやラムダの短いブロック内のみ許容

### 2. 文脈に応じた命名か

- 変数名がその処理の文脈での役割を表しているか
- 型名をそのまま変数名にしていないか（例: `getUser()` の返り値を `user` とするのではなく、その処理での役割に応じて `admin`, `operator`, `recipient` 等にする）
- 同じ型の値が複数登場する場面で区別がつく名前になっているか

### 3. 抽象度が適切か

- 具体的すぎる名前が汎用的な処理についていないか（例: `sendEmail` → 実際には通知全般を扱う）
- 逆に、抽象的すぎて何をするかわからない名前になっていないか（例: `process`, `handle`, `doSomething`）
- クラス名に `Service`, `Manager`, `Helper`, `Util` など責務が曖昧になるサフィックスを使っていないか（何でも入る箱になりやすい）

### 4. 一貫性

- 同じ概念に対して同じ単語を使っているか（`user` と `account` の混在など）
- 取得系の命名パターンが統一されているか（`get` / `fetch` / `find` / `retrieve` が混在していないか）
- 真偽値は `is`, `has`, `can`, `should` などの prefix で始まるか
- 真偽値は `true` がポジティブな状態を表すようにする（例: `isEnabled` > `isDisabled`）。ただし広く知れ渡った慣習（HTML の `disabled` 等）や、その処理で逆のほうが明らかに可読性が高い場合は例外
- コレクションは複数形か、または `List`, `Set`, `Map` などのサフィックスがあるか

### 5. スコープに応じた長さ

- スコープが広い（モジュール公開、クラスフィールド）ほど説明的な名前にする
- スコープが狭い（ローカル変数、短いクロージャ）場合は簡潔でよい
- ただし短すぎて意味不明になるのは NG

### 6. 誤解を招かないか

- 否定形の命名を避けているか（`isNotValid` → `isInvalid` or `isValid` の反転）
- 単位や型を暗示する名前が実態と合っているか（`timeoutSec` が実際にはミリ秒など）
- `tmp`, `data`, `info`, `result` など汎用的すぎる名前が安易に使われていないか

## 指摘の出し方

- 「この名前だと○○と誤解しそう。△△のほうが意図が伝わる」のように、理由と代替案をセットで出す
- プロジェクト内の既存の命名パターンに合わせた提案をする
- 好みレベルの指摘は nit として明示する
