# 可読性レビュー

コードレビュー時にコード構造・可読性の観点で指摘すべきポイントを定義する。

## Why: なぜ可読性が重要か

コードの可読性は、開発チームの生産性と品質を決定する最も重要な要素の一つである。

**根本的な理由**:
1. **コードは書く時間より読む時間のほうが圧倒的に長い**: Martin Fowlerによれば「読む時間と書く時間の比率は10:1を超える」。可読性の低いコードは、毎回の読解に余計な時間がかかり、チーム全体の速度を低下させ続ける
2. **認知負荷の削減**: 人間の作業記憶は限られている（7±2チャンク）。複雑なネスト、長い関数、不明瞭な制御フローは認知負荷を増大させ、理解を困難にする。可読性の高いコードは、必要な情報を作業記憶の範囲内に収める
3. **バグの早期発見**: 可読性の高いコードは、レビュー時やコード読解時にバグを発見しやすい。逆に、可読性の低いコードは、バグが複雑さに隠れて見つかりにくく、本番環境で発見されるリスクが高まる
4. **保守性の向上**: 将来の変更時に、コードの意図と動作を素早く理解できることは保守性の基盤。可読性が低いと、変更の影響範囲を見誤り、意図しない副作用を生む
5. **オンボーディングコスト**: 新規メンバーがコードベースを理解するまでの時間は、コードの可読性に大きく依存する。可読性の高いコードは、チームの拡張性を高める

**可読性レビューの目的**:
- コード読解時の認知負荷を最小化する
- バグの発見を容易にする
- 将来の変更を安全かつ高速にする
- チーム全体の生産性を向上させる

## 判断プロセス（決定ツリー）

可読性のレビュー時は、以下の順序で判断する。認知負荷への影響が大きい問題から確認し、問題があれば指摘する。

### ステップ1: ネストの深さが認知限界を超えていないか（最優先）

**判断基準**:
深いネストは認知負荷を指数関数的に増大させる。人間は3段以上のネストを追うのが困難。

**チェック項目**:
1. **ネストの深さを数える**
   - ネスト深さ1-2: 許容範囲
   - ネスト深さ3: リファクタリング検討（コードの複雑さによる）
   - ネスト深さ4以上: **必ず指摘する**

2. **ネストのカウント方法**:
   ```typescript
   function example() {           // レベル0
     if (condition1) {            // レベル1
       for (const item of items) { // レベル2
         if (condition2) {         // レベル3 ← 3段目
           try {                   // レベル4 ← 4段目（指摘）
             // ...
           }
         }
       }
     }
   }
   ```

**判定フロー**:
```
ネスト深さをカウント
  ↓
深さ4以上？
  ↓ Yes → [Important] 必ず指摘
  ↓ No
  ↓
深さ3？
  ↓ Yes
  → コード量は？
      50行以上 → [Important] 指摘
      50行未満 → [Minor] リファクタリング検討を推奨
```

**解消テクニック**:
1. **早期リターン・ガード節**
   ```typescript
   // ❌ ネスト深い
   function process(data) {
     if (data !== null) {
       if (data.isValid) {
         if (data.hasPermission) {
           // 処理
         }
       }
     }
   }

   // ✅ ガード節で浅く
   function process(data) {
     if (data === null) return;
     if (!data.isValid) return;
     if (!data.hasPermission) return;
     // 処理
   }
   ```

2. **関数抽出**
   ```typescript
   // ❌ ネスト深い
   for (const user of users) {
     if (user.isActive) {
       for (const order of user.orders) {
         if (order.isPending) {
           // 複雑な処理
         }
       }
     }
   }

   // ✅ 関数抽出
   for (const user of users) {
     if (user.isActive) {
       processPendingOrders(user.orders);
     }
   }
   ```

3. **条件の反転**
   ```typescript
   // ❌ ネスト深い
   if (user.hasPermission) {
     // 100行の処理
   }

   // ✅ 異常系を先に処理
   if (!user.hasPermission) return;
   // 100行の処理
   ```

### ステップ2: 関数の長さが認知限界を超えていないか（高優先度）

**判断基準**:
長い関数は、全体を把握するために多くの情報を作業記憶に保持する必要があり、認知負荷が高い。

**チェック項目**:
1. **行数による判定**
   - 50行以下: 許容範囲
   - 51-100行: 分割を検討（コードの性質による）
   - 101行以上: **指摘を強く推奨**
   - 200行以上: **必ず指摘する**

2. **段落数による判定**（行数より重要）
   - 空行で区切られた「段落」が3つ以上ある → **分割を推奨**
   - 各段落は独立した意味を持つことが多く、個別の関数に切り出せる

**判定フロー**:
```
関数の行数をカウント
  ↓
200行以上？
  ↓ Yes → [Important] 必ず指摘
  ↓ No
  ↓
100行以上？
  ↓ Yes → [Minor] 分割を推奨
  ↓ No
  ↓
段落（空行区切り）が3つ以上？
  ↓ Yes → [Minor] 分割を検討
```

**分割の判断基準**:
- **意味のある単位で分割する**（短くするためではない）
- 各段落が独立した概念を表しているか
- 分割後の関数名が自然に付けられるか
- 分割により各関数が単一責任になるか

**判定**: 意味のある単位で分割できる場合は指摘する。機械的に行数だけで判断しない。

### ステップ3: 早期リターン・ガード節が適用できないか（高優先度）

**判断基準**:
事前条件チェックが関数の本体と混在すると、正常系の流れが読みにくくなる。

**チェック項目**:
1. **事前条件チェックが冒頭にあるか**
   ```typescript
   // ❌ 事前条件が本体と混在
   function process(data) {
     // 10行の処理
     if (data === null) return;
     // さらに10行の処理
     if (!data.isValid) return;
     // 本来の処理
   }

   // ✅ ガード節を冒頭に
   function process(data) {
     if (data === null) return;
     if (!data.isValid) return;
     // 本来の処理
   }
   ```

2. **不要なelseの存在**
   ```typescript
   // ❌ 不要なelse
   if (condition) {
     return value1;
   } else {
     return value2;
   }

   // ✅ elseを削除
   if (condition) {
     return value1;
   }
   return value2;
   ```

**判定フロー**:
```
関数内のif文を確認
  ↓
事前条件チェックが関数本体と混在？
  ↓ Yes → [Minor] ガード節を冒頭に移動すべき
  ↓ No
  ↓
早期returnの後にelseがある？
  ↓ Yes → [Nit] elseを削除すべき
```

### ステップ4: 制御フローが追いやすいか（中優先度）

**判断基準**:
フラグ変数、複雑な条件式、多数のbreak/continueは、制御フローを追いにくくする。

**チェック項目**:
1. **フラグ変数での制御**
   ```typescript
   // ❌ フラグ変数で制御
   let found = false;
   for (const item of items) {
     if (item.id === targetId) {
       found = true;
       break;
     }
   }
   if (found) {
     // 処理
   }

   // ✅ 直接的な制御
   const found = items.some(item => item.id === targetId);
   if (found) {
     // 処理
   }
   ```

2. **複雑な条件式**
   - 条件式が5つ以上の論理演算子を含む → **変数・関数に切り出す**
   ```typescript
   // ❌ 複雑な条件式
   if (user.age >= 18 && user.hasVerifiedEmail &&
       (user.role === 'admin' || user.role === 'moderator') &&
       !user.isBanned && user.accountCreatedDaysAgo > 30) {
     // 処理
   }

   // ✅ 意味のある名前に切り出し
   const isEligibleUser =
     user.age >= 18 &&
     user.hasVerifiedEmail &&
     (user.role === 'admin' || user.role === 'moderator') &&
     !user.isBanned &&
     user.accountCreatedDaysAgo > 30;

   if (isEligibleUser) {
     // 処理
   }
   ```

3. **多数のbreak/continue**
   - 1つのループに3個以上のbreak/continue → **リファクタリング検討**

**判定**: 制御フローの理解に10秒以上かかる場合は指摘する

### ステップ5: 処理の順序が自然か（中優先度）

**判断基準**:
コードは上から下に読んで理解できる順序であるべき。宣言と使用が離れすぎていたり、無関係な処理が混在すると認知負荷が増す。

**チェック項目**:
1. **宣言と使用の距離**
   - 変数の宣言から使用まで20行以上空いている → **使用箇所の近くに移動**
   ```typescript
   // ❌ 宣言と使用が離れている
   function process() {
     const config = loadConfig();  // ここで宣言
     // 30行の処理
     applyConfig(config);  // ここで使用
   }

   // ✅ 使用箇所の近くで宣言
   function process() {
     // 30行の処理
     const config = loadConfig();
     applyConfig(config);
   }
   ```

2. **関連する処理のまとまり**
   - 無関係な処理が関連処理の間に挟まっている → **関連処理をまとめる**

**判定**: 読む際にスクロールバックが必要になる場合は指摘する

### ステップ6: 式が複雑すぎないか（低優先度）

**判断基準**:
一行に複数の概念を詰め込むと、各ステップの意味が不明瞭になる。

**チェック項目**:
1. **メソッドチェーンの長さ**
   - 4個以上のチェーン → **中間変数を検討**
   ```typescript
   // ❌ 長いチェーン
   const result = data.filter(x => x.active).map(x => x.value).reduce((a, b) => a + b, 0) / data.length;

   // ✅ 中間変数で意味を明示
   const activeItems = data.filter(x => x.active);
   const values = activeItems.map(x => x.value);
   const sum = values.reduce((a, b) => a + b, 0);
   const average = sum / data.length;
   ```

2. **三項演算子のネスト**
   - 2段以上のネスト → **if文に書き換え**

**判定**: 一行を理解するのに複数回読む必要がある場合は指摘する

## レビュー観点（詳細チェックリスト）

上記の判断プロセスを実行するための具体的なチェックリスト。

### 1. ネストの深さ

- [ ] if/for/try のネストが4段以上になっていないか
- [ ] ネスト3段のコードが50行以上になっていないか
- [ ] 早期リターン・ガード節で浅くできないか
- [ ] 条件の反転（異常系を先に処理）で浅くできないか
- [ ] 内側の処理を関数抽出で浅くできないか

### 2. 関数の長さと責務

- [ ] 関数が200行以上になっていないか
- [ ] 関数が100行以上で、分割できる意味のある単位があるか
- [ ] 関数内に3つ以上の段落（空行区切り）があるか
- [ ] 各段落が独立した概念を表し、関数に切り出せるか

### 3. 早期リターン・ガード節

- [ ] 事前条件のチェックが関数の冒頭にあるか
- [ ] 事前条件チェックが関数本体と混在していないか
- [ ] 早期returnの後に不要なelseがないか

### 4. 制御フローの明快さ

- [ ] フラグ変数で制御フローを操作していないか
- [ ] 条件式が5つ以上の論理演算子を含んでいないか
- [ ] 複雑な条件式を意味のある変数・関数に切り出しているか
- [ ] 1つのループに3個以上のbreak/continueがないか

### 5. 処理の順序と流れ

- [ ] 変数の宣言と使用が20行以上離れていないか
- [ ] 関連する処理がまとまっているか
- [ ] 無関係な処理が関連処理の間に挟まっていないか

### 6. 式の複雑さ

- [ ] メソッドチェーンが4個以上連なっていないか
- [ ] 三項演算子が2段以上ネストしていないか
- [ ] 複雑な式を中間変数に分解しているか

## 例外ルール

以下の場合は、通常の可読性ルールの例外として許容する。

### 許容される長い関数

```typescript
// テストコード: Given-When-Then構造で長くなることは許容
test("should process order correctly", () => {
  // Given: 50行のセットアップ
  // When: 実行
  // Then: 50行のアサーション
  // 合計100行超でも許容
});

// ルーティング設定: 宣言的で機械的な列挙
const routes = [
  { path: "/api/users", handler: usersHandler },
  { path: "/api/orders", handler: ordersHandler },
  // 100個のルート定義（許容）
];
```

### 許容される深いネスト

```typescript
// 型定義やデータ構造の宣言（構造の反映）
const config = {
  server: {
    database: {
      connection: {
        host: "localhost"  // ネスト4段でも許容（データ構造の反映）
      }
    }
  }
};
```

### 許容される複雑な条件式

```typescript
// 数学的・論理的に複雑な条件式で、分割すると逆に読みにくい場合
if (x > 0 && x < 100 && y > 0 && y < 100 && x + y < 150) {
  // 座標の範囲チェック: 一連の条件として読むほうが自然
}
```

## 指摘の出し方

### 指摘の構造

```
【問題点】このコードは○○により認知負荷が高くなっています
【理由】△△だからです
【影響】この問題により××の読解・保守コストが発生します
【提案】□□にリファクタリングすると可読性が向上します
```

### 指摘の例

#### ネストの深さの指摘

```
【問題点】このコードはネストが5段になっており、制御フローが追いにくくなっています
【理由】人間は3段以上のネストを追うのが困難で、各レベルの条件を記憶しながら読む必要があります
【影響】コード読解時の認知負荷が高く、バグの見落としやすくなります
【提案】以下のリファクタリングを推奨します
  1. 事前条件チェックをガード節で冒頭に配置（ネスト-1）
  2. 内側のループ処理を `processItem()` 関数に抽出（ネスト-2）
  結果としてネスト深さが2段に削減され、可読性が向上します
```

#### 関数の長さの指摘

```
【問題点】この関数は250行あり、4つの段落（空行区切り）で構成されています
【理由】各段落は独立した概念（バリデーション、データ取得、変換、保存）を表しています
【影響】関数全体を理解するために多くの情報を記憶する必要があり、変更時の影響範囲が不明確です
【提案】各段落を個別の関数に分割することを推奨します
  - `validateInput()`: バリデーション処理
  - `fetchData()`: データ取得処理
  - `transformData()`: データ変換処理
  - `saveData()`: データ保存処理
  各関数が単一責任になり、個別のテストも可能になります
```

#### ガード節の指摘

```
【問題点】事前条件チェックが関数本体と混在しており、正常系の流れが読みにくくなっています
【理由】関数の途中で早期returnが発生すると、どこまでが事前条件チェックで、どこからが本処理かが不明瞭です
【影響】正常系のコードパスを追うために、関数全体を読む必要があります
【提案】すべての事前条件チェックを関数の冒頭にガード節として配置すると、正常系の流れが明確になります
```

### 指摘の優先度ラベル

- **[Important]**: ネスト4段以上、関数200行以上（認知負荷が極めて高い）
- **[Minor]**: ネスト3段、関数100行以上、ガード節の欠如（改善推奨）
- **[Nit]**: 不要なelse、長いチェーン（好みに近い改善提案）

## テストケース

レビュー判断の精度を検証するためのテストケース。

### Case 1: 深いネスト（Important指摘）

**コード**:
```typescript
function processOrders(orders: Order[]) {
  for (const order of orders) {          // レベル1
    if (order.isActive) {                // レベル2
      for (const item of order.items) {  // レベル3
        if (item.inStock) {              // レベル4
          try {                          // レベル5 ← 指摘
            processItem(item);
          } catch (e) {
            logError(e);
          }
        }
      }
    }
  }
}
```

**期待される指摘**:
- [Important] ネストが5段になっている
- ガード節と関数抽出でネストを削減すべき

### Case 2: 長い関数（Important指摘）

**コード**:
```typescript
function handleUserRegistration(data: any) {
  // 50行: バリデーション処理

  // 50行: ユーザーデータの変換処理

  // 50行: データベース保存処理

  // 50行: メール送信処理

  // 合計200行
}
```

**期待される指摘**:
- [Important] 関数が200行あり、4つの独立した処理を含む
- 各処理を個別の関数に分割すべき

### Case 3: ガード節の欠如（Minor指摘）

**コード**:
```typescript
function process(data: Data | null) {
  if (data !== null) {
    if (data.isValid) {
      // 50行の処理
    }
  }
}
```

**期待される指摘**:
- [Minor] ガード節を使用してネストを削減すべき

### Case 4: 許容されるケース（指摘なし）

**コード**:
```typescript
// テストコード: 構造化されており、各セクションの役割が明確
test("should calculate order total correctly", () => {
  // Given: 30行のセットアップ
  const order = createTestOrder();
  const items = createTestItems();
  // ... 設定

  // When: 実行
  const total = calculateTotal(order);

  // Then: 30行のアサーション
  expect(total.subtotal).toBe(100);
  // ... 検証
});
```

**期待される判断**:
- テストコードは長くなることが許容される（Given-When-Then構造）
- 各セクションの役割が明確
- 指摘なし
