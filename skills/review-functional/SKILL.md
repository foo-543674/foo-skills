# 関数型指向設計レビュー

コードレビュー時に関数型プログラミングの観点で指摘すべきポイントを定義する。純粋な関数型言語に限らず、マルチパラダイム言語で関数型のアプローチを取り入れる場面にも適用する。

## レビュー観点

### 1. 純粋関数

- 同じ入力に対して常に同じ出力を返すか
- 関数内で外部状態の読み書き（グローバル変数、DB、ファイル等）をしていないか
- 純粋にできる関数が不必要に副作用を持っていないか
- 純粋な計算ロジックと副作用が混在している場合、分離できないか

### 2. イミュータビリティ

- データの変更が破壊的（in-place mutation）になっていないか
- コレクションの操作で元のデータを壊していないか（`map` / `filter` で新しいコレクションを返す）
- 共有される状態がミュータブルになっていないか（並行処理でのバグの温床）
- ただしパフォーマンスが重要なホットパスでは、局所的なミュータブル操作は許容する

### 3. 副作用の分離

- 副作用（I/O、ログ、DB操作、外部API呼び出し等）がビジネスロジックから分離されているか
- 「計算」と「実行」のフェーズが明確に分かれているか
  - 例: まず何をすべきかを決定（純粋）し、その後に実行（副作用）する
- 副作用が関数のシグネチャから読み取れるか（型、命名、アノテーション等）

### 4. データ変換パイプライン

- 命令的なループ（for + 手動 push）を宣言的な変換（map / filter / reduce）に置き換えられないか
- パイプラインの各ステップが明確な責務を持っているか
- 中間コレクションの生成がパフォーマンス上問題にならないか（大量データの場合は遅延評価やイテレータを検討）
- メソッドチェーンが長すぎて可読性を損なっていないか（適度に中間変数を挟む）

### 5. 高階関数の適切な使用

- コールバックやラムダが複雑すぎないか（長いラムダは名前付き関数に切り出す）
- 高階関数の乱用で逆に読みにくくなっていないか
- 部分適用やカリー化が意味のある場面で使われているか（テクニックのための使用を避ける）

### 6. null / 例外の代替

- null チェックの連鎖を Optional / Maybe / Result 型で置き換えられないか
- 例外を制御フローに使っていないか（Expected な失敗は戻り値の型で表現）
- ただし言語やフレームワークの慣習に反する場合は無理に適用しない

## 指摘の出し方

- 「このループは map + filter で宣言的に書ける」のように具体的な変換案を示す
- 純粋関数化の提案では、副作用をどこに移動すべきかも合わせて示す
- 関数型にすること自体を目的化しない。可読性・保守性が上がる場合にのみ提案する
