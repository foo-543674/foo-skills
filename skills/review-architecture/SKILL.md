# アーキテクチャ設計レビュー

コードレビュー時にアーキテクチャ設計の観点で指摘すべきポイントを定義する。

Clean Architecture の依存ルールと CQRS の Command/Query 分離を基本原則とする。

## レビュー観点

### 1. レイヤー構造と依存方向

クリーンアーキテクチャの本質は特定のレイヤー名（Entity, Use Case 等）ではない。**変化の頻度と主導権**がレイヤー設計に反映され、各レイヤーが抽象を介した一方通行の依存になっているかが重要。

**レイヤー設計の原則**:
- 変化の頻度が低い（安定した）層に向かって依存しているか
- ビジネスの主導権を持つ層が、技術的詳細に依存していないか
- 各レイヤーの境界が抽象（インターフェース等）で定義されているか
- レイヤー間の結合点が最小限になっているか。理想は、あるレイヤーの実装を丸ごと捨てて新規実装に差し替えられること（例: RDBMS 用 Repository を全削除し DynamoDB 用 Repository を新規実装するだけで済む）

**レビュー時の確認手順**:
1. 対象プロジェクトのディレクトリ構造・モジュール分割を確認する
2. プロジェクトが採用しているレイヤー構成を把握する（レイヤー名はプロジェクトごとに異なる）
3. そのレイヤー構成の上で、以下を確認する:
   - 依存の方向が一方通行になっているか（双方向依存・循環依存がないか）
   - 変化の頻度が高い層（UI、外部サービス連携等）が、安定した層に依存しているか
   - 安定した層が、変化の頻度が高い層の具象を import していないか
   - レイヤーをまたぐデータの受け渡しに適切なデータ構造（DTO 等）を使っているか

### 2. ユースケース設計

ユースケースが「アプリケーションが何をするか」を明確に表現しているか。

- 1 ユースケース 1 クラス（または 1 関数）になっているか
- ユースケースの名前がビジネス操作を表しているか（`CreateOrder`, `ApproveRequest` 等）
- ユースケースが UI やプレゼンテーションの関心事を含んでいないか
- ユースケースが直接 DB クエリやHTTP リクエストを発行していないか（Repository / Gateway 経由か）
- ユースケース間の呼び出しが発生していないか（共通ロジックはドメイン層に抽出）

**ユースケースとテストの関係**:

適切に設計されたユースケースは「依存の組み立てと手順の記述」に徹しており、それ自体にテストすべきロジックがほぼ残らない。ユースケースのテストを書こうとしたとき、大量のスタブを用意してモックの呼び出し順を検証するだけになるなら、それはユースケースの設計が正しい兆候。

- ユースケース内に条件分岐やデータ変換のロジックが埋まっていないか（あるならドメイン層に移すべき）
- ユースケースのテストが「モックを正しい順序で呼んでいるか」の検証に終始していないか（終始しているなら、そのテスト自体が不要である可能性が高い）
- テストを書きたくなるほどロジックがユースケースにあるなら、責務の配置を見直す

### 3. 複数集約をまたぐドメイン操作の設計

複数の Entity や集約をまたぐ操作が出てきた場合、安易に `XxxService` を作るのではなく、その操作自体をドメインの概念として設計する。

- 複数集約をまたぐ操作が `Service` という汎用的な名前のクラスに押し込められていないか。その操作にはビジネス上の名前があるはず（承認ワークフロー、注文処理フロー等）
- そのような操作は Workflow, Process, Saga 等の名前でドメイン層にクラスや関数として表現し、操作自体をドメインの第一級の概念にする
- ユースケース（アプリケーション層）とは異なり、Workflow はドメインロジックそのものである。分岐・ルール・判断を含み、テストが必要になる
  - ユースケースは「依存の組み立てと手順の記述」に徹し、ロジックがほぼない
  - Workflow は複数集約を協調させるドメインロジックであり、条件分岐やビジネスルールを持つ
- ただし集約間の操作が単純な委譲の連鎖に過ぎない場合は、ユースケース内で直接記述しても問題ない。Workflow として切り出すのはロジックの複雑さが一定以上ある場合

### 4. Command / Query の分離（CQRS）

CQRS を採用したコンテキストでは、Command 側と Read 側でレイヤー構成が完全に分離される。両者は設計思想が根本的に異なる。

**Command 側 — 適切なレイヤー設計と正規化**:
- クリーンアーキテクチャ等による適切なレイヤー分離が行われているか
- 副作用を持つ操作が明示的に Command として表現されているか
- Command の処理フローが明確か（バリデーション → ビジネスルール適用 → 永続化 → イベント発行）
- Command 側でのデータ参照（バリデーション時の存在チェック等）は、ユースケースとしての "Read" ではなく、ドメインロジックの一部。Command 側の Repository 等の IO 抽象を経由してアクセスする
- Command 側の参照はドメインロジックで使われるだけなので、参照条件は常に同一になるはず。汎用的なクエリメソッドは不要で、目的特化のメソッドで十分。抽象化していてもパフォーマンス最適化が容易

**Read 側 — ストレージの能力を活かす薄い構造**:
- Read 側は過度な抽象化を行わない。ストレージの能力を最大限活かす設計が望ましい
- 極論、1 レイヤー（ハンドラから直接クエリ）でも最適
- ユースケースとしての Read は、ユーザーが入力した検索条件から動的にクエリが組み立てられる可能性が高い。Repository 等で抽象化すると、かえって保守性が下がる
- 読み取り専用の操作が副作用を持っていないか
- 複雑な集計や検索が Command 側のドメインモデルを無理に使っていないか（専用の Read Model を使っているか）

**CQRS の適用判断**:
- そもそもこのコンテキストに CQRS の適用が最適か。書き込みと読み取りの要件が明確に異なるか
- 単純な CRUD で十分なコンテキストに CQRS を持ち込んでいないか

### 5. 境界の定義と防腐層

コンテキスト境界（Bounded Context）が意識されているか。外部システムとの接点に防腐層があるか。

- 異なるドメインコンテキスト間で同じモデルを共有していないか
- 外部 API・ライブラリとの接点に Adapter / Anti-Corruption Layer があるか
- 外部サービスの型やエラーがドメイン層まで漏れていないか
- 境界をまたぐデータ変換が一箇所にまとまっているか（散在していないか）

### 6. 依存の注入と制御の逆転

上位モジュールが下位モジュールの具象に依存せず、抽象を介して接続されているか。

- Repository, Gateway 等の外部接続がインターフェース（抽象）として定義されているか
- ユースケースがインターフェースに依存し、具象は外部から注入されているか
- テスト時にモックやスタブへの差し替えが容易な構造か
- DI の設定（Composition Root）が一箇所にまとまっているか

### 7. エラーと例外の境界設計

レイヤーをまたぐエラーの伝播が設計されているか。

- ドメイン層のエラーがドメインの言葉で表現されているか（`OrderNotFound` 等、HTTP ステータスコードではない）
- Infrastructure 層の例外（DB 接続エラー等）がドメイン層まで漏れていないか
- レイヤー境界でのエラー変換が行われているか
- ユースケースが返すエラーの種類が明確に定義されているか

## よくあるアンチパターン

- **Leaky Abstraction**: Repository インターフェースに SQL や ORM 固有の概念が漏れている
- **Anemic Domain Model**: Entity がデータの入れ物に過ぎず、ビジネスロジックがすべてユースケースにある
- **God Use Case**: 1つのユースケースが複数のビジネス操作をまとめて実行している
- **God Service**: 複数集約をまたぐ操作が `XxxService` に集約され、ドメインロジックの置き場所が曖昧になっている
- **Shared Kernel の肥大化**: 境界間の共有モデルが増え続け、実質的に境界がなくなっている
- **Query in Command**: Command の処理中に別の画面表示用データを組み立てている
- **Framework Coupling**: ドメイン層やアプリケーション層にフレームワークのモジュール・アノテーション・デコレータが入り込んでいる。これらの層はピュアな言語仕様だけで実装されているのが理想。言語自体の機能が弱い場合に Result 型等の汎用ユーティリティライブラリを使うのは許容されるが、フレームワーク固有の依存は持ち込まない

## 指摘の出し方

- 「Use Case が直接 ORM のクエリビルダを呼んでいる。Repository インターフェースを介すと、DB 変更時に Use Case の修正が不要になる」のように依存方向の問題と実害を具体的に説明する
- 過剰な分離を推奨しない。単純な CRUD に Clean Architecture のフルレイヤーを強制しない
- 「この規模・複雑さならこの程度の分離で十分」というトレードオフの判断も示す
- CQRS の厳密な分離は、読み書きの要件が明確に異なる場合にのみ推奨する
