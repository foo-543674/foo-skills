# コンポーネント指向設計レビュー

コードレビュー時にコンポーネント指向設計の観点で指摘すべきポイントを定義する。

コンポーネント指向の本質は **Props による状態制御の公開** にある。オブジェクト指向の「状態のカプセル化」とは異なり、コンポーネントは「親から受け取った状態を映し出すディスプレイ」として機能する。

## レビュー観点

### 1. Container & Presentational の分離

コンポーネントを **描画（Presentational）** と **接続（Container）** に分離しているか。

**Presentational コンポーネント**:
- State を保有していないか
- Context を使用していないか（ただし Compound パターンは例外。`Accordion.Header` のように親コンポーネント名がプレフィックスになっていて依存関係が名前から自明であり、ドキュメントコメントに依存関係が明記されていれば許容する）
- すべての見た目の状態が Props で制御可能か
- ビジネスロジックや API 呼び出しが含まれていないか
- ユーザー操作はコールバック Props で親へ通知しているか

**Container コンポーネント**:
- return で返すのは原則として単一の Presentational コンポーネントか
- ロジック層と Presentational の結合に徹しているか（API 呼び出しやロジック処理自体はロジック層に委譲する）
- 複雑な条件分岐による描画が Container 内に漏れていないか
- ロジック層がインフラの知識（Cookie、localStorage、ルーター等）に直接依存していないか（Page から注入されているか）

**Page コンポーネント**:
- パスパラメータやクエリパラメータのパースを担当しているか
- ルーターライブラリを意識した配置・設定を行っているか
- Cookie、localStorage、その他インフラへのアクセスを抽象化（関数化）し、Container に引き渡しているか
- これにより Container が呼ぶ Hooks からインフラの知識が排除されているか

### 2. Presentational コンポーネントのレイヤー

Presentational コンポーネントが以下の 4 層のどこに属するか意識されているか:

1. **基本 UI 部品**: ボタン、入力フィールド等
2. **アニメーション**: children の動作方法のみを制御
3. **レイアウト**: 配置・間隔の管理。内容には関与しない
4. **複合 UI**: 上記を組み合わせた画面単位のコンポーネント

- 異なるレイヤーの責務が一つのコンポーネントに混在していないか

### 3. Props 設計と状態制御

- 子コンポーネントが独立した内部状態を持っていないか（外部から制御できない状態は避ける）
- `$refs` 等を使った手続き的な UI 操作になっていないか
- 不必要な Props が含まれていないか（必要な Props であれば数は問題ではない）
- バケツリレーを避けるために Context 等で暗黙の依存を作っていないか（Props の明示的な受け渡しが原則）

**Pseudo State（UI 密接な状態）への対応**:
- ホバー、フォーカス等は CSS 疑似クラスで制御できないか
- 外部制御と内部状態の両立が必要な場合、ハイブリッドパターンを使っているか（Props があれば Props を優先、なければ内部状態で動作）

### 4. ロジック層との役割分担

ロジック層（React: Hooks、SolidJS: Signal/createResource、Vue: Composables 等）と Container パターンは協調関係にあり、代替関係ではない:

- **ロジック層**: ロジックの整理と再利用
- **Container**: ロジックと描画の結合ポイント
- **Presentational**: UI の再利用

確認ポイント:
- 「ロジック層があるから Container は不要」と誤解してロジックと描画が混在していないか
- ロジックがロジック層に抽出され、Container がそれを呼び出し、結果を Presentational に渡す構造になっているか

### 5. 再利用性と捨てやすさ

コンポーネント指向の目的は **関心の分離** であり、各レイヤーを独立して置き換え可能にすること。

- 汎用コンポーネントとドメイン固有コンポーネントが明確に分離されているか
- 特定のフレームワークや技術に強く結合しすぎていないか（塊で捨てやすいか）
- 過度な結合によって、一部の変更が広範囲に波及する構造になっていないか

### 6. 副作用の管理

- 副作用（API 呼び出し、購読、タイマー等）が Presentational に漏れず Container / ロジック層に集約されているか
- 副作用の開始と後片付けが対になっているか（メモリリーク防止）
- 副作用の依存関係が正しく宣言されているか

### 7. テスト方針との整合

この設計が守られていれば、テスト方針は自然に決まる:

- **Presentational**: 描画ロジックしかないので Storybook 等のカタログ化と簡単な見た目のテストで十分。単体テストを書きたくなるなら責務が漏れている兆候
- **ロジック層**: 単体テストだけでよい（インフラの知識が排除された純粋なロジックなので）
- **Container / Page**: API モックサーバーを利用した統合テストでよい

レビューで確認すること:
- Presentational に対して複雑な単体テストが書かれていないか（ロジックが混入している兆候）
- ロジック層のテストが描画やインフラに依存していないか
- Container / Page に対して単体テストを書こうとしていないか（統合テストが適切）

## 指摘の出し方

- 「このコンポーネントは State を持ちつつ描画もしている。Container と Presentational に分離すると Presentational が再利用可能になる」のようにメリットを添える
- 内部状態を持つコンポーネントには「この状態を Props で受け取れるようにすると、テストや再利用が容易になる」と伝える
- 設計の問いかけ:「なぜこのコンポーネントにこの責務があるのか？」で関心の分離が十分か確認する
