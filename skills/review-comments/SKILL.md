# コメントレビュー

コードレビュー時にコメントの観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. Prefix ルール

ドキュメンテーションコメント以外のすべてのコメントには、以下の prefix のいずれかを付ける:

| prefix | 用途 |
|--------|------|
| `TODO` | 将来対応すべきタスク |
| `NOTE` | 補足説明・背景・意図（Why コメント） |
| `FIXME` | 既知の問題・暫定対処 |
| `HACK` | ワークアラウンド・意図的な非正規手段 |

- prefix のないコメントはレビューで指摘する
- `NOTE` は Why コメントの標準形。なぜそうしているかの説明に使う
- `HACK` はなぜその手段を取らざるを得ないかの理由も併記する

### 2. 不要なコメントがないか

- コードを読めばわかることをそのまま繰り返していないか（例: `i++ // increment i`）
- コメントアウトされたコードが残っていないか（バージョン管理で復元できる）
- 自明な getter/setter に形式的な doc コメントがついていないか
- prefix 付きでも内容が自明なら不要（`// NOTE: ユーザーを取得する` → コードで明らか）

### 3. コメントが嘘をついていないか

- コードの実際の動作とコメントの記述が乖離していないか
- リファクタリング後にコメントが更新されず古い情報のままになっていないか
- 対応済みの TODO / FIXME / HACK が削除されずに残っていないか

### 4. ドキュメンテーションコメント

- 公開 API・モジュールの入口に説明があるか
- 説明から責務の境界が読み取れるか（何をするものか、だけでなく **何をしないものか** がわかるか）
- 将来的にどう発展しうるか、どのような変更が責務超過に当たるかが読み取れるか
- パラメータの制約（null 許容、範囲、単位）が記述されているか
- 戻り値の意味（特に null/empty の場合の意味）が明確か
- 例外・エラーの発生条件が書かれているか

### 5. TODO / FIXME の扱い

- TODO には対応する Issue 番号やチケット番号が紐づいているか
- 期限や担当が不明な TODO が新たに追加されていないか
- PR に含まれる TODO が意図的なものか確認する

## 指摘の出し方

- 不要なコメントには「コード自体が十分明確なのでコメントは削除してよい」と伝える
- コメント不足には「ここは○○の理由で Why コメントがあると読み手が助かる」のように具体的に伝える
- コメントと実装の乖離は即座に指摘する（バグの温床になる）
