# コンポーネント設計

要件とアーキテクチャをもとに、モジュール・クラス・関数の分割とインターフェースを設計する。

review-component、review-solid、review-functional の観点を設計時に先取りする。UI コンポーネントに限らず、バックエンドのモジュール分割にも適用する。

## プロセス

### 1. 既存コンポーネント構造の把握

既存システムがある場合、現在の分割・構成パターンを理解する。

**確認すべきこと**:
- ディレクトリ構造とファイル分割の基準
- コンポーネント・モジュールの命名規則
- 共通コンポーネント・ユーティリティの構成
- 依存関係のパターン（DI の方式、import の構造）
- テストの配置パターン

### 2. 責務の分解

要件を責務単位に分解し、各責務をコンポーネントに割り当てる。

**分解の原則**:
- 1 コンポーネント 1 責務（単一責任原則）
- 「このコンポーネントは何をするか」を 1 文で説明できるか
- 変更理由が異なるものは別コンポーネントに分ける
- ただし、責務の分離は粒度が重要。過度に分割すると全体像が掴みにくくなる

**分解の軸**:
- ビジネスロジック vs インフラストラクチャ
- 状態管理 vs 表示（UI の場合）
- 入力バリデーション vs データ変換 vs 永続化
- 共通処理 vs ドメイン固有処理

### 3. インターフェースの設計

各コンポーネントの公開インターフェースを定義する。

**設計原則**:
- インターフェースは「使う側」の視点で設計する（実装の詳細を漏らさない）
- 入力パラメータは必要最小限にする
- 戻り値の型で成功・失敗を明確に表現する（Result 型等）
- 副作用を持つ関数と純粋関数を明確に区別する

**インターフェース分離（ISP）**:
- クライアントが使わないメソッドへの依存を強制しない
- 大きなインターフェースは、用途別に分割する
- ただし、分割しすぎるとインターフェースが乱立する。実際に異なるクライアントが存在する場合に分割する

### 4. 依存関係の設計

コンポーネント間の依存関係を設計する。

**依存方向の原則**:
- 安定した層（ビジネスロジック）に向かって依存する
- 不安定な層（UI、外部サービス）から安定した層を参照する
- 循環依存を作らない
- 依存の注入（DI）により、具象ではなく抽象に依存する

**依存グラフの確認**:
- 依存の方向が一方通行か
- 双方向依存・循環依存が発生していないか
- テスト時にモック・スタブに差し替え可能か

### 5. 状態管理の設計（UI コンポーネントの場合）

UI コンポーネントの状態管理方針を設計する。

**Page / Container / Presentational の役割分担**:

- **Page**: ルーティングとインフラの境界。パスパラメータ・クエリパラメータのパース、Cookie・localStorage 等のインフラアクセスを抽象化（関数化）し、Container に引き渡す
- **Container**: ロジックと描画の結合。Hooks を呼び出し、結果を Presentational に渡す。ロジックや API 呼び出し自体は Hooks に委譲する。return で返すのは原則として単一の Presentational コンポーネント
- **Presentational**: Props を受け取り表示するだけ（純粋関数的）。State や Context を保有しない。ユーザー操作はコールバック Props で親へ通知する
- **Hooks**: ロジックの整理と再利用。Page からインフラの知識が注入されるため、Hooks 自体はインフラに直接依存しない

**状態の配置**:
- 状態はそれを使うコンポーネントにできるだけ近く配置する
- 複数コンポーネントで共有する状態のみ、上位に持ち上げるかグローバルストアに置く
- サーバー状態（API レスポンス）とクライアント状態（UI の状態）を区別する

### 6. 拡張ポイントの設計

将来の変更・拡張に備えたポイントを最小限設計する。

- 開放閉鎖原則（OCP）: 拡張に対して開かれ、修正に対して閉じた設計
- ただし、現時点で拡張の見通しがないポイントに抽象化を入れない
- 「3回同じパターンが出たら抽象化する」（Rule of Three）

拡張ポイントの判断でトレードオフが発生した場合は ADR に記録する。

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- コンポーネント分割の粒度判断（統合 vs 分割）
- 状態管理方式の選択（ローカル状態 vs グローバルストア）
- DI コンテナの導入・不導入の判断
- 共通コンポーネントの抽出基準の決定
- デザインパターンの適用判断

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **God Component**: 1 つのコンポーネントに複数の責務が集中する
- **Prop Drilling**: 深いコンポーネントツリーを通じてデータを受け渡す
- **Premature Abstraction**: 1 回しか使わないのに汎用コンポーネントを作る
- **Tight Coupling**: コンポーネント間が具象クラスで直接結合している
- **Feature Envy**: あるコンポーネントが別のコンポーネントのデータに過度にアクセスする
- **Circular Dependency**: A → B → C → A のような循環依存
