# コンポーネント設計

要件とアーキテクチャをもとに、モジュール・クラス・関数の分割とインターフェースを設計する。

review-component、review-solid、review-functional の観点を設計時に先取りする。UI コンポーネントに限らず、バックエンドのモジュール分割にも適用する。

## Why: なぜコンポーネント設計が重要か

コンポーネント設計の品質は、表示の変更容易性・ロジックの再利用性・フレームワーク移行の可能性を根本的に決定する。宣言的 UI（状態から表示を導く）の原則に基づき、表示とロジックを明確に分離することが、フロントエンド開発の成否を分ける。

**根本的な理由**:
1. **表示とロジックの分離（宣言的 UI の実現）**: 宣言的 UI では、表示（Presentational）は「状態をどう表示するか」のみを記述し、ロジック（ロジック層）は「状態をどう管理するか」を記述する。この分離により、表示の変更がロジックに影響しない。混在すると、デザイン変更のたびにロジックを含む全体をテストし直す必要があり、修正コストが膨大になる。宣言的 UI の原則を守れば、表示は捨てて作り直せ、ロジックは保護される
2. **捨てやすさの確保（フロントエンドの技術変遷の早さに対応）**: フロントエンドの技術は数年で大きく変わる（jQuery の命令的 UI → React/Vue の宣言的 UI → SolidJS の細粒度リアクティビティ等）。純粋な Presentational コンポーネント（Props を受け取り JSX を返すだけ）は、宣言的 UI の最も純粋な形であり、いつでも捨てて作り直せる。State・副作用を持つと、技術的負債が蓄積し、新しい技術への移行が困難になる。捨てやすいコンポーネントは、技術変遷に追従でき、UI の大胆な変更を可能にし、デザインの進化を阻害しない
3. **ロジックの再利用（状態管理の一元化）**: 宣言的 UI では、表示は「状態の関数」として表現される。ロジックを状態管理としてロジック層に集約すれば、異なる表示（Web、モバイル、管理画面等）で同じ状態管理ロジックを再利用できる。表示にロジックが混在すると、同じロジックが各表示に重複し、修正漏れ・不整合が発生する。ロジック層により、状態管理は一箇所で管理される
4. **フレームワーク移行の可能性**: 宣言的 UI を採用する現代のフレームワーク（React、Vue、SolidJS 等）は、表示とロジックの分離を前提としている。この原則を守れば、フレームワーク変更時に、ロジック層（状態管理）は変更不要で、表示層（宣言的 UI の記述）のみ書き直せばよい。混在すると、ロジックも含めて全体を書き直す必要があり、移行コストが膨大になる。分離により、フレームワークへの依存を表示層に限定できる
5. **認知負荷の削減（宣言的 UI の明確さ）**: 宣言的 UI では、「状態 → 表示」の流れが一方向で明確。Container にロジックを書かず、ロジック層に委譲すれば、「Container は橋渡し」「ロジック層は状態管理」「Presentational は状態の表示」と役割が明確になり、理解が容易。Container にロジックが混在すると、どこに何があるか不明確で、認知負荷が高い。役割の明確化により、チームの生産性が向上する

**コンポーネント設計の目的**:
- 宣言的 UI の原則に基づき、表示とロジックを分離する
- Presentational を純粋な「状態の関数」にし、捨てやすさを確保する
- ロジックを状態管理としてロジック層に集約し、再利用性を高める
- フレームワーク移行を可能にし、技術的な柔軟性を確保する
- 「状態 → 表示」の一方向の流れを明確にし、認知負荷を削減する

## 判断プロセス（決定ツリー）

コンポーネント設計時は、以下の順序で判断する。フロントエンドの場合は「表示とロジックの分離」「捨てやすさ」を最優先。バックエンドのモジュール分割では責務の明確化を優先する。

### ステップ1: UIコンポーネントの役割分担（UI設計の最優先）

**判断基準**:
宣言的 UI の原則に基づき、表示（Presentational）とロジック（ロジック層）を明確に分離し、Container は橋渡しに徹する。

**チェック項目**:

1. **Page / Container / Presentational / ロジック層の役割**:
   - **Page**: ルーティングとインフラの境界
     - パスパラメータ・クエリパラメータのパース
     - Cookie・localStorage等のインフラアクセスを抽象化（関数化）してContainerに引き渡す

   - **Container**: ロジック層と Presentational の橋渡し
     - ロジック層を呼び出す
     - 状態を取得し、Presentational に渡す
     - **ロジックや API 呼び出し自体はロジック層に委譲**（Container にロジックを書かない）
     - return で返すのは原則として単一の Presentational コンポーネント

   - **Presentational**: Props を受け取り表示するだけ（宣言的 UI の純粋な形）
     - **「状態をどう表示するか」のみを記述**（命令的な DOM 操作をしない）
     - State を保有しない
     - Context も原則使わない（Compound パターン例外あり）
     - ユーザー操作はコールバック Props で親へ通知

   - **ロジック層**: ロジックの実体（状態管理）
     - ビジネスロジック・API 呼び出し・状態管理の実装
     - **「状態をどう管理するか」を記述**
     - Page からインフラの知識が注入される
     - フレームワークごとに実現手段が異なる（React: Hooks、SolidJS: Signal、Vue: Composables）

**判定フロー**:
```
UIコンポーネントか？
  ↓ No → ステップ4（バックエンドモジュール）へ
  ↓ Yes
  ↓
ロジックを含むか？
  ↓ No → Presentational コンポーネント
  ↓ Yes
  ↓
インフラ（Cookie、localStorage等）にアクセスするか？
  ↓ Yes → Page コンポーネント（インフラを抽象化してContainerへ）
  ↓ No
  ↓
ロジックを実装しているか？
  ↓ Yes → ❌ NG: Container にロジックを書いている
             → ロジック層に抽出し、Container は呼び出すだけに
  ↓ No（ロジック層を呼び出しているだけ）→ ✅ Container
```

**判定**: Container にロジックを書かず、ロジック層に委譲する

### ステップ2: Presentationalコンポーネントの設計（UI設計・高優先度）

**判断基準**:
宣言的 UI の原則を徹底し、「状態 → 表示」の関数として設計する。「捨てやすさ」を重視。

**チェック項目**:

1. **宣言的 UI の原則**:
   - **「何を（What）表示するか」のみを記述**し、「どう（How）描画するか」は記述しない
   - Props を受け取り、JSX を返すだけ（`UI = f(state)` の実現）
   - 命令的な DOM 操作（`element.style.display = 'none'` 等）をしない
   - 同じ Props なら同じ結果を返す（純粋関数）

2. **純粋性の確認**:
   - State を持っていないか
   - 副作用（API 呼び出し、localStorage アクセス等）を持っていないか
   - ユーザー操作は「こうなった」と親に通知するだけ（コールバック Props）

3. **捨てやすさの確認**:
   - このコンポーネントを削除しても、他のコンポーネントに影響しないか
   - ロジックが外部（ロジック層）にあるため、表示だけ変更できるか

**判定フロー**:
```
Presentationalコンポーネントを設計
  ↓
宣言的UIか？（「何を表示するか」のみ記述）
  ↓ No（命令的DOM操作がある）→ ❌ NG: 宣言的に書き直す
  ↓ Yes
  ↓
State を持っているか？
  ↓ Yes → ❌ NG: State は Container またはロジック層へ
  ↓ No
  ↓
副作用を持っているか？
  ↓ Yes → ❌ NG: 副作用はロジック層へ
  ↓ No → ✅ OK（純粋な宣言的UIコンポーネント）
```

**判定**: Presentational は `UI = f(state)` の関数として設計し、捨てやすくする

### ステップ3: ロジック層の抽出（UI設計・高優先度）

**判断基準**:
宣言的 UI における「状態管理」をロジック層に集約し、再利用可能にする。Presentational が「状態 → 表示」なら、ロジック層は「入力 → 状態」を担う。

**チェック項目**:

1. **ロジック層に抽出すべきもの（状態管理の実体）**:
   - ビジネスロジック（計算、条件分岐、状態遷移）
   - API 呼び出し（fetch、axios 等）→ 結果を状態に反映
   - 状態管理（useState、Signal、Ref 等）
   - 副作用（useEffect、createEffect 等）→ 状態を変更

2. **宣言的 UI における役割分担**:
   - Presentational: `UI = f(state)` — 状態を受け取り、表示を返す
   - ロジック層: `state = f(input)` — 入力（ユーザー操作、API レスポンス等）を受け取り、状態を更新
   - Container: 両者を繋ぐ

3. **インフラからの分離**:
   - Page からインフラの知識（Cookie、localStorage 等）が注入される
   - ロジック層自体はインフラに直接依存しない

**判定フロー**:
```
ロジックを特定
  ↓
Container に直接書かれているか？
  ↓ Yes → ❌ NG: ロジック層に抽出する
  ↓ No（ロジック層に存在）
  ↓
ロジック層がインフラに直接依存しているか？
  ↓ Yes → ❌ NG: Page で抽象化してロジック層に注入
  ↓ No → ✅ OK
```

**判定**: ロジックは Container ではなくロジック層に実装（`state = f(input)` の実現）

### ステップ4: 責務の分解（バックエンドモジュール・または UI で責務過多の場合）

**判断基準**:
1コンポーネント（モジュール）1責務。「このコンポーネントは何をするか」を1文で説明できない場合は分割。

**チェック項目**:

1. **単一責任の確認**:
   - 変更理由が複数ある場合は分割する
   - 例（バックエンド）: ユーザー認証 + データ永続化 → 認証層と永続化層に分割
   - 例（UI）: データ取得 + 表示 → ロジック層 + Presentational に分割（すでにステップ1-3で対応）

2. **分解の軸（バックエンド）**:
   - ビジネスロジック vs インフラストラクチャ
   - 入力バリデーション vs データ変換 vs 永続化
   - 共通処理 vs ドメイン固有処理

**判定フロー**:
```
コンポーネント（モジュール）の責務を確認
  ↓
1文で説明できるか？
  ↓ No → 複数の責務が混在 → 分割
  ↓ Yes
  ↓
変更理由は1つか？
  ↓ No → 異なる変更理由 → 分割
  ↓ Yes → OK
```

**判定**: 複数の責務を持つ場合は分割

### ステップ5: 共通化の判断（過度な抽象化を避ける・中優先度）

**判断基準**:
同じパターンが3回出たら抽象化する（Rule of Three）。1-2回では具象のまま。

**チェック項目**:

1. **Rule of Three**:
   - 1回: 具象のまま
   - 2回: 様子見（まだ抽象化しない）
   - 3回: 共通化・抽象化を検討

2. **過度な抽象化の回避**:
   - 1回しか使わないのに汎用コンポーネントを作らない
   - フロントエンドには高機能な DI コンテナがないため、過度な抽象化は複雑さを増すだけ
   - 「捨てやすさ」を優先する

**判定フロー**:
```
同じパターンが何回出現するか？
  ↓ 1回 → 具象のまま（抽象化しない）
  ↓ 2回 → 様子見（まだ抽象化しない）
  ↓ 3回以上 → 共通化・抽象化を検討
```

**判定**: 3回出るまで抽象化しない（YAGNI 原則）

### ステップ6: 循環依存の回避（全般・中優先度）

**判断基準**:
A → B → A のような循環依存は必ず解消する。

**チェック項目**:

1. **循環依存の検出**:
   - モジュール間・コンポーネント間の依存グラフを確認
   - A が B に依存し、B が A に依存していないか

2. **循環依存の解消方法**:
   - 共通部分を第三のモジュールに抽出
   - イベント駆動で疎結合にする
   - 依存方向を一方向に統一する

**判定フロー**:
```
依存関係を確認
  ↓
循環依存があるか？
  ↓ Yes → 必ず解消（第三のモジュール抽出 or イベント駆動）
  ↓ No → OK
```

**判定**: 循環依存は必ず解消する

## プロセス

### 1. 既存コンポーネント構造の把握

既存システムがある場合、現在の分割・構成パターンを理解する。

**確認すべきこと**:
- ディレクトリ構造とファイル分割の基準
- コンポーネント・モジュールの命名規則
- 共通コンポーネント・ユーティリティの構成
- 依存関係のパターン（DI の方式、import の構造）
- テストの配置パターン

### 2. 責務の分解

要件を責務単位に分解し、各責務をコンポーネントに割り当てる。

**分解の原則**:
- 1 コンポーネント 1 責務（単一責任原則）
- 「このコンポーネントは何をするか」を 1 文で説明できるか
- 変更理由が異なるものは別コンポーネントに分ける
- ただし、責務の分離は粒度が重要。過度に分割すると全体像が掴みにくくなる

**分解の軸**:
- ビジネスロジック vs インフラストラクチャ
- 状態管理 vs 表示（UI の場合）
- 入力バリデーション vs データ変換 vs 永続化
- 共通処理 vs ドメイン固有処理

### 3. インターフェースの設計

各コンポーネントの公開インターフェースを定義する。

**設計原則**:
- インターフェースは「使う側」の視点で設計する（実装の詳細を漏らさない）
- 入力パラメータは必要最小限にする
- 戻り値の型で成功・失敗を明確に表現する（Result 型等）
- 副作用を持つ関数と純粋関数を明確に区別する

**インターフェース分離（ISP）**:
- クライアントが使わないメソッドへの依存を強制しない
- 大きなインターフェースは、用途別に分割する
- ただし、分割しすぎるとインターフェースが乱立する。実際に異なるクライアントが存在する場合に分割する

### 4. 依存関係の設計

コンポーネント間の依存関係を設計する。

**依存方向の原則**:
- 安定した層（ビジネスロジック）に向かって依存する
- 不安定な層（UI、外部サービス）から安定した層を参照する
- 循環依存を作らない
- 依存の注入（DI）により、具象ではなく抽象に依存する

**依存グラフの確認**:
- 依存の方向が一方通行か
- 双方向依存・循環依存が発生していないか
- テスト時にモック・スタブに差し替え可能か

### 5. 状態管理の設計（UI コンポーネントの場合）

UI コンポーネントの状態管理方針を設計する。

**Page / Container / Presentational の役割分担**:

- **Page**: ルーティングとインフラの境界。パスパラメータ・クエリパラメータのパース、Cookie・localStorage 等のインフラアクセスを抽象化（関数化）し、Container に引き渡す
- **Container**: ロジックと描画の結合。ロジック層を呼び出し、結果を Presentational に渡す。ロジックや API 呼び出し自体はロジック層に委譲する。return で返すのは原則として単一の Presentational コンポーネント
- **Presentational**: Props を受け取り表示するだけ（純粋関数的）。State を保有しない。Context も原則使わないが、Compound パターン（`Accordion.Header` 等）で親コンポーネントが定義した Context を子が参照するケースは許容する。この場合、コンポーネント名から依存関係が自明であること（例: `Accordion.*` は `Accordion` の子であると名前でわかる）と、ドキュメントコメントに依存関係を明記することを条件とする。ユーザー操作はコールバック Props で親へ通知する
- **ロジック層**: ロジックの整理と再利用。Page からインフラの知識が注入されるため、ロジック層自体はインフラに直接依存しない。フレームワークごとに実現手段が異なる（React: Hooks、SolidJS: Signal/createResource、Vue: Composables 等）

**状態の配置**:
- 状態はそれを使うコンポーネントにできるだけ近く配置する
- 複数コンポーネントで共有する状態のみ、上位に持ち上げるかグローバルストアに置く
- サーバー状態（API レスポンス）とクライアント状態（UI の状態）を区別する

### 6. 拡張ポイントの設計

将来の変更・拡張に備えたポイントを最小限設計する。

- 開放閉鎖原則（OCP）: 拡張に対して開かれ、修正に対して閉じた設計
- ただし、現時点で拡張の見通しがないポイントに抽象化を入れない
- 「3回同じパターンが出たら抽象化する」（Rule of Three）

拡張ポイントの判断でトレードオフが発生した場合は ADR に記録する。

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- コンポーネント分割の粒度判断（統合 vs 分割）
- 状態管理方式の選択（ローカル状態 vs グローバルストア）
- DI コンテナの導入・不導入の判断
- 共通コンポーネントの抽出基準の決定
- デザインパターンの適用判断

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **God Component**: 1 つのコンポーネントに複数の責務が集中する
- **Prop Drilling**: 深いコンポーネントツリーを通じてデータを受け渡す
- **Premature Abstraction**: 1 回しか使わないのに汎用コンポーネントを作る
- **Tight Coupling**: コンポーネント間が具象クラスで直接結合している
- **Feature Envy**: あるコンポーネントが別のコンポーネントのデータに過度にアクセスする
- **Circular Dependency**: A → B → C → A のような循環依存
