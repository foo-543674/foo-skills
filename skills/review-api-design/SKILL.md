# API設計レビュー

コードレビュー時にAPI設計の観点で指摘すべきポイントを定義する。REST に限らず、GraphQL や RPC スタイルにも適用できる共通原則を含む。

このスキルには筆者の設計思想が含まれる。絶対的なルールではなく、プロジェクトの文脈（API の公開範囲、クライアントの種類、チームの規模）に応じた判断が必要。

## レビュー観点

### 1. エンドポイント命名と URL 設計の一貫性

- リソース名が複数形で統一されているか（`/users`, `/orders`）
- ネストの深さが適切か（3階層以上は要検討）
- URL にアクション動詞を含んでいないか（`/users/create` ではなく `POST /users`）
- URL の命名規則がプロジェクト全体で統一されているか。URI の大文字小文字の扱いはブラウザや環境によって異なるため、kebab-case または snake_case を推奨し、camelCase は避ける
- ただし RPC スタイルの API（gRPC, tRPC）ではアクション名がエンドポイント名になるのが自然。REST の規則を機械的に当てはめない

### 2. リクエスト・レスポンス構造

- レスポンスの構造がエンドポイント間で一貫しているか（envelope pattern `{ data, meta, error }` 等）
- 不要なフィールドを返していないか（over-fetching）
- ネストが深すぎないか（クライアント側の扱いやすさ）
- 日時フォーマット、null の扱い、空配列 vs null がプロジェクトで統一されているか
- リクエストボディのバリデーションルールが明確か
- ただし BFF（Backend for Frontend）では特定の画面に最適化したレスポンスが正しい設計になる場合がある

### 3. ページネーション・フィルタリング・ソート

- リスト系エンドポイントにページネーションがあるか（無制限にレコードを返さない）
- ページネーション方式（offset vs cursor）が要件に合っているか
- フィルタ・ソートパラメータの設計が一貫しているか
- ただし cursor ベースが常に優れるわけではない。管理画面の「○ページ目に飛ぶ」要件では offset が適切

### 4. 冪等性と安全性

- GET が副作用を持っていないか
- PUT / DELETE が冪等か（同じリクエストを複数回送っても結果が同じ）
- POST で冪等性が必要な操作に Idempotency Key を設計しているか（決済等）
- ただし全ての POST に冪等性キーを求めるのは過剰。二重実行で実害がある操作に絞る

### 5. バージョニング戦略

- API のバージョニング方針が決まっているか（URL path / header / query）
- 破壊的変更の定義が明確か（フィールド追加は非破壊、フィールド削除・型変更は破壊）
- 旧バージョンの廃止計画があるか
- バージョニングを採用していない場合、後方互換性が保たれているか。破壊的変更が入る場合はバージョニングの導入を検討する
- ただし内部 API（BFF 等）でクライアントとサーバーが同時デプロイされる場合、バージョニングは不要。また、ビジネス上の移行期間が終了した等の文脈があれば、後方互換を破る変更も許容される

## よくあるアンチパターン

- **Chatty API**: 1画面の表示に大量のエンドポイントを叩く必要がある（BFF や集約エンドポイントを検討）
- **RPC in REST Clothing**: REST を名乗りつつ `POST /api/getUsers` のようなエンドポイント
- **Inconsistent Error Shape**: エンドポイントごとにエラーレスポンスの構造が異なる
- **Leaking Internal Model**: 内部の DB 構造がそのままレスポンスに露出している

## 指摘の出し方

- 一貫性の問題は既存エンドポイントとの比較で示す（「他のエンドポイントでは○○だが、ここでは△△になっている」）
- 具体的な改善案を示す（URL 構造の修正例、レスポンス構造の修正例）
- なぜその設計を推奨するかの理由（クライアントの使いやすさ、キャッシュ可能性、後方互換性等）を必ず添える
- API スタイル（REST / RPC / GraphQL）の選択自体は本スキルの範囲外。選択されたスタイルの中での一貫性と品質を評価する
