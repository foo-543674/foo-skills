# リファクタリング

既存コードの外部的な振る舞いを変えずに、内部構造を改善する。

## Why: なぜリファクタリングが重要か

リファクタリングの品質と実践は、コードベースの長期的な保守性・変更容易性・開発速度を決定する。適切なリファクタリングなしでは、技術的負債が蓄積し、開発速度が指数関数的に低下する。

**根本的な理由**:
1. **技術的負債の返済**: コードは時間とともに複雑化・劣化する。機能追加のたびに「とりあえず動くコード」が蓄積し、可読性・保守性が低下する。リファクタリングは技術的負債を返済し、コードの健全性を回復する唯一の手段。放置すれば、変更コストが指数関数的に増大し、最終的に変更不可能になる
2. **変更容易性の確保**: リファクタリングにより、変更の影響範囲が局所化される。適切に分割・抽象化されたコードは、変更が 1 箇所で完結する。リファクタリングなしでは、変更が広範囲に波及し、修正漏れ・副作用が頻発する。変更容易性は開発速度の根本的な決定要因
3. **新機能追加の加速**: リファクタリングは、既存コードを新機能追加に適した構造に改善する。適切な構造のコードベースでは、新機能が既存の部品を組み合わせて実現できる。リファクタリングなしでは、新機能追加のたびに車輪の再発明・コードの重複が発生し、開発速度が低下する
4. **理解コストの削減**: リファクタリングにより、コードの意図が明確になる。適切に命名され、適切な粒度で分割されたコードは、読むだけで理解できる。リファクタリングなしでは、コードを理解するために実装の詳細を追う必要があり、認知負荷が高く、バグの温床となる
5. **テスタビリティの改善**: リファクタリングにより、テストしやすい構造になる。依存関係が整理され、単一責任に分割されたコードは、個別にテストできる。リファクタリングなしでは、密結合・God オブジェクトが生まれ、テストに大量のセットアップが必要となり、テストコストが膨大になる

**リファクタリングの目的**:
- 技術的負債を返済し、コードの健全性を維持する
- 変更容易性を確保し、影響範囲を局所化する
- 新機能追加を加速し、再利用性を高める
- 理解コストを削減し、認知負荷を下げる
- テスタビリティを改善し、品質を担保する

## 基本原則

- リファクタリングと機能追加を同時にしない
- 十分なテストがない場合は、まずテストを追加してからリファクタリングする
- 小さなステップで進め、各ステップ後にテストが通ることを確認する
- リファクタリングの目的（何を改善するか）を事前に明確にする

## プロセス

### 1. リファクタリング対象の分析

対象コードの現状を理解し、何を改善するかを明確にする。

**現状の確認**:
- 対象コードの責務・役割
- 依存関係（このコードが依存するもの、このコードに依存するもの）
- テストの有無と品質
- 変更履歴（なぜ現在の形になっているか）

**改善目標の設定**:
- review スキル群の観点から、具体的な問題点を特定する
- 改善後にどうなるべきかを明確にする
- 改善の優先順位をつける（すべてを一度に改善しない）

### 2. 影響範囲の分析

変更が影響する範囲を特定する。

**分析手順**:
1. 対象コードを参照しているコード（呼び出し元）を列挙する
2. 対象コードが参照しているコード（呼び出し先）を確認する
3. 公開インターフェース（関数シグネチャ、型定義）を変更する必要があるか判断する
4. 公開インターフェースの変更がある場合、影響を受けるすべてのコードを把握する

**リスク評価**:
- 影響範囲が広い場合、段階的なリファクタリング計画を立てる
- 公開インターフェースの変更は影響が大きい。内部実装の変更に留められないか検討する

### 3. テストの確保

リファクタリング前にテストが十分であることを確認する。

**テストが不十分な場合**:
1. まず現在の振る舞いを保護するテスト（characterization test）を書く
2. 現在のコードの出力をそのままアサーションにする（正しさではなく「現在の振る舞い」を記録）
3. テストが十分になってからリファクタリングを開始する

**テストが十分な基準**:
- リファクタリング対象の主要なパスがテストでカバーされている
- エッジケース・異常系のテストがある
- テストを実行して全パスすることを確認済み

### 4. 段階的なリファクタリング

小さなステップで進める。

**基本的なリファクタリングパターン**:

| パターン | 場面 | 手順 |
|---------|------|------|
| Extract Function | 長い関数、重複コード | 1. 抽出する範囲を特定 2. 新関数を作成 3. 元の場所から呼び出し 4. テスト実行 |
| Rename | 意図が伝わらない名前 | 1. 新しい名前を決定 2. すべての参照箇所を変更 3. テスト実行 |
| Move | 責務が間違った場所にある | 1. 移動先を決定 2. 移動先に複製 3. 元の場所から委譲 4. テスト実行 5. 元の場所を削除 |
| Replace Conditional with Polymorphism | 複雑な条件分岐 | 1. インターフェース定義 2. 各分岐を実装クラスに 3. 条件分岐をポリモーフィズムに置換 4. テスト実行 |
| Introduce Parameter Object | パラメータが多い関数 | 1. 関連パラメータをグループ化 2. オブジェクトを定義 3. 関数シグネチャを変更 4. テスト実行 |

**各ステップのルール**:
- 1 ステップで 1 つのリファクタリングパターンのみ適用する
- ステップ後に全テストを実行する
- テストが失敗したら、そのステップを取り消して原因を調査する
- 各ステップをコミットする（ロールバック可能にする）

### 5. 最終確認

リファクタリング完了後の確認。

- 全テストが通るか
- 改善目標を達成しているか
- 新たな問題（可読性の低下、パフォーマンスの劣化等）を生んでいないか
- 不要なコード・ファイルが残っていないか
- コミット履歴は適切か（リファクタリングの意図が伝わるか）

## リファクタリングの判断基準

### やるべきとき
- 機能追加・修正の前に、対象コードの理解・変更を困難にしている問題がある
- review スキル群で繰り返し同じ指摘が出る箇所
- テストが書きにくい構造になっている

### やるべきでないとき
- 動いているコードを「きれいにしたい」だけの動機
- リファクタリングの影響範囲が把握できない（まずテストを書く）
- リリース直前
- 変更予定のないコード

## リファクタリング完了チェックリスト

リファクタリング完了時に以下を確認する。

### 安全性の確認
- [ ] リファクタリング前にテストが十分にある
- [ ] 全テストが通ることを確認した
- [ ] 外部的な振る舞いが変わっていない
- [ ] リファクタリングと機能追加を同時にしていない

### 改善目標の達成確認
- [ ] 事前に設定した改善目標を達成している
- [ ] review スキル群で繰り返し指摘されていた問題を解消している
- [ ] 新たな問題（可読性の低下、パフォーマンス劣化等）を生んでいない

### コード品質の確認
- [ ] review-readabilityの観点: 関数の長さ、ネストの深さ、早期リターン
- [ ] review-solidの観点: 単一責任、開放閉鎖、依存性逆転
- [ ] review-functionalの観点: 純粋関数の優先、副作用の分離
- [ ] review-namingの観点: 意図が伝わる名前の改善

### 段階的リファクタリングの確認
- [ ] 小さなステップで進めた（1ステップ1パターン）
- [ ] 各ステップ後に全テストを実行した
- [ ] 各ステップをコミットした（ロールバック可能）
- [ ] テストが失敗した場合、そのステップを取り消して原因を調査した

### 影響範囲の確認
- [ ] 影響範囲を事前に分析した
- [ ] 公開インターフェースの変更がある場合、影響を受けるすべてのコードを修正した
- [ ] 不要なコード・ファイルが残っていない

### コミット履歴の確認
- [ ] コミット履歴がリファクタリングの意図を伝えている
- [ ] commit-messageで改善内容が説明されている

## よくあるアンチパターン

- **Big Bang Refactor**: 大量のコードを一度にリファクタリングし、問題の切り分けが困難になる
- **Refactor Without Tests**: テストなしでリファクタリングし、デグレードに気づかない
- **Refactor and Add Feature**: リファクタリングと機能追加を同じコミットで行う
- **Premature Refactor**: 問題が発生していないコードを予防的にリファクタリングする
- **Refactor Loop**: リファクタリングが完了せず、永遠にリファクタリングし続ける
