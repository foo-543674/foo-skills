# Issue 本文テンプレート

`skill-feedback` が発行する Issue の標準フォーマット。1 指摘 = 1 Issue。

## タイトル

```
[<対象原料カテゴリ>] <指摘の要約（1 行）>
```

- `<対象原料カテゴリ>`: 例 `philosophy/core-principles`, `perspectives/testing`, `bootstrap/template`。不明な場合は `[unknown]`
- `<指摘の要約>`: 生成物のどこにどういうズレがあったかを端的に。1 行に収める
- タイトルに絵文字は入れない
- 英語でも日本語でも良いが、プロジェクトの既存 Issue に合わせる（迷ったら日本語）

### 良い例

```
[perspectives/testing] 生成された CLAUDE.md にプロパティベーステストの判断基準が欠けている
[philosophy/core-principles] 生成物が「語彙定義義務」を具体例なしで記述してしまう
[bootstrap/template] code-reviewer エージェントにセキュリティ観点が含まれない
```

### 悪い例

```
フィードバック                          ← 何に対する何の指摘か不明
[testing] 改善して                      ← 具体性が欠如
[生成物] もっと良くしてほしい            ← 対象・内容ともに曖昧
```

## ラベル

必須：

- `feedback`: このスキル由来の Issue

確信度（いずれか 1 つ）：

- `confirmed`: 原料の改善で明確に再発防止できる
- `candidate`: グレーゾーン。却下される可能性もあるが記録として残す

対象（判明時のみ）：

- `source:<name>`: 改善すべき原料（例 `source:perspectives/testing`）。推定の場合も付ける（本文に「推定」と明記）

## 本文

```markdown
## 対象

- **改善すべき原料**: `<原料パス>`（推定の場合は「推定」と明記）
- **影響を受けた生成物**: <CLAUDE.md / エージェント定義 / アーキテクチャテスト 等>
- **場面**: <作業のどの段階で起きたか。設計中 / 実装中 / レビュー中 など>

## 生成物のズレ

<生成された環境がどう振る舞ったか、foo-543674 が何を期待していたか、そのギャップは何か。
事実ベースで 3-5 行。感情や評価は入れない。>

## 対話の抜粋

<該当する AI の出力 → foo-543674 の指摘、の順で引用。
引用は要点のみ。長文は省略可（`...` で省略箇所を明示）。>

> **AI**:
> <AI の出力の該当部分>

> **foo-543674**:
> <指摘の該当部分>

## 想定される原料改善

<どの原料ファイルのどの部分をどう変えれば次回の生成物から防げるかの仮説。
断定せず「〜すれば防げそう」「〜の追加で補える見込み」のように書く。
仮説が複数あれば箇条書きで併記する。>

## 類似 Issue

<既存 Issue 検索で見つかった類似ケースへの参照。なければ「なし」>

- #N: <タイトル>（採用 / 却下 / オープン）

## 判定

- **確信度**: confirmed | candidate
- **判定理由**: <なぜその分類にしたか。1-2 行>
```

## テンプレート使用時の注意

- **対話の抜粋は最小限**: 全文貼り付けは禁止。Issue を読む人（foo-543674 本人・未来の AI）が状況を把握できる最小限の引用に絞る
- **想定される原料改善は「仮説」として書く**: 断定しない。修正 PR を書く段階で改めて判断される前提
- **類似 Issue のリンクは採用/却下の結果も添える**: 「過去どう判断されたか」が次の判断材料になるため、単なる参照ではなく判定結果も一緒に記載する
- **判定理由は短く**: 長文での正当化は不要。「テンプレートに観点が欠落しているため confirmed」程度で十分
- **絵文字は使わない**: タイトル・本文とも
