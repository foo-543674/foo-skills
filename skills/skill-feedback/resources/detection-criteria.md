# 指摘検知の判定規則

会話履歴から「指摘」を抽出する際の採用/除外ルールと、分類（confirmed / candidate）の判定軸。

## 「指摘」の定義

**指摘 = foo-skills の原料（bootstrap 原料 = philosophy / perspectives / bootstrap テンプレート、および各スキル定義）を改善すれば、次回から防げるようになるもの**

この定義は Why に直結している。foo-skills は **ジェネレータ + 内蔵スキル群** であり、bootstrap がプロジェクトに AI 自走環境を生成する一方で、consult 等の plugin 内蔵スキルも foo-543674 の作業環境にもたらされる振る舞いを構成する。プラグインを通じてもたらされた振る舞いに感じた「ズレ」は、いずれかの原料がまだ十分でないことの兆候として扱う。bootstrap 生成物への指摘も plugin 内蔵スキルへの指摘も同等に対象。**定義から外れるもの（ユーザー起因・正当振る舞い・モデル能力の限界）は Issue 化しない**。混ぜるとトリアージ負荷が跳ね上がり、ループが回らなくなる。

## 採用/除外の判定フロー

各候補発話に対して以下を順に確認する。

```
1. この失敗の原因が foo-543674 側にあると、foo-543674 自身が明示的に認めているか？
   （「省いてた」「スマホ入力めんどくさくて」など本人の発言がある場合のみ Yes）
   └─ Yes → 除外
   └─ No／不明 → 2 へ（推測でユーザー起因と判定して除外してはならない）

2. AI の振る舞いは、生成された環境の設計として正当なプロセスの一部だったか？
   （生成された CLAUDE.md の指示通りの動作、エージェントの定義通りの振る舞い等）
   └─ Yes → 除外
   └─ No  → 3 へ

3. その失敗は foo-skills の原料の改善で再発防止できそうか？
   （philosophy の価値観追加、perspectives の観点強化、bootstrap テンプレートの記述改善、または各スキル定義の補強など）
   └─ Yes、明確に防げる          → confirmed として採用
   └─ グレー、補える可能性がある  → candidate として採用
   └─ No、モデル能力の限界        → 除外
```

**迷ったら candidate に倒す**。false negative（取りこぼし）の損失のほうが大きい。判断の妥当性は GitHub 側のトリアージで再検討される。

## 除外カテゴリの判定基準

### カテゴリ A: ユーザー起因

**除外の唯一の十分条件**: foo-543674 が自身の入力ミスや省略を**明示的に認めている**（「省いてた」「スマホ入力めんどくさくて」など本人の発言がある）こと。

以下はユーザー起因を疑う参考シグナルだが、これらだけでは除外の根拠にならない。**推測でユーザー起因と判定して除外してはならない**。

- 直前のユーザー発話が曖昧・情報不足で、AI の解釈違いが発生していた
- 必要な前提情報（プロジェクト固有のコンテキスト等）が会話に提供されていなかった

**「foo-543674 が明示的にユーザー起因だと述べた場合」以外は迷わず採用側に倒す**。

### カテゴリ B: 正当な振る舞い

以下のいずれかに該当すれば除外。

- 生成された CLAUDE.md に記述された指示に従った振る舞い
- 生成されたエージェント定義通りの動作
- 生成されたアーキテクチャテストが正しく機能している結果
- bootstrap が意図した通りのコンテキスト生成結果
- plugin 内蔵スキル（consult 等）の SKILL.md / resources の定義通りの動作

判定の手がかり：**対応する原料ファイル（CLAUDE.md、エージェント定義、内蔵スキルの SKILL.md 等）を読み直したとき、「これは原料の定義通りの正しい動作」と言えるか？** 言えるなら除外。

### カテゴリ C: モデル能力の限界

以下のいずれかに該当すれば除外。

- コンテキストをどれだけ積んでも補えない推論能力の限界
- モデル固有の知識カットオフに起因する失敗
- 原料の改善で再発防止できる見込みがない

ただし、一見「推論能力の問題」に見えても、**原料の改善で誘導できる余地があるもの**は candidate として採用する。例: 「philosophy に判断基準を追加していれば気付けた」「perspectives に観点を強化していれば検出できた」など。

## 分類の判定軸（confirmed vs candidate）

### confirmed（確定指摘）

以下のいずれかが明確なら confirmed。

- **生成物の品質不足**: 生成された CLAUDE.md やエージェント定義に、本来あるべき判断基準・観点・制約が欠けている
- **原料の欠落**: philosophy/ や perspectives/ に記述されるべき価値観・品質基準が不足しており、生成物に反映されていない
- **テンプレートの不足**: bootstrap のテンプレートに本来含まれるべきセクション・構造が欠けている

### candidate（候補）

以下のいずれかに該当するなら candidate。

- 原料改善で補えるかどうか、その場で判定しきれない
- 推論能力の問題にも見えるが、原料の工夫で誘導できる可能性がある
- 同種の問題が再発する確信が持てない（単発のブレの可能性）

## 改善すべき原料の特定

指摘を Issue 化する際、どの原料を改善すべきかを特定する。

判定手順：

1. **指摘の性質から原料カテゴリを推定**:
   - 価値観・判断基準の不足 → `philosophy/` のいずれか
   - 品質観点の不足・偏り → `perspectives/` のいずれか
   - 生成物の構造・セクションの不足 → `bootstrap/template`
   - bootstrap のフロー自体の問題 → `bootstrap/skill`
   - plugin 内蔵スキル（consult 等）の振る舞いに対する指摘 → 該当スキルの `SKILL.md` または `resources/` 配下のファイル
2. **カテゴリ内の具体ファイルを推定**:
   - philosophy: core-principles / technology-choices / quality-standards / development-values
   - perspectives: 20 観点のいずれか（architecture, testing, naming 等）
   - skills: 該当スキルの `SKILL.md` または `resources/` 配下のファイル（およびその中の不足セクション。例: 起動条件 / 振る舞い / 判定規則）
3. **`source:` ラベル付与の判断**:
   - 推定したカテゴリに対応する `source:<name>` ラベルが既に整備されていれば付与する
   - 整備されていない場合（例: plugin 内蔵スキル）は **新規ラベルを AI 側で作成せず、`source:` ラベルを省略**。本文「対象」セクションで対象原料パスを明記する
   - どうしても特定できない場合も同様に省略し、本文で「対象原料: 不明（会話履歴からは特定困難）」と明記する

## アンチパターン

検知時に避けるべき判断：

- **ユーザー起因だと勝手に決めつける**: foo-543674 が明示的にそう言わない限り、採用側に倒す
- **正当な振る舞いと指摘を混同する**: 生成物 or 内蔵スキルの定義通りに動いた結果のユーザーリアクションを指摘と誤認する
- **1 指摘を複数 Issue に分解する**: 1 つの失敗パターンは 1 Issue にまとめる（関連する別の失敗は別 Issue）
- **指摘を 1 Issue に統合する**: 異なる失敗パターンを「まとめて Issue」にしない（修正 PR の粒度を壊す）
- **ラベル `source:<name>` の推測を断定調で書く**: 推定なら推定と明記する
- **既存にない `source:<name>` ラベルを新規作成する**: 未整備領域では `source:` を省略し本文に明記する。新規ラベル体系の整備は人間側の意思決定事項
- **「bootstrap 生成物ではないから対象外」と判定する**: plugin 内蔵スキルへの指摘も対象。スコープを生成物に限定しない
