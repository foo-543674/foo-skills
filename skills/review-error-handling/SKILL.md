# エラーハンドリングレビュー

コードレビュー時にエラーハンドリング・異常系設計の観点で指摘すべきポイントを定義する。

## Why: なぜエラーハンドリングが重要か

エラーハンドリングはシステムの信頼性と保守性を決定する重要な設計要素である。

**根本的な理由**:
1. **システムの信頼性**: 適切なエラーハンドリングがないシステムは、予期しないエラーでデータが破損したり、一貫性のない状態に陥る。エラーは必ず発生する前提で設計する必要がある
2. **障害の早期発見と修復**: エラーを握りつぶすと、問題が発生してもそれが検知されず、障害が潜在化・長期化する。適切なエラーハンドリングとロギングは、問題を早期に発見し修復する基盤となる
3. **ユーザー体験**: 不適切なエラーハンドリングは、ユーザーに「何が起きたか」「どうすればよいか」を伝えられず、フラストレーションを生む。エラーメッセージはユーザーとのコミュニケーション手段である
4. **セキュリティ**: 内部実装の詳細がエラーメッセージとして外部に漏れると、攻撃者に攻撃の糸口を与える。エラーの境界での適切な変換はセキュリティの要である
5. **開発者の生産性**: 詳細なエラー情報とコンテキストがないと、バグ修正に膨大な時間がかかる。エラーの可観測性は開発者の生産性に直結する

**エラーハンドリングレビューの目的**:
- システムがエラー時にも安全な状態を保つことを保証する
- エラーを早期発見・早期修正できる仕組みを作る
- ユーザーに有用なフィードバックを提供する
- セキュリティリスクを回避する
- 開発者がデバッグしやすい情報を残す

## 判断プロセス（決定ツリー）

エラーハンドリングのレビュー時は、以下の順序で判断する。優先度の高い問題から確認し、問題があれば指摘する。

### ステップ0: そもそも例外で表現すべきエラーか（最優先）

**判断基準**:
エラーハンドリングの手法（例外 vs Result型）を間違えると、制御フローが複雑化し、型安全性が失われ、予期しないバグを生む。

**根本的な判断軸**:
1. **予測可能性**: このエラーは予測可能な正常系の一部か、予測不能な異常か
2. **制御フロー**: エラーを呼び出し側が明示的に処理すべきか、自動的に伝播すべきか
3. **型システム**: エラーを戻り値の型として表現できるか

**チェック項目**:

#### 1. Result型を使うべきケース（予測可能な失敗）

**特徴**:
- ビジネスルールによる失敗（予測可能）
- 呼び出し側が明示的に処理すべき
- 型システムで表現可能

**具体例**:
```typescript
// ❌ 例外を使っている（間違い）
function validateEmail(email: string): void {
  if (!email.includes("@")) {
    throw new Error("Invalid email");  // バリデーションは予測可能な失敗
  }
}

// ✅ Result型を使う（正しい）
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

function validateEmail(email: string): Result<Email, ValidationError> {
  if (!email.includes("@")) {
    return { ok: false, error: new ValidationError("Email must contain @") };
  }
  return { ok: true, value: new Email(email) };
}
```

**Result型を使うべき典型例**:
- **バリデーション失敗**: 入力値が仕様を満たさない
- **ビジネスルール違反**: 在庫不足、権限不足、状態遷移の不整合
- **パース失敗**: JSONパース、日付パース等（不正な入力）
- **検索結果なし**: データベースに該当レコードがない
- **計算結果の範囲外**: 割り算の0除算、オーバーフロー（予測可能な場合）

**判定**: 上記のケースで例外を使っている場合は**必ず指摘**

#### 2. 例外を使うべきケース（予測不能な異常）

**特徴**:
- システムレベルの障害（予測不能）
- 通常の制御フローでは処理できない
- 型システムで表現する意味がない（どこで発生するか不明）

**具体例**:
```typescript
// ✅ 例外を使う（正しい）
async function fetchUserData(userId: string): Promise<User> {
  try {
    const response = await fetch(`/api/users/${userId}`);
    // ネットワーク障害、タイムアウト等は例外で伝播
  } catch (e) {
    // 予期しない障害は例外のまま上位に伝播
    throw e;
  }
}
```

**例外を使うべき典型例**:
- **インフラ障害**: ネットワーク障害、データベース接続失敗、ディスク満杯
- **プログラミングエラー**: null参照、配列境界外アクセス、型不整合
- **システムリソース不足**: メモリ不足、ファイルハンドル枯渇
- **外部依存の予期しない障害**: 外部APIの500エラー、タイムアウト
- **不変条件の違反**: 「ここには到達しないはず」のケース

**判定**: 上記のケースでResult型を使っている場合、「例外でよいのでは」と提案

#### 3. ハイブリッドケース（状況による）

一部のケースは、文脈によって判断が分かれる：

| ケース | Result型を使う場合 | 例外を使う場合 |
|--------|------------------|---------------|
| ファイル読み込み | ユーザーが指定したファイル（存在しない可能性が高い） | 設定ファイル等、必ず存在すべきファイル |
| API呼び出し | ユーザー入力に基づくAPI呼び出し（失敗が正常系） | システム起動時の必須API呼び出し |
| データベースクエリ | 検索結果0件（正常系） | マスターデータの取得失敗（異常） |

**判定**: 文脈を確認し、「予測可能な正常系の一部」ならResult型を推奨

#### 4. 判定フロー

```
エラーハンドリングを発見
  ↓
このエラーは予測可能か？（ビジネスルールによる失敗か）
  ↓ Yes（バリデーション、権限不足、在庫不足等）
  → 例外を使っているか？
      ↓ Yes → [Important] Result型を使うべき
      ↓ No（Result型使用）→ OK
  ↓ No（予測不能な異常）
  → Result型を使っているか？
      ↓ Yes → [Minor] 例外のほうが適切では？
      ↓ No（例外使用）→ OK
```

**チェックリスト**:
- [ ] バリデーション失敗を例外で表現していないか
- [ ] ビジネスルール違反（在庫不足、権限不足等）を例外で表現していないか
- [ ] 検索結果なしを例外で表現していないか
- [ ] パース失敗を例外で表現していないか
- [ ] 逆に、予測不能なインフラ障害をResult型で表現していないか

**判定**: 例外とResult型の使い分けが不適切な場合は**必ず指摘**

**言語制約の考慮**:
- Go、Rust等、Result型が言語標準のケース → Result型を優先
- Java、C#等、チェック例外のあるケース → 予測可能なエラーはチェック例外も許容
- JavaScript/TypeScript等 → Result型ライブラリの使用を推奨（型安全性向上）

### ステップ1: エラーの握りつぶしがないか（高優先度）

**判断基準**:
エラーを握りつぶすと、障害が潜在化し、データ破損や一貫性違反につながる危険性が最も高い。

**チェック項目**:
1. **空のcatchブロック**: `catch (e) { }` のパターン
   - 判定: **必ず指摘する**（例外: 正当な理由があり、明示的なコメントがある場合）
2. **ログだけして処理続行**: `catch (e) { log(e); }` のパターン
   - 判定基準: このエラーは回復可能か？
     - 回復不能 → **上位に伝播すべき**（必ず指摘）
     - 回復可能 → リカバリ処理が必要（指摘）
     - 意図的に無視 → コメントで理由を明示すべき（指摘）
3. **上位に伝播すべきエラーを止めている**:
   - ビジネスロジックのエラー（バリデーション失敗、権限不足等）を途中でキャッチして処理を続行
   - 判定: **必ず指摘する**

**判定フロー**:
```
catchブロックを発見
  ↓
catchブロックが空 or ログのみ？
  ↓ Yes
  → 回復不能なエラーか？
      ↓ Yes → [Critical] 上位に伝播すべき
      ↓ No
      → 回復可能なエラーか？
          ↓ Yes → [Important] リカバリ処理を実装すべき
          ↓ No → [Important] 無視する理由をコメントで明示すべき
```

### ステップ2: エラーの握りつぶしがないか（高優先度）

**判断基準**:
エラーを握りつぶすと、障害が潜在化し、データ破損や一貫性違反につながる危険性が高い。

**注意**: ステップ0で例外とResult型を使い分けた上で、例外を使う場合のエラー握りつぶしをチェックする。

**チェック項目**:
1. **空のcatchブロック**: `catch (e) { }` のパターン
   - 判定: **必ず指摘する**（例外: 正当な理由があり、明示的なコメントがある場合）
2. **ログだけして処理続行**: `catch (e) { log(e); }` のパターン
   - 判定基準: このエラーは回復可能か？
     - 回復不能 → **上位に伝播すべき**（必ず指摘）
     - 回復可能 → リカバリ処理が必要（指摘）
     - 意図的に無視 → コメントで理由を明示すべき（指摘）
3. **上位に伝播すべきエラーを止めている**:
   - システム障害のエラーを途中でキャッチして処理を続行
   - 判定: **必ず指摘する**

**判定フロー**:
```
catchブロックを発見
  ↓
catchブロックが空 or ログのみ？
  ↓ Yes
  → 回復不能なエラーか？
      ↓ Yes → [Critical] 上位に伝播すべき
      ↓ No
      → 回復可能なエラーか？
          ↓ Yes → [Important] リカバリ処理を実装すべき
          ↓ No → [Important] 無視する理由をコメントで明示すべき
```

### ステップ3: エラーの粒度が適切か（高優先度）

**判断基準**:
広すぎるtry-catchは、異なる性質のエラーを同じ方法で扱ってしまい、誤ったハンドリングにつながる。

**チェック項目**:
1. **try-catchの範囲が広すぎないか**
   - 100行以上のtryブロック → **指摘を検討**
   - 異なる性質の処理を1つのtryで囲んでいる → **必ず指摘**
   - 判断基準: tryブロック内の各処理が同じエラーハンドリング戦略で扱えるか
2. **汎用的な例外型でキャッチしていないか**
   - `catch (Exception e)` や `catch (Error e)` のパターン
   - 判定基準:
     - 複数の具体的な例外型を個別にキャッチできるか → **指摘する**
     - トップレベルのエラーハンドラーで最終的にキャッチするケース → 許容
3. **エラーの種類に応じた処理の分岐があるか**
   - catchブロック内で例外の型や内容で分岐している → **リファクタリング推奨**（個別catch推奨）

**判定フロー**:
```
try-catchを発見
  ↓
tryブロックの範囲をチェック
  ↓
異なる性質の処理が含まれる？
  ↓ Yes → [Important] tryブロックを分割すべき
  ↓ No
  ↓
汎用例外型でキャッチ？
  ↓ Yes
  → トップレベルハンドラーか？
      ↓ Yes → 許容
      ↓ No → [Important] 具体的な例外型でキャッチすべき
```

### ステップ4: エラー情報が失われていないか（高優先度）

**判断基準**:
エラー情報が失われると、デバッグが困難になり、障害対応の時間が増大する。

**チェック項目**:
1. **catch-rethrowで原因がチェーンされているか**
   ```typescript
   // ❌ 原因が失われる
   catch (e) { throw new Error("Failed"); }

   // ✅ 原因がチェーンされる
   catch (e) { throw new Error("Failed", { cause: e }); }
   ```
   - 判定: catchして再throwする場合は**必ず指摘**
2. **スタックトレースが保持されているか**
   - エラーオブジェクトを文字列化してログに出している → **指摘**（スタックトレース消失）
   - エラーオブジェクトをそのままロギングライブラリに渡しているか確認
3. **エラーのコンテキストが記録されているか**
   - 何を処理中だったか
   - どの入力値でエラーが発生したか
   - システムのどの状態でエラーが発生したか
   - これらが不足している場合 → **指摘**

**判定**: エラー情報の欠損は**必ず指摘する**

### ステップ5: 境界でのバリデーションが適切か（高優先度）

**判断基準**:
システム境界での入力検証が不十分だとセキュリティリスクとなり、過剰だとパフォーマンスと可読性が低下する。

**チェック項目**:
1. **境界での入力検証があるか**
   - システム境界の定義:
     - ユーザー入力（HTTPリクエスト、CLIコマンド引数、ファイルアップロード）
     - 外部APIレスポンス
     - ファイル読み込み、DB読み込み（信頼できないデータソース）
   - これらの境界で検証がない → **必ず指摘**
2. **内部処理での過剰な防御がないか**
   - バリデーション済みのデータを再度検証している
   - 型システムで保証されている条件を再チェックしている
   - これらは **リファクタリング推奨**
3. **バリデーションエラーメッセージの品質**
   - ユーザーに何が問題か伝わるか
   - どう修正すればよいか示しているか
   - 不十分な場合 → **指摘**

**バリデーションの判断基準**:
```
データの入口を特定
  ↓
システム境界からのデータか？
  ↓ Yes
  → バリデーションがあるか？
      ↓ No → [Critical] バリデーションを追加すべき
      ↓ Yes
      → エラーメッセージは有用か？
          ↓ No → [Important] メッセージを改善すべき
  ↓ No（内部データ）
  → 過剰な防御をしていないか？
      ↓ Yes → [Minor] 不要なチェックを削除すべき
```

### ステップ6: リカバリ戦略が適切か（中優先度）

**判断基準**:
不適切なリカバリ戦略は、障害を悪化させたり、データ整合性を破壊する。

**チェック項目**:
1. **リトライの適切性**
   - リトライしている処理は冪等か → 非冪等な処理をリトライ → **必ず指摘**
   - リトライ戦略が実装されているか（指数バックオフ、上限回数）
     - 無制限リトライ → **指摘**
     - 固定間隔リトライ（サービス過負荷のリスク）→ **指摘**
2. **フォールバック値の妥当性**
   - デフォルト値を返している → その値でビジネスロジックが誤動作しないか確認
   - 不正な状態を無言で作らないか → 疑わしい場合は**指摘**
3. **部分失敗の扱い**
   - バッチ処理で一部が失敗した場合の挙動が定義されているか
   - 全体ロールバック？部分コミット？再実行時の冪等性は？
   - 明確でない場合 → **指摘**

**リトライ判定フロー**:
```
リトライ処理を発見
  ↓
処理は冪等か？
  ↓ No → [Critical] 非冪等処理のリトライは禁止
  ↓ Yes
  ↓
リトライ戦略は適切か？
  - 上限回数あり？
  - 指数バックオフ？
  ↓ No → [Important] リトライ戦略を改善すべき
```

### ステップ7: リソースが確実に解放されるか（中優先度）

**判断基準**:
リソースリークは、時間とともにシステムを不安定にし、最終的にサービス停止につながる。

**チェック項目**:
1. **言語機構の活用**
   - try-with-resources (Java)、using (C#)、defer (Go)、contextmanager (Python)
   - 手動でfinally書いている → **言語機構の使用を推奨**
2. **finallyブロックの安全性**
   - finally内で例外が発生する可能性はないか
   - finally内の処理が失敗すると元の例外が失われる → **指摘**
3. **確実な解放**
   - エラー発生時にファイルハンドル、DB接続、ロック、HTTP接続が解放されるか
   - 確認できない場合 → **指摘**

**判定**: リソースリークの可能性がある場合は**必ず指摘**

### ステップ8: エラーの可観測性が確保されているか（低優先度）

**判断基準**:
エラーログは障害対応の生命線。不十分なログは障害対応時間を増大させる。

**チェック項目**:
1. **ログのコンテキスト**
   - 以下が含まれているか:
     - エラーが発生した処理の種類
     - 入力データ（機密情報を除く）
     - エラー発生時のシステム状態
     - ユーザーID、リクエストID、トレースID等の追跡情報
   - 不足している場合 → **指摘**
2. **ログレベル**
   - error: 回復不能、人間の介入が必要
   - warn: 回復可能だが注目すべき
   - info: 通常の動作
   - ログレベルが不適切な場合 → **指摘**
3. **構造化ログ**
   - ログが構造化されているか（JSON等）
   - アラート・監視システムで処理可能か
   - フリーテキストのみ → **構造化を推奨**

**判定**: 障害対応に必須の情報が欠けている場合は**指摘**

## レビュー観点（詳細チェックリスト）

上記の判断プロセスを実行するための具体的なチェックリスト。

### 0. 例外 vs Result型の使い分け

- [ ] バリデーション失敗を例外で表現していないか（Result型を使うべき）
- [ ] ビジネスルール違反（在庫不足、権限不足等）を例外で表現していないか
- [ ] 検索結果なし（0件）を例外で表現していないか
- [ ] パース失敗（JSON、日付等）を例外で表現していないか
- [ ] 予測可能な計算エラー（0除算等）を例外で表現していないか
- [ ] 逆に、予測不能なインフラ障害（ネットワーク、DB接続失敗等）をResult型で表現していないか
- [ ] 言語の特性（Go、Rust、TypeScript等）に応じた適切な手法を使っているか

### 2. エラーの握りつぶし

- [ ] 空のcatchブロックがないか
- [ ] catchしてログだけ出して処理続行していないか（回復不能なエラーの場合）
- [ ] エラーを無視する正当な理由がコメントで明示されているか
- [ ] システム障害のエラーを途中で止めていないか

### 3. エラーの粒度

- [ ] tryブロックの範囲が広すぎないか（100行以上、異なる性質の処理を含む）
- [ ] 汎用的な例外型（Exception, Error）でキャッチしていないか
- [ ] catchブロック内でエラーの型や内容による分岐がないか（個別catchを推奨）

### 4. エラーの伝播

- [ ] catch-rethrowで原因（cause）をチェーンしているか
- [ ] スタックトレースが保持されているか
- [ ] エラーのコンテキスト（何を、どの入力で、どの状態で）が記録されているか
- [ ] API境界でのエラー変換が適切か（内部詳細の漏洩なし）

### 5. 境界でのバリデーション

- [ ] システム境界（ユーザー入力、外部API、ファイル）で入力を検証しているか
- [ ] バリデーション済みのデータを内部で再検証していないか（過剰防御）
- [ ] バリデーションエラーメッセージがユーザーに有用か

### 6. リカバリ戦略

- [ ] リトライ処理は冪等な操作のみに適用されているか
- [ ] リトライ戦略（指数バックオフ、上限回数）が実装されているか
- [ ] フォールバック値・デフォルト値が妥当か（不正な状態を作らないか）
- [ ] 部分失敗（バッチ処理等）への対応が定義されているか

### 7. リソースの解放

- [ ] エラー発生時にリソース（ファイル、DB接続、ロック）が確実に解放されるか
- [ ] 言語機構（try-with-resources等）を活用しているか
- [ ] finallyブロック内で新たな例外が発生する可能性はないか

### 8. エラーの可観測性

- [ ] エラーログに十分なコンテキスト（処理種類、入力、状態、追跡情報）が含まれているか
- [ ] ログレベル（error/warn/info）が適切か
- [ ] ログが構造化されているか（監視・アラートで処理可能か）

  // アナリティクスの失敗はユーザー体験に影響しないため無視

## 指摘の出し方

### 指摘の構造

```
【問題点】この処理は○○のリスクがあります
【理由】△△だからです
【影響】この問題により××の障害が発生する可能性があります
【提案】□□に変更すると安全性が向上します
```

### 指摘の例

#### エラーの握りつぶしの指摘

```
【問題点】このcatchブロックはエラーをログ出力するだけで処理を続行しています
【理由】データベース接続エラーは回復不能なエラーです
【影響】データが不完全な状態でコミットされ、データ整合性が破壊される可能性があります
【提案】エラーを上位に伝播させて、トランザクション全体をロールバックすべきです
```

#### エラーの粒度の指摘

```
【問題点】このtryブロックは200行にわたり、3つの異なるAPI呼び出しを含んでいます
【理由】各API呼び出しは異なるリカバリ戦略が必要です（API1はリトライ可、API2は即失敗すべき）
【影響】すべてのエラーが同じ方法で扱われ、不適切なリカバリが実行される可能性があります
【提案】各API呼び出しを個別のtryブロックで囲い、適切なエラーハンドリングを実装しませんか
```

#### 境界でのバリデーションの指摘

```
【問題点】外部APIからのレスポンスをバリデーションなしで使用しています
【理由】外部APIは信頼できないデータソースです（仕様変更、障害時の不正レスポンス等）
【影響】予期しないデータ構造により、実行時エラーやデータ破損が発生する可能性があります
【提案】API境界でレスポンスのバリデーションを追加し、不正な場合は明示的にエラーにすべきです
```

#### リトライ戦略の指摘

```
【問題点】注文作成処理（非冪等）をリトライしています
【理由】この処理は冪等ではないため、リトライにより重複注文が作成される可能性があります
【影響】ユーザーが意図しない重複注文により、過剰請求や在庫問題が発生します
【提案】以下のいずれかの対策を推奨します
  1. 注文IDをクライアント側で生成し、サーバー側で重複チェック（冪等化）
  2. リトライを削除し、エラー時はクライアントに通知してユーザー判断に委ねる
```

### 指摘の優先度ラベル

- **[Critical]**: エラーの握りつぶし、境界バリデーションなし、データ破損リスク（必ず修正すべき）
- **[Important]**: エラー情報の欠損、不適切なリトライ、リソースリーク（修正を強く推奨）
- **[Minor]**: ログの改善、過剰防御の削除、リファクタリング（修正を推奨）
- **[Nit]**: ログレベルの微調整等（修正は任意）

## テストケース

レビュー判断の精度を検証するためのテストケース。

### Case 0: 例外 vs Result型の使い分け（Important指摘）

**コード**:
```typescript
// バリデーション失敗を例外で表現
function createUser(email: string, password: string): User {
  if (!email.includes("@")) {
    throw new Error("Invalid email");  // バリデーションは予測可能な失敗
  }
  if (password.length < 8) {
    throw new Error("Password too short");
  }
  return new User(email, password);
}
```

**期待される指摘**:
- [Important] バリデーション失敗を例外で表現している
- バリデーションは予測可能な正常系の一部であり、Result型を使うべき
- 提案: 以下のようにResult型を使用
  ```typescript
  type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

  function createUser(email: string, password: string): Result<User, ValidationError> {
    if (!email.includes("@")) {
      return { ok: false, error: new ValidationError("Invalid email") };
    }
    if (password.length < 8) {
      return { ok: false, error: new ValidationError("Password too short") };
    }
    return { ok: true, value: new User(email, password) };
  }
  ```

### Case 1: エラーの握りつぶし（Critical指摘）

**コード**:
```typescript
async function processOrder(order: Order) {
  try {
    await db.beginTransaction();
    await db.createOrder(order);
    await paymentService.charge(order.amount);
    await db.commit();
  } catch (e) {
    console.error("Error processing order", e);
    // 処理続行
  }
  return { success: true };
}
```

**期待される指摘**:
- [Critical] エラーを握りつぶして処理を続行している
- データベーストランザクションが中途半端な状態でコミットされる
- エラーを上位に伝播し、トランザクションをロールバックすべき

### Case 2: エラーの粒度（Important指摘）

**コード**:
```typescript
async function processUserRegistration(data: RegistrationData) {
  try {
    // 100行: バリデーション、DB保存、メール送信、決済処理
    await validateUser(data);
    await saveToDatabase(data);
    await sendWelcomeEmail(data.email);
    await chargeInitialFee(data.paymentInfo);
  } catch (e) {
    // すべてのエラーを同じ方法で処理
    console.error("Registration failed", e);
    throw e;
  }
}
```

**期待される指摘**:
- [Important] tryブロックが広すぎて、異なる性質のエラーを同じ方法で扱っている
- バリデーションエラー、DB接続エラー、メール送信失敗、決済エラーは異なるリカバリ戦略が必要
- 各処理を個別のtryブロックで囲い、適切なエラーハンドリングを実装すべき

### Case 3: 境界でのバリデーション不足（Critical指摘）

**コード**:
```typescript
app.post("/api/users", (req, res) => {
  const user = req.body;  // バリデーションなし
  db.insertUser(user.name, user.email);
  res.json({ success: true });
});
```

**期待される指摘**:
- [Critical] ユーザー入力のバリデーションがない
- SQLインジェクション、XSS等のセキュリティリスク
- 境界でのバリデーションを追加すべき

### Case 4: 非冪等処理のリトライ（Critical指摘）

**コード**:
```typescript
async function createPayment(amount: number) {
  let retries = 3;
  while (retries > 0) {
    try {
      return await paymentGateway.charge(amount);
    } catch (e) {
      retries--;
      await sleep(1000);
    }
  }
  throw new Error("Payment failed");
}
```

**期待される指摘**:
- [Critical] 決済処理（非冪等）をリトライしている
- 重複課金のリスク
- 冪等性の確保またはリトライの削除が必要

### Case 5: エラー情報の欠損（Important指摘）

**コード**:
```typescript
try {
  const result = await processData(data);
} catch (e) {
  throw new Error("Processing failed");
  // 元のエラーが失われる
}
```

**期待される指摘**:
- [Important] 元のエラー情報が失われている
- デバッグが困難になる
- causeチェーンで元のエラーを保持すべき

### Case 6: 許容されるケース（指摘なし）

**コード**:
```typescript
try {
  cache.warmUp();
} catch (e) {
  // キャッシュのウォームアップ失敗は無視してよい
  // 次回アクセス時に必要に応じて構築される
}
```

**期待される判断**:
- ベストエフォート処理で明示的な理由がある
- 指摘なし
