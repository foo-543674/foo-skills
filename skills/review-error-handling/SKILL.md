# エラーハンドリングレビュー

コードレビュー時にエラーハンドリング・異常系設計の観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. エラーの握りつぶし

- catch ブロックが空、またはログだけ出して処理を続行していないか
- エラーを無視してよい正当な理由があるか（あるならコメントで明示）
- 上位に伝播すべきエラーを途中で止めていないか

### 2. エラーの粒度

- 大きすぎる try-catch で本来キャッチすべきでない例外まで拾っていないか
- 汎用的な例外型（`Exception`, `Error`）でキャッチしていないか（具体的な型でキャッチ）
- エラーの種類に応じた適切なハンドリングをしているか

### 3. エラーの伝播

- エラー情報（元の例外、スタックトレース）が途中で失われていないか
- catch して再 throw する際に原因（cause）をチェーンしているか
- API 境界でのエラー変換（内部エラー → ユーザー向けエラー）が適切か
- 内部の実装詳細がエラーメッセージとして外部に漏れていないか

### 4. 境界でのバリデーション

- システム境界（ユーザー入力、外部 API レスポンス、ファイル読み込み等）で入力を検証しているか
- バリデーション後の内部処理では信頼して過剰な防御をしていないか
- バリデーションエラーのメッセージがユーザーにとって有用か（何が問題で、どうすればよいか）

### 5. リカバリ戦略

- リトライが適切か（冪等な操作のみリトライ、指数バックオフ、上限回数）
- フォールバック値やデフォルト値が妥当か（無言で不正な状態にならないか）
- 部分的な失敗への対応（バッチ処理で一部失敗した場合の挙動）

### 6. リソースの解放

- エラー発生時にリソース（ファイルハンドル、DB接続、ロック等）が確実に解放されるか
- try-with-resources / using / defer 等の言語機構を活用しているか
- finally ブロック内で新たな例外が発生する可能性はないか

### 7. エラーの可観測性

- エラーログに十分なコンテキスト（何を、どの入力で、どの状態で処理中だったか）が含まれているか
- ログレベルが適切か（回復可能 → warn、回復不能 → error）
- 監視やアラートに必要な情報が構造化されているか

## 指摘の出し方

- 「この catch は○○のケースでエラーを握りつぶしている。上位に伝播するか、リカバリ処理を入れるべき」のように具体的なケースを示す
- エラーハンドリングの追加提案は、どのエラーがどの経路で発生しうるかを合わせて説明する
- 過剰な防御的プログラミングには「この時点では○○が保証されているのでチェック不要」と伝える
