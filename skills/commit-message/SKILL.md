# Commit Message Generation

## Why: なぜコミットメッセージの規約が重要か

コミットメッセージの品質は、変更履歴の追跡可能性・バグの原因特定速度・チームのコミュニケーション効率を根本的に決定する。一貫性のないコミットメッセージでは、変更の意図が不明で、過去の変更を追うコストが膨大になる。

**根本的な理由**:
1. **変更履歴の追跡可能性**: 一貫した prefix と明確なメッセージにより、`git log` で特定の種類の変更（機能追加、バグ修正、リファクタリング等）を即座に抽出できる。prefix がないと、コミット一覧から変更の種類を判別できず、履歴の追跡が困難になる。変更履歴の追跡が困難だと、「いつこの変更が入ったか」「どのバージョンにバグ修正が含まれるか」の特定に時間がかかる
2. **バグ原因の高速特定**: バグ発生時に `git bisect` や `git log` で原因のコミットを特定する際、メッセージが明確なら、そのコミットが原因かどうかを即座に判断できる。「fixed bug」のようなメッセージでは、コミットの diff を読まないと判断できず、原因特定に時間がかかる。明確なメッセージは、デバッグ時間を大幅に短縮する
3. **リリースノートの自動生成**: prefix と一貫したメッセージにより、コミット履歴から自動的にリリースノートを生成できる。手動でリリースノートを作成すると、記載漏れ・誤記が発生し、ユーザーへの情報提供が不完全になる。自動生成により、正確で網羅的なリリースノートが即座に作成できる
4. **コミット粒度の適正化**: コミット粒度のルール（prefix が異なる変更は分ける、目的が異なる変更は分ける等）により、意味のある単位でコミットが分割される。ビッグバンコミットでは、revert が困難で、部分的なロールバックができない。適切な粒度のコミットは、変更の取り消しを容易にし、リスクを最小化する
5. **チームのコミュニケーション効率**: 一貫したコミットメッセージにより、チームメンバーが他の開発者の変更を理解するコストが下がる。メッセージが不統一だと、意図を読み取るために diff を詳細に読む必要があり、レビュー・調査のコストが膨大になる。一貫性により、コミュニケーションが効率化される

**コミットメッセージ規約の目的**:
- 変更履歴を追跡可能にし、過去の変更を即座に特定できるようにする
- バグ原因を高速特定し、デバッグ時間を短縮する
- リリースノートを自動生成し、正確な情報提供を実現する
- コミット粒度を適正化し、変更の取り消しを容易にする
- チームのコミュニケーションを効率化し、変更の理解コストを削減する

## ワークフロー

このスキルが呼び出されたら、以下の手順で自動的にコミットを作成する。

1. `git status` と `git diff` で未コミットの変更を把握する
2. 変更内容を分析し、コミット粒度ルールに基づいて意味のある単位にグルーピングする
3. グループごとに以下を繰り返す（順序は変更の依存関係に基づいて判断する）
   a. 対象ファイルを `git add` でステージングする（ファイル内で粒度を分ける必要があれば `git add -p` を使う）
   b. コミットメッセージを生成する
   c. ユーザーにステージング内容とメッセージを提示し、確認を取る
   d. 承認されたら `git commit` を実行する

### 注意事項

- 必ずユーザーの確認を取ってからコミットする
- グルーピングの判断に迷う場合はユーザーに相談する
- 既にステージング済みの変更がある場合は、その意図をユーザーに確認してから進める

## Format

```
[prefix] #ref message
```

- `prefix`: 変更の種類を示す（後述）
- `#ref`: Issue や PR 番号。なければ省略
- `message`: 英語、1 行、絵文字禁止

## Prefix 一覧

| prefix | 用途 |
|--------|------|
| feat | 新規機能の追加 |
| fix | バグ修正 |
| update | 既存機能の仕様変更・更新 |
| docs | ドキュメントの変更 |
| style | コードスタイル・フォーマットの変更（動作に影響しない） |
| improve | 外部インターフェース（trait 等の言語が提供する抽象）に変更が及ぶ既存コードの改善 |
| refactor | 外部インターフェースを変えない内部構造の変更。シグネチャ変更がモジュール内部に閉じている場合も含む |
| test | テストの追加・修正 |
| chore | ビルド、CI、依存関係など開発補助の変更 |

### Prefix 判定プロセス

以下の順序で判定する。**「何を追加したか」ではなく「何を変更したか」を先に確認する。**

1. **変更対象の特定**: 変更しているのは既存のファイル・機能か、新規のファイル・機能か
2. **既存ファイルの変更の場合**: そのファイルの役割を確認する
   - そのファイルはシステムの挙動を決めるものか（コード、設定ファイル、CI 定義、スキル定義等）→ `docs` ではない
   - 仕様・振る舞いが変わるか → `update`
   - 仕様は変えず品質改善か → `improve` or `refactor`
3. **新規ファイルの追加の場合**: そのファイルの役割を確認する
   - ユーザー向け機能を提供するものか → `feat`
   - ビルド、デプロイ、CI、開発補助のためのものか → `chore`
   - テストか → `test`
   - ドキュメントか → `docs`
4. **新規追加と既存変更が混在する場合**: コミットを分ける（後述の粒度ルール参照）

**よくある誤判定**:
- ❌ 既存のスキル定義や設定ファイルに観点を追加 → `feat`（新しい概念を追加したから）
- ✅ 既存のスキル定義や設定ファイルに観点を追加 → `update`（既存の仕様を変更しているから）
- ❌ マークダウンファイルの変更 → `docs`（拡張子が .md だから）
- ✅ マークダウンファイルの変更 → ファイルの役割で判断する（挙動を定義するファイルなら `update` 等）
- ❌ 新規のデプロイスクリプトやビルド設定 → `feat`（新しいファイルだから）
- ✅ 新規のデプロイスクリプトやビルド設定 → `chore`（開発補助だから）

### update と improve の境界

判断基準は **仕様が変わるかどうか**。

- **update**: 仕様の変更を伴う。原則としてテストの変更や追加が発生するはず
- **improve**: 仕様は変えずに、品質・パフォーマンス等を改善する。外部インターフェースの変更は伴うが、既存テストの期待値は変わらない

### improve と refactor の境界

判断基準は **外部レイヤーが触るインターフェースに変更が及ぶかどうか**。

- **refactor**: 関数の効率化、無駄なリソース排除などでシグネチャが一切変わらない場合。シグネチャが変わっても、その変更がモジュール内部に閉じていて、外部に公開された抽象（trait など）が変化しない場合も refactor
- **improve**: 外部インターフェースの変更を伴う改善

## メッセージの書き方

- **必ず英語で書く。日本語禁止**
- **何をしたかではなく、なぜその変更が必要なのかにフォーカスする**
- 1 行に収まる長さにする
- 絵文字は使わない

### 良い例

```
[fix] #123 prevent race condition on concurrent session creation
[improve] reduce N+1 queries in order listing for faster page load
[refactor] extract auth logic to prepare for OAuth provider support
```

### 悪い例

```
[fix] fixed bug          ← なぜ直したのかわからない
[feat] add function      ← 何の機能かわからない
[feat] 🚀 add feature   ← 絵文字禁止
```

## コミット粒度

- 意味のある単位ごとに小さくコミットする
- 複数の意味や目的を含むビッグバンコミットを作らない。同じ prefix でも目的が異なれば分ける
  - 例: 新機能のために既存コードをリファクタした場合、`[refactor]` と `[feat]` は別コミットにする
  - 例: 独立した2つの機能追加は、それぞれ別の `[feat]` コミットにする
  - 例: 同じ種類の改善（決定木追加）を複数の独立した機能（各スキル）に適用する場合も、各機能ごとに分ける
- 「小さく」とはクラス単位ではなく、**使用者の視点で一つの塊として扱える機能・モジュール単位**
  - 使用者はエンドユーザーの場合も、開発者の場合もある（例: チーム開発でドメイン層とインフラ層の担当が分かれている場合、ドメイン側が実装したモジュールの使用者はインフラ側の実装者になる）
  - 独立した機能・モジュール（例: 各スキル、各コンポーネント、各API）は、同じ改善を適用する場合でも、それぞれ別コミット

### 粒度の判定プロセス

グルーピング時に以下を必ず確認する。

1. **prefix が異なる変更が混在していないか** → 分ける
2. **新規追加と既存変更が混在していないか** → 分ける（`[feat]` 新規ファイル + `[update]` 既存ファイル変更は別コミット）
3. **同じ prefix でも目的が異なる変更が混在していないか** → 分ける
   - 「目的」とは: 変更が影響する機能・モジュールの目的
   - 例: review-naming スキルと review-performance スキルは、それぞれ異なる目的（命名の評価、パフォーマンスの評価）を持つ独立した機能
   - 同じ種類の改善を適用する場合でも、独立した機能には異なる目的があるため、分ける
4. **独立した機能・モジュールが混在していないか** → 分ける
   - 複数の独立した機能・モジュール（各スキル、各コンポーネント、各API等）に同じ改善を適用する場合、各機能・モジュールごとに分ける
   - 例: 全スキルに決定木を追加する場合、各スキルごとに1コミット
5. **依存関係があるか** → 依存される側を先にコミットする
