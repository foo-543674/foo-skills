# テスト価値マトリクス: 何をテストし、何をテストしないか

「とりあえず全部テストする」「とりあえずモックする」を避けるために、レイヤー別にテスト価値を見極める。

前提として `../../design-architecture/resources/principles.md` の語彙定義義務と逃げの禁止を適用する。特に **逃げ E (テストでの設計問題吸収)** を避ける。

## 1. 原則: テスト価値の最小化を見極める

すべてのコードに同じ密度でテストを書くのは、時間配分として効率が悪い。**テストで本当に価値が出る場所** と **テストしてもモック検証になるだけの場所** を見極める。

### 価値が高いテストの条件

- ロジックがある (分岐・計算・状態遷移)
- 入力 → 出力の関係を純粋関数として検証できる
- リファクタリングしてもテストが壊れにくい (公開インターフェース基準)
- 失敗したときに「実装の問題」を示唆する (テストの問題ではなく)

### 価値が低いテストの兆候

- ロジックがなく、ポート/モックを呼んだ回数を検証するだけ
- 実装の詳細に密結合していて、リファクタリングするたびに壊れる
- モックの設定がテスト本体より長い
- テストが落ちても「実装が間違ってる」のか「モックの設定が間違ってる」のか切り分けられない

## 2. バックエンドのレイヤー別テスト戦略

| レイヤー | テスト方法 | 価値 |
|---|---|---|
| domain の値オブジェクト・集約メソッド・Validator・Factory | 純粋関数として入力 → 出力を検証。property-based testing 積極利用 | **高: ロジックの本丸**。ここのカバレッジを最大化する |
| application の具体的責務クラス (TransactionRunner, Authorizer, RetryPolicy 等) | 単体テスト。ロジックを持つので意味がある | **高: アプリ固有ロジック** |
| application のユースケース本体 | **書かない** | **低**: ロジックがなくモック検証になるだけ |
| infrastructure (リポジトリ実装、ハッシュ実装等) | 実 DB / 実依存に対する統合テスト (testcontainers 等) | **高**: マッピング・SQL の正しさを担保 |
| interfaces/http (ルーティング、DTO 変換) | フレームワークのテストユーティリティでルーターを直接叩く結合テスト | 中: HTTP 境界の正しさを担保 |
| API 全体 | 言語非依存の統合テスト (Postman / Newman 等) | **高**: 実際に動くかの最終担保 |

### ユースケース本体の単体テストを書かない理由

ユースケースは「Result/Either を `?` / `flatMap` で繋ぐ薄いグルー」に徹する設計 (詳細は `design-architecture/resources/application-layer.md`)。ロジックがない以上、単体テストは「ポートをモックして呼ばれた回数を検証するだけ」になる。

代わりに以下で担保する:
- ロジックの正しさ → domain の単体テスト + application 具体的責務クラスの単体テスト
- つなぎ込みの正しさ → infrastructure 統合テスト + API 統合テスト

ユースケースに分岐・計算が出てきたら、それは抽出のサイン (ドメインルールなら domain へ、アプリ固有なら具体的責務クラスへ)。テストを書きたくなったら設計を疑う。

## 3. フロントエンドのレイヤー別テスト戦略

| レイヤー | テスト方法 | 価値 |
|---|---|---|
| 純粋ロジック (`lib/` 直下) | 単体テスト + property-based testing。日付計算・バリデーション・整形など | **高**: ロジックの本丸 |
| インフラ抽象 (`lib/infra/`) の実装 | テストしない (I/O ラッパなのでテスト対象がない) | **低** |
| サーバー状態リアクティブ単位 (`features/<f>/queries/`, `mutations/`) | **単体テスト**。API クライアントを差し替えて、キー・引数・成功時の `invalidateQueries`・エラー時の挙動を検証する。**API モックライブラリ (MSW 等) は使わずクライアントを差し替える** | **高**: つなぎ込みのロジック |
| 画面状態リアクティブ単位 (`features/<f>/state/`) | **単体テスト**。初期値・更新・派生計算・永続化連動などをリアクティブに検証する | **高**: 状態管理のロジック |
| Generic Presentational (`components/`) | **Storybook でカタログ化 + VRT (Chromatic 等)**。Props のバリエーションを Story として並べる | **高**: 視覚的正しさ |
| Domain Presentational (`features/<f>/components/`) | 同上。空状態 / ロード中 / エラー / 複数行データなどのバリエーションを Story として並べる | **高**: 視覚的正しさ |
| Container (`features/<f>/containers/`) | **テストしない** (薄いグルーなのでモック検証になるだけ) | **低** |
| Page (`routes/`) | **テストしない** (実 HTTP 経路は API 統合テストで担保) | **低** |

### Presentational に API モックを使わない理由

「コンポーネントテストで API モックライブラリ (MSW 等) が必要」になったら設計のサイン。Presentational が API を直接呼んでいないか確認し、呼んでいたら Container に切り出す。

Presentational は Storybook で Props のバリエーションを描画できればテスト目的を果たすので、API モックは不要。「とりあえず MSW でモック + Testing Library で叩く」は React 由来の慣習を雑に持ち込んでいるだけ (逃げ E)。

### サーバー状態リアクティブ単位で API モックを使わない理由

リアクティブ単位の中で `fetch` を直接呼ぶのではなく、API クライアント (openapi-fetch / 自前のクライアント) を経由する設計にしておけば、テスト時はクライアントを差し替えるだけで済む。MSW のような HTTP レベルのモックは「実際の HTTP 動作を再現したい」場合 (= 統合テスト) 専用にする。

## 4. テスト対象の優先順位 (バックエンド + フロント共通)

1. **ビジネスロジック** (条件分岐、計算、状態遷移) — 単体テスト
2. **データアクセス層** (Repository のクエリ) — 統合テスト
3. **API エンドポイント** (リクエスト → レスポンスの一連の流れ) — 統合テスト
4. **ユーザーの主要操作フロー** — E2E (最少)
5. **視覚的正しさ** — Storybook + VRT

**テスト不要な対象**:
- フレームワークが保証する機能 (ルーティング等)
- 単純な getter/setter
- 外部ライブラリの機能そのもの
- 薄いグルー (ユースケース本体、Container、Page)
- I/O ラッパ (インフラ抽象の実装そのもの)

## 5. 「逃げ」のチェックリスト (実装時の自問)

テストを書こうとしてつまずいたら、以下を自問する:

- [ ] モックの設定がテスト本体より長くなっていないか? (なっていたら設計の問題のサイン)
- [ ] テスト対象に分岐・計算・状態遷移があるか? なければそもそも書かない選択肢を検討
- [ ] 「とりあえず E2E でカバー」になっていないか? (Ice Cream Cone)
- [ ] Presentational に API モックを使っていないか? (使っていたら設計の問題のサイン)
- [ ] ユースケース本体の単体テストを書いていないか? (書いていたら抽出すべきロジックがあるか確認)
- [ ] テストの書きにくさを「テストしない」で回避していないか? (回避するなら判断記録に Why を残す)
