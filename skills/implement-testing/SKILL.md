# テスト戦略・実装

プロジェクトまたは機能のテスト戦略を策定し、テストを実装する。

implement-feature がテストファーストで個別機能を実装するのに対し、このスキルはテスト全体の戦略設計と、既存コードへのテスト追加・改善を担う。

## このスキルのスコープ

**扱う** (このスキルで決めること):
- テスト戦略の策定 (テストピラミッド、レイヤー別テスト価値マトリクス)
- 何をテストし何をテストしないかの判断
- テストケースの設計 (同値分割、境界値分析、状態遷移、デシジョンテーブル)
- テストコードの実装 (Arrange-Act-Assert, テストダブル, パラメータ化)
- フィクスチャ・ファクトリの命名と配置
- カバレッジの考え方
- フレイキーテストの予防

**扱わない** (他のスキルに任せる):
- 設計問題の修正 → モックが多すぎる / テストが書けないのは設計の問題のサイン。`design-architecture` / `design-component` に差し戻す
- 本番コードのリファクタリング → `implement-refactor`
- テストファーストでの新規機能実装 → `implement-feature` (このスキルは「既存コードへの追加・改善」と「戦略策定」を担う)
- 受入基準そのものの定義 → `requirements-specification`

スコープ外の項目は **出力しない**。テストコードを書こうとして設計問題に当たったら、テスト側で吸収せず差し戻す。低品質なテスト (モック検証だけのテスト、設計問題を吸収しただけのテスト) は存在自体が害悪なので、書かない方がマシ。

## 基礎原則 (必読)

設計判断のすべてに以下 2 つの基礎原則を適用する。詳細は `../design-architecture/resources/principles.md` を必ず読むこと。

### 1. 語彙定義義務

「テストヘルパー」「テストユーティリティ」のような曖昧な語をモジュール名・クラス名として使わない。「動詞 + 目的語」で 1 文定義できる名前を付ける (例: `UserFixtureBuilder` = 「テスト用 User インスタンスを構築する」)。

### 2. 逃げの禁止と説明責任 (テストでの典型逃げ)

- **逃げ E (テストでの設計問題吸収)**: モックが多すぎる / テストが書けないのは設計の問題のサイン。設計を直す
- **「とりあえず全部テスト」**: 価値の低い場所 (薄いグルー、I/O ラッパ) にもテストを書いて保守コストを膨らませる
- **「とりあえず E2E でカバー」**: テストピラミッドを逆転させる Ice Cream Cone
- **「とりあえずモック」**: モックが本体より長いテストや、何を検証しているか分からないテスト
- **「テストできないから書かない」**: 設計を直すべきところを回避する

逃げざるを得ない場合は判断記録に Why を 3 点 (なぜ / 代替検討 / 見直しトリガー) 残す。

## テスト価値の最小化を見極める

すべてのコードに同じ密度でテストを書くのは効率が悪い。**テストで本当に価値が出る場所** と **テストしてもモック検証になるだけの場所** を見極める。

レイヤー別の具体的なテスト戦略マトリクス (バックエンド + フロントエンド) は `resources/test-value-matrix.md` を必ず参照。

要点:
- **書く**: domain (値オブジェクト・集約・Validator・Factory) / application 具体的責務クラス / infrastructure 統合 / 純粋ロジック / リアクティブ単位 / Presentational の Storybook + VRT
- **書かない**: ユースケース本体 / Container / Page / インフラ抽象実装 (I/O ラッパ)

## Why: なぜテスト戦略が重要か

テスト戦略の品質は、コードベースの品質・変更容易性・開発速度を根本的に決定する。適切なテスト戦略なしでは、テストの効果が最大化されず、保守コストが膨大になる。

**根本的な理由**:
1. **リグレッション防止の効率化**: 適切なテスト戦略により、最小限のテストで最大限のリグレッション防止が実現できる。テストピラミッド（ユニット多数、統合中程度、E2E 最少）を無視すると、遅い E2E テストばかりになり、フィードバックループが遅延し、開発速度が低下する。適切な粒度のテストは、変更の影響を即座に検出し、修正コストを最小化する
2. **テスト保守コストの制御**: テスト戦略がないと、実装の詳細をテストする脆いテストが増え、リファクタリングのたびにテストが壊れる。テスト保守コストが実装保守コストを上回り、テストが負債になる。適切なテスト戦略は、公開インターフェースをテストし、内部実装の変更に強いテストを設計する
3. **品質の可視化と改善**: テストカバレッジ・テスト結果により、品質が可視化される。可視化されない品質は改善されない。テスト戦略により、何をどの粒度でテストするかが明確になり、未テスト箇所が発見され、品質改善のサイクルが回る。テスト戦略なしでは、場当たり的なテストが蓄積し、品質が不透明になる
4. **設計品質の強制**: テスト可能な設計は、疎結合・単一責任原則が守られた設計である。テスト戦略により、テスタビリティが設計の評価軸になり、密結合・God オブジェクトの発生が防がれる。テスト不可能なコードは設計の問題を示唆する。テスト戦略は設計品質を担保する仕組み
5. **開発速度の持続性**: 適切なテストがあれば、リファクタリング・機能追加が安全に実行できる。テストなしでは、変更が怖くなり、技術的負債が蓄積し、開発速度が指数関数的に低下する。テスト戦略は、長期的な開発速度の持続可能性を担保する投資

**テスト戦略の目的**:
- リグレッション防止を最小コストで実現する
- テスト保守コストを制御し、テストを資産にする
- 品質を可視化し、継続的改善を可能にする
- 設計品質を強制し、テスタビリティを担保する
- 開発速度の持続可能性を確保する

## プロセス

### 1. 既存テスト状況の把握

テスト戦略を策定する前に、現在のテスト状況を理解する。

**確認すべきこと**:
- テストフレームワーク・テストランナーの構成
- テストの配置規則（`__tests__/`, `*.test.ts`, `*.spec.ts` 等）
- 既存テストの粒度・種類（ユニット、統合、E2E）
- カバレッジの現状
- CI/CD でのテスト実行方法
- テストヘルパー・ファクトリの有無

### 2. テスト戦略の策定

テストピラミッドに基づいて、何をどの粒度でテストするかを決める。**ただし「全部テストする」ではなく、テスト価値の最小化を見極める** (詳細は `resources/test-value-matrix.md`)。

**テストピラミッドの層**:

| 層 | 量 | 対象 | 実行速度 |
|---|---|------|---------|
| ユニットテスト | 最多 | ドメインロジック、ビジネスルール、データ変換、リアクティブ単位 | 最速 |
| 統合テスト | 中程度 | コンポーネント間の連携、Repository + DB、API 全体 | 中 |
| E2E テスト | 最少 | ユーザーシナリオの主要パス | 最遅 |
| 視覚回帰テスト (フロント) | 中程度 | Presentational の Props バリエーション (Storybook + VRT) | 中 |

**テスト対象の優先順位**:
1. ビジネスロジック (条件分岐、計算、状態遷移) — ユニットテスト
2. データアクセス層 (Repository のクエリ) — 統合テスト
3. API エンドポイント (リクエスト → レスポンスの一連の流れ) — 統合テスト
4. ユーザーの主要操作フロー — E2E テスト
5. UI の視覚的正しさ — Storybook + VRT

**テストしない対象 (書かない判断を明示する)**:
- フレームワークが保証する機能 (ルーティング等)
- 単純な getter/setter
- 外部ライブラリの機能そのもの
- **薄いグルー: ユースケース本体、Container、Page** (ロジックがないのでモック検証にしかならない)
- **I/O ラッパ: インフラ抽象の実装そのもの** (テスト対象がない)

詳細なレイヤー別マトリクスは `resources/test-value-matrix.md` を参照。

### 3. テスト設計

テストケースを設計する。

**テストケース設計の手法**:
- 受け入れ基準からのテストケース導出
- 同値分割: 同じ振る舞いを期待する入力グループを特定し、各グループから代表値でテスト
- 境界値分析: 境界の前後でテスト（0, 1, 最大値, 最大値+1）
- 状態遷移: 状態マシンの各遷移パスをテスト
- デシジョンテーブル: 複数条件の組み合わせを網羅

**テスト名の設計**:
- テスト実行結果に仕様を説明する文章が出力されること
- 具体的な条件と期待結果を含む
- 良い例: `should reject registration when email has no at sign`
- 悪い例: `should return error`, `test validation`

### 4. テストコードの実装

設計したテストケースを実装する。

**テスト構造**:
```
Arrange: テストの前提条件をセットアップする
Act:     テスト対象を実行する
Assert:  結果を検証する
```

**テストダブルの使い方**:
- スタブ: 外部依存の戻り値を固定する（DB, API 等）
- モック: 相互作用の検証が本当に必要な場合のみ使う
- テストダブルの振る舞いが実際の依存先と乖離しないよう注意する
- モックが多すぎるテストは設計の問題を示唆する

**パラメータ化テスト**:
- 同じ構造で入力だけが異なるテストが並んだら、パラメータ化テストにまとめる
- ただし、テストケース間で Arrange が大きく異なる場合は個別テストの方が読みやすい

### 5. テスト品質の確認

実装したテストの品質を確認する。

**品質チェック**:
- テストが本当に失敗すべきとき失敗するか（対象コードを壊して確認）
- フレイキーテストの要因がないか（時刻依存、乱数依存、並行処理依存）
- テスト間に実行順序の依存がないか
- テストの実行速度が許容範囲内か
- テストコード自体の可読性・保守性

**カバレッジの考え方**:
- カバレッジは「テストされていない箇所の発見」に使う
- カバレッジの数値目標を達成すること自体を目的にしない
- 条件分岐カバレッジ（各分岐の真偽両方を通過）を重視する

## テスト実装完了チェックリスト

テスト実装完了時に以下を確認する。

### テスト戦略の達成確認
- [ ] テストピラミッドに従っている（ユニット多数、統合中程度、E2E最少）
- [ ] テスト対象の優先順位に従っている（ビジネスロジック→データアクセス→API→主要フロー）
- [ ] テスト不要な対象（フレームワーク機能、単純なgetter/setter等）をテストしていない

### テストケース設計の確認
- [ ] 受け入れ基準からテストケースを導出している
- [ ] 同値分割・境界値分析を実施している
- [ ] 状態遷移がある場合、各遷移パスをテストしている
- [ ] 正常系・異常系・境界条件を網羅している

### テスト品質の確認
- [ ] review-testingの観点: テスト名で仕様を説明、独立性、テストダブルの適切な使用
- [ ] テストが本当に失敗すべきとき失敗する（対象コードを壊して確認）
- [ ] フレイキーテストの要因がない（時刻依存、乱数依存、並行処理依存）
- [ ] テスト間に実行順序の依存がない
- [ ] テストの実行速度が許容範囲内

### テスト構造の確認
- [ ] Given-When-Then（Arrange-Act-Assert）で構造化している
- [ ] テストダブルは外部依存の置き換えに限定している
- [ ] パラメータ化テストで同じ構造のテストをまとめている（適切な場合）

### カバレッジの確認
- [ ] カバレッジを「テストされていない箇所の発見」に使っている
- [ ] カバレッジの数値目標を達成すること自体を目的にしていない
- [ ] 条件分岐カバレッジ（各分岐の真偽両方を通過）を重視している

### テストコードの保守性確認
- [ ] テストコード自体の可読性が高い
- [ ] テストヘルパー・ファクトリで重複を排除している
- [ ] テストデータがテスト内に明示されている（Mystery Guestを避ける）

## よくあるアンチパターン

- **Ice Cream Cone**: E2E テストが最も多く、ユニットテストが少ない (ピラミッドの逆転)
- **Testing Implementation**: 公開インターフェースではなく内部実装をテストする
- **Fragile Test**: 実装の微小な変更でテストが壊れる
- **Mystery Guest**: テストデータがテストファイルの外にあり、テスト単体で理解できない
- **Shared Fixture**: テスト間で可変な共有状態を使い、テストの独立性が損なわれる
- **Coverage Worship**: カバレッジの数値を上げることが目的になり、意味のないテストが増える
- **薄いグルーの単体テスト**: ユースケース本体・Container・Page のような薄いグルーをモック検証で「テストした気になる」
- **Presentational に API モック**: Storybook + VRT で済むはずの Presentational に MSW 等を差し込む (Container と Presentational の区分けが甘いサイン)
- **モック設定が本体より長い**: 設計が密結合になっているサイン
- **「テストできないから書かない」で逃げる**: 設計を直すべきところを回避する (説明責任なしの場合)
- **「テストヘルパー」「テストユーティリティ」のひと括り**: 語彙定義義務違反
