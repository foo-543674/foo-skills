# データモデル設計

要件をもとにデータモデルを設計する。

review-data-modeling の観点を設計時に先取りし、正規化・リレーション・履歴管理等の問題を事前に回避する。

## プロセス

### 1. 既存データモデルの把握

既存システムがある場合、現在のデータモデルを理解する。

**確認すべきこと**:
- 既存テーブル・コレクションの構造
- リレーション・外部キーの設定
- インデックスの設計
- マイグレーションの管理方法
- 既存の命名規則

### 2. エンティティの識別

要件から永続化が必要なエンティティを抽出する。

**識別の手順**:
1. 要件に登場する「名詞」を候補として列挙する
2. 各候補が独立したライフサイクル（生成・変更・削除）を持つか判断する
3. Entity（独自の識別子を持ち状態が変化する）と Value Object（識別子を持たず値で比較する）を区別する
4. 集約の境界を決める（トランザクション整合性を保証する範囲）

**命名規則**:
- テーブル名・カラム名はプロジェクトの既存規則に合わせる
- 既存規則がない場合は snake_case を推奨
- 省略せず、ドメインの用語をそのまま使う

### 3. リレーション設計

エンティティ間の関係を定義する。

**設計の判断基準**:
- 1:N か N:M か（中間テーブルの必要性）
- 外部キー制約を付けるか（参照整合性の保証レベル）
- カスケード削除の方針（安易にカスケードしない）
- NULL 許容の方針（NULL を避け、意味のあるデフォルト値や状態で表現できないか）

**集約をまたぐ参照**:
- 異なる集約間は ID による参照のみ（オブジェクト参照しない）
- 外部キー制約を張るかは、整合性要件とパフォーマンスのトレードオフ

### 4. 正規化と非正規化の判断

**正規化（Command 側の原則）**:
- 第3正規形を基本とする
- データの重複を排除し、更新異常を防ぐ
- 正規化のコストが利点を上回る場合のみ非正規化する

**非正規化（Query 側の最適化）**:
- 読み取りパフォーマンスのために必要な場合に限定する
- 非正規化したデータの同期方法を必ず設計する
- CQRS を採用している場合、Query 側は非正規化が自然

正規化 vs 非正規化の判断は ADR に記録する。

### 5. 履歴・監査の設計

データの変更履歴や監査ログの要件を設計に反映する。

**Immutable モデリング**:
- 変更を上書きではなく、新しいレコードの追加として表現する
- 有効期間（valid_from, valid_to）で時点指定クエリを可能にする
- 状態遷移をイベントとして記録する

**履歴が不要な場合**:
- updated_at カラムで最終更新のみ追跡すれば十分か判断する
- 論理削除（soft delete）の要否を検討する

### 6. マイグレーション戦略

既存データの移行と、スキーマ変更の安全な適用方法を計画する。

- 破壊的変更（カラム削除、型変更）はどう段階的に適用するか
- データ移行が必要な場合の手順とロールバック方法
- ダウンタイムの許容範囲と、オンラインマイグレーションの必要性

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- 正規化 vs 非正規化のトレードオフ判断
- 論理削除 vs 物理削除の選択
- Immutable モデリングの採用・不採用
- ID 生成戦略の選択（UUID, ULID, 連番等）
- 集約の境界設定で複数の選択肢がある場合

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **God Table**: 1 テーブルにあらゆる属性を詰め込む
- **Polymorphic Association**: type カラムで参照先テーブルを切り替える（外部キー制約が効かない）
- **EAV (Entity-Attribute-Value)**: 柔軟性のために正規化を完全に放棄する
- **Premature Denormalization**: パフォーマンス問題が発生する前から非正規化する
- **Missing Index**: クエリパターンを考慮せずにインデックスを設計しない
- **Soft Delete Everywhere**: すべてのテーブルに論理削除を入れ、クエリが複雑化する
