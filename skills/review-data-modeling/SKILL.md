# データモデリングレビュー

コードレビュー時にデータベース設計とエンティティモデリングの観点で指摘すべきポイントを定義する。RDB を主な対象とするが、原則はドキュメント DB 等にも応用できる。

このスキルはイミュータブルデータモデリングを基本思想とする。Nullable カラムの精査とテーブル分離により UPDATE を排除し、ほぼすべての操作を INSERT で表現するデータ構造を目指す。これによりスケーラビリティの向上と変更履歴の自然な保持が得られる。ただしこの思想はトレードオフを伴い、テーブル数の増加やクエリの複雑化を招く場合がある。プロジェクトの規模・要件に応じた判断が必要。

**UPDATE / DELETE に対する基本的な考え方**: 現実世界には UPDATE や DELETE に相当する事象は存在しない。システムの UPDATE は「最初からその状態だったことにする」という過去への遡及的な改変であり、DELETE も同様に「最初から存在しなかったことにする」操作である。正当なビジネスフローの中で UPDATE や DELETE が必要になっている場合、それはモデリングに問題がある兆候と捉える。ユーザーの入力ミス修正などイレギュラー対応として UPDATE が必要になるケースはあるが、それは例外的な運用操作であり、通常のビジネスフローとは区別して設計する。

## レビュー観点

### 1. Nullable カラムの精査とテーブル分離

- Nullable カラムが「本当に optional な値」なのか「INSERT タイミングで未確定なだけでビジネス上は必要な値」なのかを区別しているか
  - 例: `employees.retirement_date` が Nullable → 退職日は「未確定」なのではなく「退職というイベントがまだ発生していない」。`retirements` テーブルを別途作るべき
- 後者の場合、そのカラムは別テーブルに分離し、イベント発生時に INSERT する設計にできないか
- この分離を徹底すると、ほぼすべての操作が INSERT のみで表現され、UPDATE が排除される。結果としてロック競合の削減・スケーラビリティの向上・変更履歴の自然な保持が得られる
- ただしテーブル分離はJOIN の増加を招く。読み取りパフォーマンスが重要な場面では CQRS の Read Model で非正規化した View を用意する等の対策を検討する

### 2. エンティティ設計と正規化の判断

- エンティティがビジネス概念を正しく反映しているか
- 正規化のレベルが適切か（第3正規形を基本とし、必要に応じて非正規化）
- 非正規化する場合、その理由（パフォーマンス、読み取りパターン）が明確か
- ただし CQRS の Read Model では積極的な非正規化が正しい設計。Command 側の正規化と Read 側の非正規化を混同しない

### 3. リレーション設計

- 1:N, N:M のリレーションが正しくモデル化されているか
- N:M の中間テーブルに属性が必要ないか（単なるリンクテーブルで十分か）
- 外部キー制約が適切に設定されているか
- カスケード削除の影響範囲を把握しているか
- ただしマイクロサービス間では外部キー制約が使えない。結果整合性の設計が別途必要

### 4. マイグレーション戦略

- カラム追加・削除が既存データと互換性を保つか
- NOT NULL カラムの追加にデフォルト値があるか
- カラム名変更・型変更を安全に行う手順が設計されているか（expand-contract pattern）
- ロールバック可能なマイグレーションになっているか
- 大量データのテーブルに対するマイグレーションのロック影響を考慮しているか

### 5. インデックス設計

- WHERE / JOIN / ORDER BY で使われるカラムにインデックスがあるか
- 複合インデックスのカラム順序が検索パターンに合っているか
- 不要なインデックスが残っていないか（書き込みパフォーマンスへの影響）
- ただしインデックスは実際のクエリパターンと計測に基づいて設計する。推測で追加しない

### 6. 削除の設計

- 「削除」と呼ばれている操作が本当に削除なのかを分析する。多くの場合「削除」ではなく「状態遷移」である（例: 退会、無効化、アーカイブ）。状態遷移ならイベントテーブルへの INSERT で表現する（イミュータブルデータモデリングとの整合）
- 本当に削除であるなら、物理削除が最もシンプルで適切
- 復旧要件や監査要件など非機能要件でデータ保持が求められる場合は、退避テーブル（アーカイブテーブル）への移動を検討する。元テーブルはクリーンに保ちつつ、復旧可能性を担保できる
- 論理削除（`deleted_at`）は全クエリへの `WHERE deleted_at IS NULL` 付与漏れ、ユニーク制約との干渉、テーブル肥大化を招く。上記のいずれかのアプローチで代替できるため、論理削除が妥当なケースはほぼない

### 7. 時系列データと履歴管理

- 変更履歴が必要なデータにイベントソーシングまたは履歴テーブルが設計されているか
- 「現在の状態」と「履歴」のクエリパターンが分離されているか
- 有効期間（valid_from / valid_to）を持つデータの重複・ギャップ防止策があるか
- 監査ログ（誰が・いつ・何を変更したか）の要件が満たされているか

## よくあるアンチパターン

- **EAV（Entity-Attribute-Value）の安易な採用**: ユーザー定義属性やプラグインシステム等、スキーマの動的拡張が本質的に必要な場面では有効な選択肢だが、固定的なビジネス要件を EAV で表現するとクエリの複雑化・型安全性の喪失・データ整合性の担保困難を招く
- **Polymorphic Association**: 1つの外部キーカラムが複数テーブルを参照する（型安全性の喪失）
- **God Table**: 無関係な概念が1つのテーブルに詰め込まれている
- **Nullable as Pending**: ビジネス上必要な値を Nullable カラムで「未確定」として同一テーブルに持たせ、後から UPDATE する設計
- **Soft Delete as Default**: 分析なしに論理削除をデフォルト採用し、実際には「状態遷移」であるものを削除フラグで表現している
- **Implicit State Machine**: 状態遷移がカラムの値の組み合わせで暗黙的に表現されている

## 指摘の出し方

- データモデルの問題はクエリパターンやデータ量の想定と合わせて議論する
- 正規化 / 非正規化の議論では読み書き比率と想定データ量を根拠にする
- マイグレーションの指摘では具体的な手順（expand-contract 等）を示す
- なぜその設計を推奨するかの理由を必ず添える。「将来の変更容易性」だけでなく「現時点での運用上の実害」も示す
