# アプリケーション層: 営みのつなぎ合わせとユースケースの薄グルー化

前提として `principles.md` の語彙定義義務と逃げの禁止を必ず適用する。

## 1. アプリケーション層が引き受ける関心事

アプリケーション層は「現実の営みをアプリケーションにしたから初めて発生した文脈・ロジック」を引き受ける。具体的には以下:

- 永続化のオーケストレーション (トランザクション開始・コミット・ロールバック、ポート呼び出し)
- 構造化ログ・トレースの取得ポイント (ユースケース開始/終了、失敗時のドメインエラー記録)
- メトリクス (処理時間、失敗率)
- ユースケースレベルの認可 (「このプリンシパルでこのリソースを触れるか」を集約取得時に強制する)
- レート制限の適用ポイント
- 外部入力 DTO を domain の入力型に詰め替える
- Validator が必要とする外部事実を「形 A / 形 B / 形 C」のいずれかでロードして渡す
- 出力結果を domain 型から外部向け DTO に詰め替える

これらは紙の帳簿時代には存在せず、ソフトウェアにしたから必要になった関心事。

## 2. やらないこと

- ビジネスルールの判定 (現実の営みのルールなので domain に委ねる)
- 不変条件の検証 (同上)
- HTTP / JSON / Cookie の知識を持つこと (interfaces 層に閉じる)
- DB ベンダー固有の SQL や ORM の型を扱うこと (infrastructure 層に閉じる)

## 3. アプリケーション層に置くもの

### 3.1 永続化ポート (インターフェース)

**定義**: 「集約を永続化境界に対して保存・取り出しする」インターフェース。例: `UserRepository`, `ExerciseRepository`, `SessionStore`。

これらは「アプリケーションが永続化のために外界に求める能力」を表現する。実装は infrastructure 層が提供する。

domain に置かない理由: 永続化は紙の帳簿時代には存在しなかった営み。詳細は `domain-modeling.md` の「営みの起源によるレイヤーの定義」参照。

### 3.2 具体的責務クラス (「アプリケーションサービス」とまとめない)

「アプリケーションサービス」を構造的概念として使わない (語彙定義義務違反)。代わりに、各責務を具体的責務クラスとして個別に表現する。

| 責務クラスの例 | 1 文定義 |
|---|---|
| TransactionRunner | 渡された手続きを 1 つのトランザクション境界の中で実行する |
| Authorizer | プリンシパルとアクションを受け取り、許可/拒否を返す |
| RetryPolicy | 失敗した手続きを規定の規則に従って再実行する |
| RateLimitCounter | プリンシパルごとの呼び出し回数をカウントし、上限超過を判定する |
| RequestLoggingDecorator | ユースケース呼び出しの前後で構造化ログを出力する |
| AuditLogSink | 監査対象イベントを永続化先に追記する |

新しい責務を追加するときは、必ず「動詞 + 目的語」で 1 文定義してから命名する。クラス名から責務が想像できないなら、その責務はまだ言語化されていない。

### 3.3 ユースケース (Command / Query の入口)

**定義**: 外部からのリクエストに対応する 1 つの操作の入口。**ロジックを持たない薄いグルー** に徹する (詳細は次節)。

例: `CreateExerciseUseCase`, `StartSessionUseCase`, `ListSessionsByDateUseCase`。

## 4. ユースケースは薄いグルー

### 原則

ユースケース本体は **ロジックを持たない**。やることは以下だけ:

1. 入力 DTO を domain の入力型に詰め替える
2. 具体的責務クラス (TransactionRunner, Authorizer 等) と domain の Validator・Factory・集約メソッドを順に呼ぶ
3. それぞれの結果を Result/Either の `?` / `flatMap` / `map` でつないで伝搬する
4. 最終結果を出力 DTO に詰め替えて返す

### 抽出のサイン

ユースケースに **分岐・計算・条件判定** が出てきたら、それは抽出のサイン:

- ドメインルール由来の判定なら → domain 層の Validator や集約メソッドへ
- アプリケーション固有のロジック (複数操作のトランザクション境界の張り方、リトライ条件、ログのフォーマット等) なら → application 層の具体的責務クラスへ

この規律により、ユースケース本体は「Result/Either を縦に並べて `?` / `flatMap` でつなぐだけ」の薄いグルーになる。

### ユースケースの単体テストを書かない

ロジックがない以上、ユースケースの単体テストは「ポートをモックして呼ばれた回数を検証するだけ」になり、書く価値が低い割に保守コストが高い。代わりに以下をテストする:

| 対象 | テスト方法 | 価値 |
|---|---|---|
| domain の値オブジェクト・集約メソッド・Validator・Factory | 純粋関数として入力 → 出力を検証 (property-based testing 推奨) | 高: ロジックの本丸 |
| application の具体的責務クラス (TransactionRunner 等) | 単体テスト (ロジックがあるので意味がある) | 高: アプリ固有ロジック |
| ユースケース本体 | 書かない | 低: モック検証になるだけ |
| 全体の統合 | infrastructure を差したまま test DB に対する結合テスト + API 統合テスト | 高: 実際に動くかの担保 |

詳細は `implement-testing` スキルの `test-value-matrix.md` を参照。
