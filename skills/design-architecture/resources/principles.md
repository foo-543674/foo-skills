# 設計の基礎原則: 語彙定義義務と逃げの禁止

設計判断 (アーキテクチャ・コンポーネント・データモデル・API・テスト・要件仕様) を行うすべての場面で適用する 2 つの基礎原則。design-architecture / design-component / design-data-model / design-api / requirements-specification / implement-testing から参照される正典。

## 1. 語彙定義義務

### 原則

リポジトリで構造的な語彙 (層名・モジュール名・クラス名・ロール名) として使う単語は、その単語 1 つで何を意味するかが厳密に定義されていなければならない。

**装飾語 (ドメイン名・形容詞) で範囲を狭めるのは定義ではない。**

### なぜ必要か

曖昧な語 (サービス、マネージャ、ヘルパー、ユーティリティ、プロセッサ等) を構造的概念として使うと、定義がないまま「なんとなく合うもの」を吸い込み続け、単一責任を保てなくなる。最終的には「変更理由が複数ある」「テストできない」「分割の指針がない」という God オブジェクトに退化する。

「思いつかないからとりあえずこれ」を避けるための最初の防壁が、語彙の厳密定義義務。

### 判定テスト

その単語をリポジトリの用語集 (glossary) に書いたとき、**「動詞 + 目的語」の責務として 1 文で書けるか**。書けないならその単語を構造的概念として使ってはいけない。

| 語 | 1 文定義の試み | 判定 |
|---|---|---|
| サービス | 「〜に関するロジックを集めたもの」(動詞がない) | ❌ |
| マネージャ | 「〜を管理するもの」(管理 = 何でも入る) | ❌ |
| ヘルパー / ユーティリティ | 「便利な関数の置き場所」 | ❌ |
| プロセッサ | 「〜を処理するもの」(処理 = 何でも入る) | ❌ |
| ハンドラ | 「HTTP リクエストを受けてレスポンスを返す」(MVC の文脈なら可) / 「イベントを処理する」(不可) | △ 文脈次第 |
| Validator | 「入力と外部事実を受け取り、検証済み入力 or エラーを返す」 | ✅ |
| Factory | 「検証済み入力から集約を生成する (失敗しない)」 | ✅ |
| Repository | 「集約を永続化境界に対して保存・取り出しする」 | ✅ (リポジトリの用語集に明示すること) |
| TransactionRunner | 「渡された手続きを 1 つのトランザクション境界の中で実行する」 | ✅ |
| Authorizer | 「プリンシパルとアクションを受け取り、許可/拒否を返す」 | ✅ |
| RetryPolicy | 「失敗した手続きを規定の規則に従って再実行する」 | ✅ |

### 装飾語による偽装の見破り方

`ExerciseService` のように装飾語をつけても、語そのものが定義されていなければ意味がない。「エクササイズのサービス = エクササイズに関わる全て」となり、結局肥大化する。

- ❌ `ExerciseService` — 「エクササイズの何をするもの?」が答えられない
- ❌ `UserManager` — 「ユーザーを何するもの?」が答えられない
- ✅ `ExerciseRepository` — 「Exercise 集約を永続化境界に対して保存・取り出しする」
- ✅ `PasswordHashingPolicy` — 「平文パスワードを規定のアルゴリズムでハッシュ化する」
- ✅ `RequestLoggingDecorator` — 「ユースケース呼び出しの前後で構造化ログを出力する」

### 既定の禁止語リスト

以下の語は、リポジトリの用語集に「動詞 + 目的語」の 1 文定義を書けない限り、構造的概念として使ってはいけない:

- サービス / Service
- マネージャ / Manager
- ヘルパー / Helper
- ユーティリティ / Utility / Util
- プロセッサ / Processor
- ハンドラ / Handler (HTTP リクエストハンドラのような確立した文脈以外で)
- ロジック (層名や概念名としては可、クラス名としては不可)
- ワーカー / Worker
- エンジン / Engine

「ドメインサービス」「アプリケーションサービス」も同じ理由で構造的概念としては使わない。代わりに具体的責務クラスの集合として説明する (例: ドメイン層の Validator + Factory + 集約メソッド、アプリケーション層の TransactionRunner + Authorizer + RequestLoggingDecorator など)。

### 新しい構造的語を導入するときの手順

1. その語の責務を「動詞 + 目的語」で 1 文に書く
2. リポジトリの用語集 (`docs/glossary.md` 等) にその定義を追加する。用語集が存在しなければ作成を提案する
3. その定義に収まらないものを後から追加するときは、定義を書き換えるか別の語を導入する (定義を曖昧化させない)

## 2. 逃げの禁止と説明責任

### 原則

設計判断において、「思いつかないからとりあえずこれ」「難しいから簡単な方に逃げる」を避ける。逃げざるを得ない場合は、判断記録に Why を残す説明責任を負う。

### 典型的な逃げパターン

#### 逃げ A: 曖昧な箱への退避

**問題**: 責務の置き場所が決まらないとき、「サービス」「マネージャ」「ワークフロー」のような曖昧な箱に放り込んで終わらせる。

**なぜ駄目か**: 語彙定義義務に違反し、責務が言語化されないまま肥大化する。後から見ると「ここに何が入っているか」が誰にも分からなくなる。

**代わりにすべきこと**: 責務を「動詞 + 目的語」で言語化してから具体的責務クラスを作る。

#### 逃げ B: 数値的構造の文字列化

**問題**: 日付・金額・期間・座標など、数値の構造を持つデータを、parse やタイムゾーンの扱いが難しいという理由で文字列として持つ。

**なぜ駄目か**: 比較・計算・並び替え・範囲検索ができなくなり、本来 DB やライブラリが担保してくれる境界条件のバグが頻発する。「問題を後段に押し付けただけ」で、後段の各所で同じ parse 処理を再発明することになる。

**代わりにすべきこと**: 数値の構造を持つデータは数値型 (DATE, DECIMAL, DURATION 等) として扱い、TZ や parse の責務を「どの層でやるか」を仕様で確定させる。

#### 逃げ C: 直交する次元の排他列挙化

**問題**: 「重量 + 回数」「時間 + 任意で重量」のように独立した属性の組み合わせを、`weight_reps` `time_only` のような排他列挙で表現する。

**なぜ駄目か**: 組み合わせが増えるたびに列挙を拡張する必要があり、既存データのマイグレーションも発生する。「新しい組み合わせを追加できない」という設計的天井を作る。

**代わりにすべきこと**: 直交する次元 (重量 / 回数 / 時間 / セット数 / 符号 / etc.) を独立した属性として識別し、組み合わせは「どの属性が必須か」のセットとして表現する。

#### 逃げ D: 安易な抽象化への退避

**問題**: 1-2 回しか出てこないパターンを「将来また出るかも」と推測して抽象化する。または、責務分割が決まらないときに「ワークフロー」「サービス」のような大きな箱を作って中で何でもできるようにする。

**なぜ駄目か**: 過剰抽象化は具体性を失わせ、変更コストを上げる。曖昧な箱は語彙定義義務違反。

**代わりにすべきこと**: Rule of Three を守る (3 回出るまで具象のまま)。責務分割は具体的責務名で表現する。

#### 逃げ E: テストでの設計問題の吸収

**問題**: 設計が悪くてテストが書きにくいとき、モックを大量に積んで無理やり通す、または「テストできないから書かない」で済ませる。

**なぜ駄目か**: テストの書きにくさは設計の問題のサインを潰してしまう。本来直すべき設計を直さず、テスト側に負債を作る。

**代わりにすべきこと**: モックが多すぎる / テストが書けないのは設計の問題のサイン。設計を直す。詳細は implement-testing スキルを参照。

#### 逃げ F: 言語の例外機構による検証エラーの表現

**問題**: 入力検証の失敗やビジネスルール違反を、言語の例外機構 (throw / panic / raise) で表現する。

**なぜ駄目か**: 「未検証 / 検証済み」を型レベルで区別できなくなり、検証していない値が集約に紛れ込むことをコンパイル時に防げない。例外は隠れた制御フローになり、呼び出し側がエラーを処理し忘れる。

**代わりにすべきこと**: ユーザー入力・外界由来の失敗は Result/Either 型で表現する。例外/panic はプログラマのバグを表す中断 (到達したらバグと言い切れる箇所) 専用にする。

### 説明責任 (どうしても逃げる場合)

逃げの判断を採用する場合、以下を必ず判断記録 (ADR / 設計ドキュメント / コードコメント) に残す:

1. **なぜそうせざるを得ないのか** (制約・要因の具体)
2. **本当にそれ以外の方法がないか** を検討した経緯 (検討したが棄却した代替案)
3. **将来この制約が外れたら見直すトリガー** (「ユーザー数が 10 万を超えたら」「TZ 機能が要件に入ったら」等)

この 3 点が書けない逃げは採用しない。

### 設計時のセルフチェック

設計判断のたびに以下を自問する:

- [ ] この語を 1 文で定義できるか?
- [ ] この置き場所は「思いつかないからとりあえず」になっていないか?
- [ ] 数値構造を文字列化していないか?
- [ ] 直交する次元を排他列挙で潰していないか?
- [ ] 抽象化は Rule of Three を満たしているか?
- [ ] 検証エラーを例外で逃がしていないか?
- [ ] 逃げざるを得ないなら Why を 3 点記録したか?
