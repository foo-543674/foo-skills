# アーキテクチャ設計

要件をもとにシステムのアーキテクチャを設計する。

review-architecture の観点を設計時に先取りし、レビューで指摘されるような問題を事前に回避する。

## このスキルのスコープ

**扱う** (このスキルで決めること):
- バックエンドのレイヤー定義 (営みの起源で分ける)
- ポート配置 (現実の概念ポート vs アプリの営みポート)
- ドメインモデリングの大枠 (置き場所優先順位、集約生成 4 役割、外部事実の渡し方)
- アプリケーション層の関心事と具体的責務クラスの輪郭
- 境界文脈 (Bounded Context) の分割
- CQRS の適用判断
- エラー境界の方針 (Result/Either による表現方針)
- 技術選定 (システム全体のフレームワーク・ランタイム・ストレージ等)
- バックエンドのレイヤー単位ディレクトリ構造

**扱わない** (他のスキルに任せる):
- フロントエンドのコンポーネント区分・ファイル構造・状態管理 → `design-component`
- レイヤー内のクラス・モジュールの細かい責務分割やインターフェース設計 → `design-component`
- エンティティの属性詳細・テーブル構造・正規化・データ型選定・マイグレーション → `design-data-model`
- HTTP エンドポイント形状・リクエスト/レスポンス DTO・バージョニング → `design-api`
- テスト戦略・テスト実装 → `implement-testing`
- 受入基準・NFR 定量化 → `requirements-specification`

スコープ外の項目は **出力しない**。成果物テンプレートで該当する箇所にはプレースホルダ (`→ <スキル名> で決定`) を置き、低品質な内容を埋めるより空欄にする。低品質な出力は存在自体が害悪なので、書かない方がマシ。

## 基礎原則 (必読)

設計判断のすべてに以下 2 つの基礎原則を適用する。詳細は `resources/principles.md` を必ず読むこと。

### 1. 語彙定義義務

リポジトリで構造的な語彙 (層名・モジュール名・クラス名・ロール名) として使う単語は、その単語 1 つで何を意味するかが厳密に定義されていなければならない。装飾語 (ドメイン名・形容詞) で範囲を狭めるのは定義ではない。

**判定テスト**: その単語を「動詞 + 目的語」の責務として 1 文で書けるか。書けないならその単語を構造的概念として使ってはいけない。

**既定の禁止語**: サービス / マネージャ / ヘルパー / ユーティリティ / プロセッサ / ワーカー / エンジン / (確立した文脈以外の) ハンドラ。「ドメインサービス」「アプリケーションサービス」も同じ理由で構造的概念としては使わない。代わりに具体的責務クラスの集合として説明する (TransactionRunner, Authorizer, RequestLoggingDecorator など)。

### 2. 逃げの禁止と説明責任

「思いつかないからとりあえずこれ」「難しいから簡単な方に逃げる」を避ける。典型的な逃げパターン:

- **逃げ A**: 曖昧な箱 (サービス、マネージャ、ワークフロー) への退避
- **逃げ B**: 数値的構造 (日付・金額・期間) の文字列化
- **逃げ C**: 直交する次元の排他列挙化
- **逃げ D**: 安易な抽象化への退避 (Rule of Three を破る)
- **逃げ E**: テストでの設計問題の吸収 (大量モック・E2E への退避)
- **逃げ F**: 言語の例外機構による検証エラーの表現 (Result/Either で表現すべき)

逃げざるを得ない場合は、判断記録 (ADR / 設計ドキュメント) に以下 3 点を残す:

1. なぜそうせざるを得ないのか
2. 本当にそれ以外の方法がないか検討した経緯
3. 将来この制約が外れたら見直すトリガー

## Why: なぜアーキテクチャ設計が重要か

アーキテクチャ設計の品質は、システムの長期的な保守性・拡張性・技術的負債の蓄積速度を決定する。不適切なアーキテクチャは、後から修正が極めて困難な構造的問題となる。

**根本的な理由**:
1. **変更容易性の決定**: アーキテクチャは変更の影響範囲を決定する。適切なレイヤー分離と境界定義により、変更が局所化される
2. **技術的負債の蓄積速度**: 明確な境界がないアーキテクチャでは、機能追加ごとに複雑さが指数関数的に増大する
3. **テスタビリティの確保**: 各レイヤーが独立してテスト可能なアーキテクチャは、テストコストを大幅に削減する
4. **技術品質の組み込み**: セキュリティ・オブザーバビリティ・障害耐性等は、アーキテクチャに組み込まれていなければ後付けが困難
5. **チームスケーラビリティ**: 明確な境界はチームが並行して開発できる範囲を広げる

## 判断プロセス

### ステップ1: 既存アーキテクチャの把握 (最優先)

既存アーキテクチャを無視した理想論は、移行コストと既存価値の破壊により失敗する。

**チェック項目**:
- ディレクトリ構造・モジュール分割の現状
- 採用されているレイヤー構成 (名前はプロジェクトごとに異なる)
- 依存関係の方向
- 既存の技術スタック・ADR
- リポジトリの用語集 (`docs/glossary.md` 等) の有無と内容

**判定フロー**:
```
既存プロジェクトか?
  No  → 新規設計へ
  Yes → 既存アーキテクチャを確認
       → 構造的な問題があるか?
            Yes → 段階的改善 or 刷新を ADR で記録
            No  → 既存構造を尊重して拡張
```

### ステップ2: レイヤーの定義 (営みの起源で分ける)

レイヤーを「Clean Architecture だから」「DDD だから」のような既成の枠組みで決めない。代わりに、各レイヤーが扱う関心事の **起源** で分ける。

| レイヤー | 起源 | 1 文定義 |
|---|---|---|
| domain | 紙の帳簿時代から存在した | アプリケーションになる前から、現実でその営みをする上で必要だったロジックを表現する |
| application | アプリにしたから生まれた | 現実の営みをアプリケーションにしたときに初めて発生する文脈 (永続化、Tx 境界、ログ、認可等) を引き受ける |
| infrastructure | 外界への接続 | domain と application が外界に求める能力 (ポート) の具体実装を提供する |
| interfaces | 外部プロトコル境界 | HTTP / gRPC / CLI 等のプロトコルとの境界。リクエスト/レスポンスの DTO 変換と認証ミドルウェア |

**重要な帰結**:

1. **リポジトリは domain ではなく application に置く** — 永続化は紙の帳簿時代には存在しなかった営み。「Clean Architecture の典型図ではリポジトリは domain」と無批判に当てはめない
2. **Clock や PasswordHasher のような「現実の概念のポート」は domain に置く** — 時刻も秘密の変換も紙の帳簿時代から存在した概念
3. **infrastructure は domain にも依存する** — Clock 実装が domain の `Clock` を実装するため

詳細は `resources/domain-modeling.md` の「営みの起源によるレイヤーの定義」を参照。

### ステップ3: ドメインロジックの置き場所の優先順位

ドメイン層の中で、ロジックをどこに置くかを以下の優先順位で判断する。**上から検討し、上のレベルで表現できないときだけ下に降りる**。

1. **値オブジェクト** — 単一の値が満たすべき制約。`tryNew` / `parse` が Result/Either を返す
2. **集約 / エンティティのメソッド** — 既存集約の状態に対する単純な操作。外部事実は引数で受け取る
3. **Input → Validator → Validated → Factory の型付きパイプライン** — 集約生成や複雑なミューテーション
4. **ドメインワークフロー** — 上記のいずれにも収まらず、本当に複数集約にまたがる不可分な判断が発生した場合のみ。多くのプロジェクトでは不要

「ドメインサービス」を最初に作るのは禁止 (語彙定義義務違反)。集約を貧血エンティティにするのも禁止 (不変条件を守る責務がなくなる)。

詳細とコード例 (4 役割の型付きパイプライン、外部事実の渡し方の 3 形式) は `resources/domain-modeling.md` を必ず参照。

### ステップ4: アプリケーション層の設計

アプリケーション層は「アプリにしたから生まれた」関心事を引き受ける。具体的責務クラスとして個別に表現する (「アプリケーションサービス」とまとめない)。

**典型的な具体的責務クラス**:
- `TransactionRunner` — 渡された手続きを 1 つのトランザクション境界で実行する
- `Authorizer` — プリンシパルとアクションを受け取り、許可/拒否を返す
- `RetryPolicy` — 失敗した手続きを規定の規則で再実行する
- `RequestLoggingDecorator` — ユースケース呼び出しの前後で構造化ログを出力する

**ユースケースは薄いグルー**:
- 入力 DTO を domain の入力型に詰め替える
- 具体的責務クラスと domain の Validator/Factory/集約メソッドを順に呼ぶ
- Result/Either を `?` / `flatMap` でつないで伝搬する
- 出力 DTO に詰め替えて返す
- 分岐・計算が出てきたら抽出のサイン (ドメインルールなら domain へ、アプリ固有なら具体的責務クラスへ)
- **ユースケース本体の単体テストは書かない** (ロジックがなくモック検証になるだけ)

詳細は `resources/application-layer.md` を必ず参照。

### ステップ5: CQRS の適用判断

書き込みと読み取りで要求が大きく異なる場合のみ適用する。

**判定フロー**:
```
書き込みと読み取りで要求が大きく異なるか?
  No  → CQRS 不要 (単一モデル)
  Yes → パフォーマンス要件が異なるか?
         Yes → CQRS 適用
         No  → 将来的に分離が必要になる見通しは?
                Yes → CQRS 適用を検討 (ADR で記録)
                No  → 現時点では不要
```

**段階的移行の余地**: アプリケーション層では Command / Query を別クラスとして配置しておくと、後から Query 側だけを別実装に差し替えられる。

### ステップ6: 境界の定義

コンテキスト境界と外部システムとの接点を明確に定義し、変更の波及を防ぐ。

- 異なるドメインコンテキストを特定し、境界ごとに独立したモデルを持つ (同じモデルを共有しない)
- 集約間の参照は **ID のみ** にして、トランザクション境界を集約単位に閉じる
- 外部 API・ライブラリとの接点に Anti-Corruption Layer 相当の薄いアダプタを置く
- 境界をまたぐデータ変換を一箇所にまとめる

### ステップ7: 技術選定

要件と制約、チームの状況を総合的に評価して選定する。

**選定基準**:
- 要件 (機能要件・非機能要件) を満たせるか
- チームの習熟度・学習コスト
- コミュニティの成熟度・長期サポート
- ライセンス・コスト制約
- 既存技術スタックとの整合性

トレードオフが発生した場合は ADR を作成し、選ばなかった選択肢とその理由も明記する。

### ステップ8: エラー境界の設計

レイヤー間でエラーを適切に変換し、ドメイン層に技術的詳細を漏らさない。

**原則**:
- **ユーザー入力・外界由来の失敗は Result/Either 型で表現する** (例外/panic で表現しない = 逃げ F の禁止)
- 例外/panic は「プログラマのバグを表す中断」専用。到達したらバグと言い切れる箇所にしか使わない
- ドメイン層のエラーはドメインの言葉で表現する (HTTP ステータスコードではない)
- infrastructure 層の例外がドメイン層まで漏れない設計にする
- レイヤー境界でのエラー変換ルールを定義する
- ユースケースが返すエラーの種類を列挙する

## 基本原則: 技術品質を設計段階から組み込む

技術品質 (テスタビリティ、オブザーバビリティ、セキュリティ、障害耐性等) は後付けではなく、アーキテクチャの設計対象として最初から扱う。

設計時に以下を明示的に検討する:
- **テスタビリティ**: 各レイヤーが独立してテスト可能な構造か
- **オブザーバビリティ**: ログ・メトリクス・トレースの取得ポイントはどこか (= application 層)
- **セキュリティ**: 信頼境界はどこか、認証認可の責務はどのレイヤーが持つか
- **障害耐性**: 外部サービス障害時の振る舞いをどう設計するか

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- レイヤー構成の選択 (Clean / Hexagonal / 簡易レイヤー等)
- ポート配置の判断 (特に Repository を application に置く判断)
- CQRS の適用・不適用の判断
- 技術選定で複数の候補からの選択
- 既存アーキテクチャからの変更判断
- 境界の引き方で複数の選択肢がある場合
- 「逃げ」を採用した判断 (上記 6 種のいずれかを意図的に採用したとき)
- 既定の禁止語 (サービス、マネージャ等) を構造的概念として使う判断

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **Resume Driven Architecture**: チームの習熟度を無視して流行の技術を採用する
- **Big Ball of Mud**: 明確なレイヤー・境界なしに機能を追加し続ける
- **Overengineering**: 将来の拡張性を過度に意識し、現在の要件に対して複雑すぎる設計にする
- **Cargo Cult Architecture**: 有名企業の事例 (Clean Architecture 典型図含む) をそのまま模倣し、自プロジェクトの文脈を無視する
- **Framework Coupling**: ドメイン層にフレームワーク固有の依存を持ち込む
- **「ドメインサービス」最初置き**: 値オブジェクト・集約メソッド・Validator/Factory パイプラインを検討せずに「ドメインサービス」に放り込む
- **貧血エンティティ**: 集約を getter/setter のみにして不変条件を守る責務をなくす
- **リポジトリ = domain**: Clean Architecture の典型図を雑に当てはめ、永続化が application の営みであることを意識しない
- **インスタンス化時例外**: コンストラクタで例外を投げて検証する (型レベルで「未検証 / 検証済み」を区別できなくなる)
- **総ロード地獄**: Validator にプレーンな値だけを渡す方針を画一的に適用し、データ量増加で全件ロードに退化する
