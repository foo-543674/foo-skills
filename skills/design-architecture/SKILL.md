# アーキテクチャ設計

要件をもとにシステムのアーキテクチャを設計する。

review-architecture の観点を設計時に先取りし、レビューで指摘されるような問題を事前に回避する。

## 基本原則

**技術品質を設計段階から組み込む**:

技術品質（テスタビリティ、オブザーバビリティ、セキュリティ、障害耐性等）は後付けではなく、アーキテクチャの設計対象として最初から扱う。これらは製品の機能ではないが、欠けたときの影響が広範囲に波及する。

設計時に以下を明示的に検討する:
- テスタビリティ: 各レイヤーが独立してテスト可能な構造か
- オブザーバビリティ: ログ・メトリクス・トレースの取得ポイントはどこか
- セキュリティ: 信頼境界はどこか、認証認可の責務はどのレイヤーが持つか
- 障害耐性: 外部サービス障害時の振る舞いをどう設計するか

## プロセス

### 1. 既存アーキテクチャの把握

新規プロジェクトでない限り、まず現在のアーキテクチャを理解する。

**確認すべきこと**:
- ディレクトリ構造・モジュール分割の現状
- 採用されているレイヤー構成（名前はプロジェクトごとに異なる）
- 依存関係の方向（どの層がどの層に依存しているか）
- 既存の技術スタック・フレームワーク
- 既存の設計判断の記録（ADR 等）

既存アーキテクチャを無視して理想のアーキテクチャを持ち込まない。既存の設計は過去に稼働し価値を生んできたもの。変更する場合は段階的に、合意を得て進める。

### 2. レイヤー構成の設計

変化の頻度と主導権に基づいてレイヤーを設計する。

**設計原則**:
- 変化の頻度が低い（安定した）層に向かって依存する
- ビジネスの主導権を持つ層が、技術的詳細に依存しない
- 各レイヤーの境界を抽象（インターフェース等）で定義する
- レイヤー間の結合点を最小限にする

**レイヤー設計の判断**:
- プロジェクトの複雑さに見合ったレイヤー数か（単純な CRUD にフルレイヤーは過剰）
- 各レイヤーの責務は明確か
- レイヤーをまたぐデータの受け渡し方法は決まっているか

### 3. CQRS の適用判断

Command（書き込み）と Query（読み取り）の分離が必要かを判断する。

**CQRS を適用すべき兆候**:
- 書き込みと読み取りで要求されるデータモデルが大きく異なる
- 読み取りのパフォーマンス要件が書き込みと異なる
- 複雑なビジネスルールを持つ書き込みと、多様な検索条件を持つ読み取りが共存する

**適用する場合の設計**:
- Command 側: クリーンアーキテクチャ等による適切なレイヤー分離、正規化されたモデル
- Query 側: 薄い構造、ストレージの能力を活かす設計、読み取り専用モデル

**適用しない場合**:
- 単純な CRUD で十分なコンテキストでは分離しない

この判断は ADR に記録する。

### 4. 境界の定義

コンテキスト境界と外部システムとの接点を設計する。

- 異なるドメインコンテキスト間の境界を明確にする
- 境界ごとに独立したモデルを持つ（同じモデルを共有しない）
- 外部 API・ライブラリとの接点に防腐層（Adapter / Anti-Corruption Layer）を設ける
- 境界をまたぐデータ変換を一箇所にまとめる

### 5. 技術選定

要件と制約に基づいて技術スタックを選定する。

**選定基準**:
- 要件（機能要件・非機能要件）を満たせるか
- チームの習熟度・学習コスト
- コミュニティの成熟度・長期サポート
- ライセンス・コスト制約
- 既存技術スタックとの整合性

技術選定でトレードオフが発生した場合は ADR を作成する。

### 6. エラー境界の設計

レイヤーをまたぐエラーの伝播方針を設計する。

- ドメイン層のエラーはドメインの言葉で表現する（HTTP ステータスコードではない）
- Infrastructure 層の例外がドメイン層まで漏れない設計にする
- レイヤー境界でのエラー変換ルールを定義する
- ユースケースが返すエラーの種類を列挙する

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- レイヤー構成の選択（Clean Architecture, Hexagonal, 簡易レイヤー等）
- CQRS の適用・不適用の判断
- 技術選定で複数の候補からの選択
- 既存アーキテクチャからの変更判断
- 境界の引き方で複数の選択肢がある場合

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **Resume Driven Architecture**: チームの習熟度を無視して流行の技術を採用する
- **Big Ball of Mud**: 明確なレイヤー・境界なしに機能を追加し続ける
- **Overengineering**: 将来の拡張性を過度に意識し、現在の要件に対して複雑すぎる設計にする
- **Cargo Cult Architecture**: 有名企業の事例をそのまま模倣し、自プロジェクトの文脈を無視する
- **Framework Coupling**: ドメイン層にフレームワーク固有の依存を持ち込む
