# 開発レポート生成

各開発フェーズ（要件定義・設計・実装）の成果物を1つのレポートに統合する。

review-report がコードレビューの結果を統合するのに対し、このスキルは開発プロセス全体の成果物と判断の記録を統合する。

## Why: なぜ開発レポートが重要か

開発レポートの品質は、意思決定の透明性・知識の継承・プロジェクトの説明責任を根本的に決定する。成果物が散在すると、後から判断の経緯が追えず、同じ議論を繰り返し、組織の学習が停滞する。

**根本的な理由**:
1. **意思決定の透明性**: 開発中に行った判断（技術選定、設計方針、スコープ決定等）の理由が記録されていないと、後から「なぜこうなっているのか」が分からず、変更が恐怖になる。レポートにより、判断の経緯・理由・トレードオフが明文化され、後続の開発者が安心して変更できる。判断の透明性がないと、誤った判断を修正できず、技術的負債が蓄積する
2. **知識の継承**: 開発プロセスで得た知見（要件の背景、設計の理由、実装の工夫等）が個人の頭の中にしかないと、担当者の異動・退職により知識が失われる。レポートにより、知識が明文化され、チーム全体で共有される。知識の継承がないと、同じ失敗を繰り返し、組織の学習が停滞する
3. **プロジェクトの説明責任**: ステークホルダーへの報告・他チームへの共有・経営層への説明において、開発の成果と判断を明確に示せないと、信頼を失う。レポートにより、何を作り、何を作らず、なぜそう判断したかが明確になり、説明責任が果たされる。説明責任が果たされないと、プロジェクトの価値が正しく評価されず、次のリソース配分で不利になる
4. **残課題の明示化**: 技術的負債・スコープ外の要件・改善提案が暗黙のまま放置されると、後続の開発で問題が顕在化し、対応コストが膨大になる。レポートにより、残課題が明示化され、計画的な対応が可能になる。残課題の明示化がないと、問題が蓄積し、プロジェクトが破綻する
5. **フェーズ間の整合性確認**: 要件・設計・実装が整合していないと、実装漏れ・過剰実装・仕様齟齬が発生する。レポート作成時に整合性を確認することで、問題を早期発見できる。整合性確認がないと、リリース後に仕様齟齬が発覚し、顧客への影響が発生する

**開発レポートの目的**:
- 意思決定を透明化し、後続開発者が安心して変更できるようにする
- 知識を継承し、組織の学習を加速する
- 説明責任を果たし、プロジェクトの価値を正しく評価させる
- 残課題を明示化し、計画的な対応を可能にする
- フェーズ間の整合性を確認し、仕様齟齬を早期発見する

## レポート形式

```markdown
# Development Report: [機能/プロジェクト名]

## Summary

- フェーズ: [完了したフェーズ]
- ADR: N件
- 実装ファイル: N件
- テスト: N件（全件パス / N件失敗）

## Requirements（要件定義フェーズ）

### 背景・目的
[機能の背景と目的の要約]

### スコープ
- **In**: [対応した項目]
- **Out**: [対応しなかった項目と理由]

### 主要な要件
1. [要件1の要約]
2. [要件2の要約]

### 受け入れ基準
- [ ] [基準1]
- [ ] [基準2]

## Design（設計フェーズ）

### アーキテクチャ
[採用したアーキテクチャの要約]

### データモデル
[設計したデータモデルの要約]

### API
[設計した API の要約]

### コンポーネント
[設計したコンポーネント構成の要約]

## Implementation（実装フェーズ）

### 実装したファイル
| ファイル | 変更種別 | 概要 |
|---------|---------|------|
| `path/to/file` | 新規/変更 | [概要] |

### テスト結果
- ユニットテスト: N件パス
- 統合テスト: N件パス
- カバレッジ: [該当する場合]

### テストファーストの実践記録
[Red-Green-Refactor のサイクル記録（主要なもの）]

## Decisions（意思決定の記録）

### ADR 一覧
| ADR | タイトル | ステータス |
|-----|---------|-----------|
| ADR-NNN | [タイトル] | Accepted |

### 主要なトレードオフ
- [判断1]: [選択肢A] vs [選択肢B] → [選択した理由]

## Quality（品質の達成状況）

### 技術品質
担保されていると暗黙的に期待される品質。欠けると広範囲に影響する。
- テスト: [カバレッジ状況、テスト戦略の実施状況]
- セキュリティ: [認証認可、入力バリデーション、既知の脆弱性対応]
- エラーハンドリング: [異常系の網羅状況、障害時の振る舞い]
- オブザーバビリティ: [ログ、メトリクス、トレースの実装状況]

### 製品品質
製品が生み出す価値。セールスポイント。
- [製品品質の観点1]: [達成状況]
- [製品品質の観点2]: [達成状況]

## Remaining Issues（残課題）

### 技術的負債
- [負債1]: [内容と影響]

### 未対応の要件（Out of Scope）
- [項目]: [将来の対応方針]

### 改善提案
- [提案1]: [内容と期待される効果]
```

## ルール

### レポートの出力先

レポートはリポジトリにコミットしない。プロジェクトの情報集約場所に出力する。

**出力先の判断**:
1. GitHub で管理されているプロジェクト → PR の description または関連 Issue のコメントに記載する
2. プロジェクト固有の情報集約場所（Wiki、Notion、Confluence 等）がある → そこに記載する
3. 上記いずれもない場合 → ユーザーに出力先を確認する

レポートをファイルとしてコミットするのは、プロジェクトの方針として明示的に求められた場合のみ。

### レポートの構成

- 完了したフェーズのみ記載する（設計フェーズまでなら Implementation は省略）
- 各フェーズの詳細は元の成果物を参照する形にし、レポートには要約を記載する
- ADR は一覧として参照可能にする

### Summary の書き方

- 完了したフェーズ、成果物の数、テスト結果を端的にまとめる
- 問題・残課題がある場合は Summary にも言及する

### Decisions の書き方

- ADR として記録された意思決定を一覧にする
- ADR に至らなかった軽微な判断は記載不要
- 主要なトレードオフは、選択肢と選択理由を簡潔に記載する

### Remaining Issues の書き方

- 技術的負債: 意識的に受け入れた妥協点
- 未対応の要件: スコープ外とした項目の将来方針
- 改善提案: 時間・スコープの制約でできなかったが、やるべきこと

### フェーズ間の整合性確認

レポート作成時に以下の整合性を確認する:

- 要件の受け入れ基準がすべて設計でカバーされているか
- 設計がすべて実装されているか
- 実装がテストでカバーされているか
- ADR の決定事項が設計・実装に反映されているか
- 技術品質（テスト、セキュリティ、エラーハンドリング、オブザーバビリティ）が先に作り込まれているか

整合性の問題が見つかった場合、レポートの Remaining Issues に記載する。

### レビュースキルとの関係

開発レポートはレビューの入力にもなる。レビュー実施時は review-report を使い、開発レポートの設計・実装内容に対してレビュー観点を適用する。
