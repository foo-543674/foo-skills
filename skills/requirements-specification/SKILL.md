# 要件の仕様化

引き出した要件を、実装者がテストを書ける粒度まで具体化する。

このスキルのゴールは「要件の完了をどう検証するか」を定義すること。受け入れ基準が書けない要件は、まだ十分に具体化されていない。

## Why: なぜ要件の仕様化が重要か

要件の仕様化の品質は、実装の正確性・テスト可能性・完了判定の明確さを根本的に決定する。受け入れ基準が曖昧だと、実装者の解釈が分かれ、テストが書けず、完了判定ができない。

**根本的な理由**:
1. **実装の正確性の担保**: 受け入れ基準が明確なら、実装者は「何を作れば完了か」を明確に理解できる。曖昧な要件では、実装者が推測で実装し、後から「これじゃない」と判明する。仕様化により、実装者の判断の余地を排除し、正確な実装を担保する
2. **テストファーストの実現**: 受け入れ基準が Given-When-Then で書かれていれば、実装前にテストを書ける。テストを書けない要件は、仕様が不十分な証拠。仕様化により、テストファースト開発が実現可能になり、品質が担保される
3. **完了判定の明確化**: 「動く」ことと「完了」は異なる。受け入れ基準がないと、実装者が「動いたから完了」と判断し、異常系・境界条件が未実装のまま放置される。仕様化により、完了の定義が明確になり、品質基準が統一される
4. **境界条件の明示化**: 曖昧な境界（「大量データ」「長い文字列」等）は、実装者ごとに解釈が異なり、エッジケースでバグが頻発する。仕様化により、境界を具体的な値で定義し、エッジケースのバグを防ぐ
5. **非機能要件の定量化**: 「速く」「安全に」のような定性的な非機能要件は、測定不可能で、実装後に「期待と違う」と判明する。仕様化により、非機能要件を定量的な基準に変換し、検証可能にする

**要件仕様化の目的**:
- 実装者の判断の余地を排除し、実装の正確性を担保する
- テストファースト開発を実現可能にし、品質を担保する
- 完了の定義を明確にし、品質基準を統一する
- 境界条件を明示化し、エッジケースのバグを防ぐ
- 非機能要件を定量化し、検証可能にする

## 判断プロセス（決定ツリー）

要件仕様化時は、以下の順序で判断する。受け入れ基準の明確化から始め、段階的に具体化する。

### ステップ1: 受け入れ基準の定義（最優先）

**判断基準**:
Given-When-Then形式で、完了を判定する基準を明確に定義する。受け入れ基準が書けない要件は、まだ十分に具体化されていない。

**チェック項目**:

1. **Given-When-Then形式**:
   ```
   Given [前提条件],
   When [ユーザーのアクション],
   Then [期待される結果].
   ```

2. **網羅性の確認**:
   - 正常系: 主要なユースケースパスをカバー
   - 代替フロー: 正常だが異なるパスをカバー
   - 異常系: 不正入力、権限不足、リソース不足等
   - 境界条件: 入力値の上限・下限、空、null

**判定フロー**:
```
要件ごとに受け入れ基準を定義
  ↓
Given-When-Thenで書けるか？
  ↓ No → 要件が不明確 → 要件引き出しに戻る
  ↓ Yes
  ↓
正常系・異常系・境界条件を網羅しているか？
  ↓ No → 不足しているケースを追加
  ↓ Yes → OK
```

**判定**: 受け入れ基準が書けない要件は仕様化不可

### ステップ2: 境界条件の明確化（高優先度）

**判断基準**:
曖昧さが残りやすい境界を具体的な値・条件で定義する。

**チェック項目**:

1. **明確化すべき項目**:
   - 入力値の許容範囲（最小値、最大値、文字数制限）
   - リストの要素数制限（0件の場合の振る舞い、上限）
   - 時間に関する境界（タイムゾーン、日付の切り替わり、有効期限）
   - 同時操作（同じリソースを複数ユーザーが操作した場合）
   - 権限境界（ロール別にできること・できないこと）

2. **振る舞いの明記**:
   - 各境界条件に対して「その値のとき、システムはどう振る舞うべきか」を明記

**判定フロー**:
```
境界条件を列挙
  ↓
各境界で振る舞いが明確か？
  ↓ No → 「大量データ」→「100万レコード」等、具体的な値に変換
  ↓ Yes → OK
```

**判定**: 曖昧な境界（「大量」「長い」等）は具体的な値に変換

### ステップ3: 非機能要件の定量化（高優先度）

**判断基準**:
定性的な非機能要件を、測定可能な基準に変換する。

**チェック項目**:

1. **定量化の例**:
   | 定性的 | 定量的 |
   |--------|--------|
   | 「速くレスポンスを返す」 | 「95パーセンタイルで200ms以内」 |
   | 「大量データに対応」 | 「100万レコードでリスト表示1秒以内」 |
   | 「高可用性」 | 「月間稼働率99.9%」 |
   | 「安全に」 | 「OWASP Top 10の各項目に対策済み」 |

2. **仮置きの許容**:
   - 定量化が難しい場合は、現時点での想定値と根拠を記録
   - 設計フェーズで検証・調整する前提で仮置きしてよい

**判定フロー**:
```
非機能要件を確認
  ↓
測定可能な基準になっているか？
  ↓ No → 定量的な基準に変換（仮置きでも可）
  ↓ Yes → OK
```

**判定**: 定性的な非機能要件は測定可能な基準に変換

### ステップ4: ユーザーストーリーの品質確認（中優先度）

**判断基準**:
INVEST原則に基づいて、ユーザーストーリーの品質を確認する。

**チェック項目**:

1. **INVEST原則**:
   - **Independent**: 他のストーリーと独立して実装・リリースできる
   - **Negotiable**: 実装方法は交渉・調整の余地がある
   - **Valuable**: ユーザーに価値を提供する
   - **Estimable**: 実装規模を見積もれる程度に具体的
   - **Small**: 1イテレーションで完了できるサイズ
   - **Testable**: 完了を検証できる

2. **分割の判断**:
   - 大きすぎるストーリーは分割する
   - 分割の軸: ワークフローのステップ、データバリエーション、操作の種類（CRUD）

**判定フロー**:
```
ユーザーストーリーを確認
  ↓
INVEST原則を満たすか？
  ↓ No (Smallでない) → ストーリーを分割
  ↓ No (Testableでない) → 受け入れ基準を追加
  ↓ Yes → OK
```

**判定**: INVEST原則に基づいてストーリーの品質を確認

### ステップ5: 画面・インタラクションの仕様（UIがある場合・低優先度）

**判断基準**:
画面の仕様をワイヤーフレームレベルで定義する。ビジュアルデザインは扱わない。

**チェック項目**:

1. **画面仕様の項目**:
   - 画面に表示する情報項目
   - ユーザーの操作手順（フロー）
   - 入力フォームのフィールドとバリデーションルール
   - エラー時の表示・メッセージ
   - 状態ごとの表示切り替え（ローディング、空状態、エラー状態）

2. **設計との境界**:
   - 仕様書では「何を」表示するかを定義
   - 「どう」表示するか（レイアウト、色等）は設計・デザイン段階で決定

**判定**: UIがある場合、情報と操作の仕様を定義（ビジュアルデザインは除く）

## プロセス

### 1. ユーザーストーリーの記述

要件をユーザー視点の物語として記述する。

```
As a [ユーザー種別],
I want to [達成したいこと],
so that [得られる価値・目的].
```

**記述の品質基準（INVEST）**:
- **Independent**: 他のストーリーと独立して実装・リリースできる
- **Negotiable**: 実装方法は交渉・調整の余地がある
- **Valuable**: ユーザーに価値を提供する
- **Estimable**: 実装規模を見積もれる程度に具体的
- **Small**: 1 イテレーションで完了できるサイズ
- **Testable**: 完了を検証できる

大きすぎるストーリーは分割する。分割の軸はワークフローのステップ、データバリエーション、操作の種類（CRUD）など。

### 2. 受け入れ基準の定義

各ユーザーストーリーに対して、完了を判定するための基準を定義する。

**Given-When-Then 形式**:
```
Given [前提条件],
When [ユーザーのアクション],
Then [期待される結果].
```

**受け入れ基準の網羅性チェック**:
- 正常系: 主要なユースケースパスをカバーしているか
- 代替フロー: 正常だが異なるパスをカバーしているか
- 異常系: 不正入力、権限不足、リソース不足等のケースがあるか
- 境界条件: 入力値の上限・下限、空、null のケースがあるか
- 状態遷移: 対象リソースの状態によって振る舞いが変わるケースがあるか

### 3. 境界条件の明確化

曖昧さが残りやすい境界を具体的な値・条件で定義する。

**明確化すべき項目**:
- 入力値の許容範囲（最小値、最大値、文字数制限）
- リストの要素数制限（0件の場合の振る舞い、上限）
- 時間に関する境界（タイムゾーン、日付の切り替わり、有効期限）
- 同時操作（同じリソースを複数ユーザーが操作した場合）
- 権限境界（ロール別にできること・できないこと）

各境界条件に対して「その値のとき、システムはどう振る舞うべきか」を明記する。

### 4. 非機能要件の定量化

定性的な非機能要件を、測定可能な基準に変換する。

| 定性的 | 定量的 |
|--------|--------|
| 「速くレスポンスを返す」 | 「95パーセンタイルで 200ms 以内」 |
| 「大量データに対応」 | 「100万レコードでリスト表示 1秒以内」 |
| 「高可用性」 | 「月間稼働率 99.9%」 |
| 「安全に」 | 「OWASP Top 10 の各項目に対策済み」 |

定量化が難しい場合は、現時点での想定値と、その根拠を記録する。設計フェーズで検証・調整する前提で仮置きしてよい。

### 5. 画面・インタラクションの仕様（UI がある場合）

画面の仕様をワイヤーフレームレベルで定義する。

- 画面に表示する情報項目
- ユーザーの操作手順（フロー）
- 入力フォームのフィールドとバリデーションルール
- エラー時の表示・メッセージ
- 状態ごとの表示切り替え（ローディング、空状態、エラー状態）

ビジュアルデザインはここでは扱わない。情報と操作の仕様に集中する。

## 成果物

```markdown
# 仕様書: [機能名]

## ユーザーストーリー

### US-1: [ストーリータイトル]
As a [ユーザー種別],
I want to [達成したいこと],
so that [得られる価値].

**受け入れ基準**:
- [ ] Given [前提], When [操作], Then [結果]
- [ ] Given [前提], When [操作], Then [結果]

**境界条件**:
- [項目]: [条件と振る舞い]

## 非機能要件

| 項目 | 基準 | 根拠 |
|------|------|------|
| [項目] | [定量的基準] | [根拠・前提] |

## 画面仕様（該当する場合）

### [画面名]
- 表示項目: [リスト]
- 操作: [ユーザーアクション]
- 状態: [状態ごとの表示]
```

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- 受け入れ基準の定義で、ビジネスルールの解釈に複数の選択肢がある場合
- 境界条件の設定で、ユーザビリティとシステム制約のトレードオフがある場合
- 非機能要件の水準設定で、コスト・開発工数とのトレードオフがある場合

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくある失敗パターン

- **Vague Acceptance Criteria**: 「正しく動作すること」のような検証不能な基準
- **Missing Edge Cases**: 正常系の基準だけ定義し、異常系・境界を放置
- **Premature Design**: 仕様書に実装方法やDB構造を書いてしまう
- **Unmeasurable NFR**: 「十分なパフォーマンス」のような定量化されていない非機能要件
- **Overspecification**: UI の色やレイアウトまで仕様に含め、デザインの柔軟性を奪う
