# 実装計画

設計済みのタスクを **(1) 俯瞰し** **(2) スコープを確定し** **(3) バーチカルスライスに分割した** 実装計画書を作成し、`.contexts/plans/<task-name>.md` に出力する。出力された計画書は実装エージェント（`implement-feature` 等）への引き継ぎパックとして機能する。

## Why: なぜ実装計画が必要か

**痛みの本質**: 計画なしで実装に入ると **「どこで区切ればいいかわからない」** ためビッグバン PR が量産され、フィードバックループが致命的に遅くなる。

これを防ぐために必要な3本柱:

### 1. 全体俯瞰

着手前に全体を一度俯瞰しないと、以下が **すべて** 起きる:

- 後半で「最初のあれも直さないと」が頻発し手戻りする
- 局所最適な設計を選んでしまい、全体として歪む
- 影響範囲を見誤って関係ない場所を壊す
- 実装エージェントが局所的なファイルしか見ず、既存パターンと不整合な実装をする

俯瞰はスキップ不可の前提工程である。

### 2. ゴール/スコープ事前確定（AC / DoD）

スコープを先に決めないと:

- **入口**: スコープクリープが起きる（やってるうちに「ついでにこれも」が積もる）
- **出口**: 完了判定ができず、いつまでもダラダラ続く

スコープには「これは対象外」と弾く **入口の防壁** と「ここまでやったら完了」と止める **出口の判定基準** の2つの役割がある。AC（Acceptance Criteria）と DoD（Definitions of Done）は本質的にこのスコープそのものである。

### 3. バーチカルスライス分割

レイヤー積み上げ（DB → リポジトリ → ユースケース → ハンドラ）ではなく **バーチカルスライス**（端から端まで貫通する縦切り）で分割する。

**Why**: フィードバックループを早くするため。プログラムの動きだけでなく、使いやすさ、UX、諸々を含めた **「ケーキの味」** を早く確かめたい。苺・スポンジ・クリームを個別に完璧に作っても、最後まで「ケーキの味」はわからない。**試験管サイズでいい、層が綺麗じゃなくていい、まず完成したケーキを一口味わう**。これがバーチカルスライスの本質である（トレーサーバレット / ウォーキングスケルトンの思想）。

**スライスの最小粒度に固定ルールはない**。極論「白紙ページが表示されるだけ」でも 1 ステップとして成立する。end-to-end で何かしら通っていればよい。

## 位置づけ

```
requirements-* → design-* → [plan-implementation] → implement-feature
                                ↑ このスキル
```

このスキルは **要件・設計が出揃っている前提** で動く。`requirements-elicitation` / `requirements-specification` / `design-*` の出力を消費し、実装スライス計画に変換する位置づけ。

## スコープ

**含む**:

- 設計済みタスクの俯瞰、AC / DoD の確定
- バーチカルスライス分割
- 各スライスへの参考メモ（実装エージェントの再調査コスト削減）

**含まない**:

- 要件定義そのもの（`requirements-*` の領域）
- 詳細設計（`design-*` の領域）
- 実装そのもの（`implement-feature` の領域）

## 絶対に守る品質ルール

以下のルールに違反する計画書を出力してはならない:

1. **「確かめたいこと」必須**: 各スライスは「このスライスで何を確かめたいか／どんなフィードバックが欲しいか」を必須項目として持つ。書けないスライスは存在を許さない。書けない場合は分割をやり直す。
2. **End-to-end 貫通必須**: 各スライスはたとえ白紙ページでも、入口から出口まで何かしら通っている必要がある。レイヤー単位で切るのを禁止する（「DB スキーマだけ」「リポジトリ層だけ」のスライスは禁止）。
3. **粒度はフィードバック駆動**: 粒度の主軸は「次に何を確かめたいか」。副軸は「不確実性の高さ」。固定の粗さ／細かさを押し付けない。
4. **参考メモは best-effort**: 各スライスの参考メモ（関連ファイル、想定手順、依存、リスク）は完璧でなくてよい。目的は「実装エージェントが0から再調査しなくて済む」ことであり、それを超えて深掘りしない。

## プロセス

### Step 0: 前提チェック（ガードレール）

実装計画の前提となる要件・設計ドキュメントが揃っているか確認する。

確認対象:

- `.contexts/` 配下に要件定義ドキュメント（`requirements*.md` 等）があるか
- `.contexts/` 配下に設計ドキュメント（`design*.md` / `architecture*.md` 等）があるか
- リポジトリ内に既存の関連実装があるか（Glob/Grep で大まかに把握）

**前提が揃っていない場合**: このスキルでの計画作成を中断し、ユーザーに以下を伝える:

```
実装計画の前提となるドキュメントが見つかりません。
先に以下のスキルを使って要件・設計を整理することを推奨します:

- 要件が曖昧な場合: /foo-skills:requirements-elicitation
- 受け入れ基準が未定義の場合: /foo-skills:requirements-specification
- 設計が未着手の場合: /foo-skills:design-architecture / design-component / design-api / design-data-model

設計なしで進めたい場合は、その旨を明示してください（その場合インタビューで補完します）。
```

ユーザーが「設計なしで進める」と明示した場合のみ、Step 1 以降に進む。

### Step 1: 既存コンテキスト読み込み（Claude 単独作業）

Claude が自力で以下を実行する。ユーザーには結果を共有するが質問しない。

1. `.contexts/` の関連ドキュメントを読み込む
2. 関連コードベースを Glob / Grep / Read で探索
3. 既存状態と要件のギャップを仮整理

完了したら、以下の形式でユーザーに調査結果を提示する:

```
【俯瞰調査結果】

■ 要件（既存ドキュメントから）
- ...

■ 価値（なぜこれをやるか）
- ...

■ 既存状態との差分
- 既にあるもの: ...
- 不足しているもの: ...
- 変更が必要なもの: ...

■ 影響範囲（コードベース探索結果）
- ...
```

### Step 2: 軽量インタビュー（対話）

Step 1 の調査結果を踏まえて、不明点だけを **1ターン1質問** で確認する。漠然質問は禁止。

確認すべき項目（該当するものだけ聞く）:

- **今回のタスクで一番達成したいこと**（複数の解釈があり得る場合のみ）
- **明示的に対象外にすること**（スコープ境界が曖昧な場合のみ）
- **既存のどのパターンに倣うか / 避けたい選択肢**（探索で複数候補が見つかった場合のみ）

各質問は **二択 + その他** の形式で行う。例:

```
このタスクの主目的は次のどちらですか？
- A. ユーザー体験の改善（レスポンスタイム短縮）
- B. 既存バグの修正（不正なレスポンス返却）
- その他（自由記述）
```

調査結果から自明な項目は **質問しない**。インタビューはあくまで補完用であり、ユーザーへの負荷を最小化する。

### Step 3: AC / DoD のドラフト提示と合意

調査結果とインタビュー結果を統合し、AC（Acceptance Criteria）と DoD（Definitions of Done）のドラフトを提示する。

**AC と DoD の違い（製品品質 vs 技術品質）**:

- **AC（Acceptance Criteria）= 製品品質の基準**: そのタスクの **バリュー・セールスポイント**。「あったら嬉しい」属性。ユーザーが意識的に語る部分。「このタスクで何ができるようになるか」をユーザー視点で書く。
  - 例: ユーザーがメールアドレスとパスワードでログインできる / 認証失敗時に 401 が返る / 有効な JWT で `/me` が 200 を返す
- **DoD（Definitions of Done）= 技術品質の基準**: そのタスクが当然担保されていると **暗黙的に認知されている** もの。「なかったら問題」属性。誰も口に出さないのに必須なもの。テスト・lint・型・ドキュメント・CI・オブザーバビリティ等。
  - 例: 単体テスト全パス / 統合テスト追加 / lint・format・型チェックパス / CI グリーン / 構造化ログ出力

**重要**: 製品品質（AC）はユーザーが語るが、技術品質（DoD）は **誰も口に出さないのに必須** である。このスキルの役割は、ユーザーが言わなくても DoD を先回りして埋めることにある。AC だけ書いて DoD を空にすることは禁止する。

提示形式:

```
■ Acceptance Criteria（案）
- [ ] ユーザーがメールアドレスとパスワードでログインできる
- [ ] 認証失敗時に 401 が返る
- [ ] 有効な JWT で /me が 200 を返す

■ Definitions of Done（案）
- [ ] 単体テストが全パス
- [ ] 統合テストが追加されている
- [ ] lint / format / 型チェックがパス
- [ ] CI がグリーン

これで進めてよろしいですか？追加・削除はありますか？
```

ユーザー承認を得てから Step 4 に進む。

### Step 4: バーチカルスライス分割案の提示と合意

AC / DoD を踏まえ、バーチカルスライスへの分割案を提示する。

**分割時の判断軸**（この順序で考える）:

1. **次に何を確かめたいか**（フィードバック駆動 = 主軸）
2. **不確実性の高さ**（前例のない技術スタックは細かく刻む / 慣れた技術は粗くてよい = 副軸）

**禁止事項**（再掲・絶対）:

- レイヤー単位の切り方（「DB だけ」「リポジトリ層だけ」など）
- 「確かめたいこと」を書けないスライスの提案

各スライスを以下の形式でドラフト提示する:

```
### Slice 1: 固定ユーザーでログイン疎通
**確かめたいこと**: ルーティング・認証ミドルウェア・JWT の基本構造が end-to-end で繋がること。DB やパスワードハッシュは後回し。
**完了条件**: 固定ユーザー (`alice@example.com` / `password`) で `/login` を叩くと固定の JWT が返り、その JWT で `/me` が 200 を返す。

### Slice 2: ...
```

ユーザーに以下を確認する:

- スライスの数と粒度感は適切か？
- 各スライスの「確かめたいこと」は本当に確かめたいことか？
- 順序は妥当か？

合意が取れたら Step 5 に進む。

### Step 5: 計画書ファイル出力

`.contexts/plans/<task-name>.md` に計画書を出力する。`.contexts/plans/` ディレクトリが存在しない場合は作成する。

`<task-name>` はタスク内容を表す kebab-case の短い名前（例: `user-auth`, `payment-refund-api`）。

各スライスに **参考メモ** を best-effort で追記する。Step 1 の調査結果から得られた情報を、対応するスライスに振り分ける形で記載する。完璧でなくてよい。

出力後、ユーザーに以下を伝える:

1. 計画書のパス
2. 計画書を確認・調整してから実装エージェントに渡せること
3. 実装は `/foo-skills:implement-feature` でスライス単位に進めるのが推奨されること

## 計画書テンプレート

```markdown
# Implementation Plan: <task-name>

## 1. 俯瞰

### 要件
[このタスクで実現したいこと。requirements-* ドキュメントの該当部分を要約・引用]

### 価値（Why）
[なぜこのタスクをやるのか。誰のどんな問題を解くのか]

### 既存状態との差分
- **既にあるもの**: [既存実装・既存ドキュメントのうち再利用するもの]
- **不足しているもの**: [新規に作る必要があるもの]
- **変更が必要なもの**: [既存だが手を入れる必要があるもの]

### 影響範囲
[このタスクが触る/壊す可能性のあるファイル・モジュール一覧]

## 2. Acceptance Criteria

- [ ] [機能的な受け入れ基準1]
- [ ] [機能的な受け入れ基準2]
- ...

## 3. Definitions of Done

- [ ] [技術品質基準1: テスト全パス等]
- [ ] [技術品質基準2: lint/format/型チェック等]
- [ ] [技術品質基準3: ドキュメント更新等]
- ...

## 4. バーチカルスライス分割

### Slice 1: <短い名前>

**確かめたいこと**:
[このスライスで何を確かめたいか。どんなフィードバックが欲しいか。必須項目]

**完了条件**:
[このスライスが完了したと判定する具体的な基準。動かして確認できる形で書く。必須項目]

#### 参考メモ（best-effort・実装エージェント向けの引き継ぎ情報）

- **スコープ**: [このスライスに含むもの／含まないもの]
- **関連ファイル**: [触る可能性が高いファイル・ディレクトリ]
- **想定手順**: [大まかな実装ステップ。詳細は実装エージェントが詰める]
- **依存**: [前のどのスライスに依存するか／なしなら「なし」]
- **リスク・不確実性**: [詰まりそうなポイント、検証したい仮説]

### Slice 2: <短い名前>

**確かめたいこと**: ...
**完了条件**: ...

#### 参考メモ
- ...

### Slice N: ...
```

## アンチパターン

以下は **やってはいけない例**:

❌ **レイヤー積み上げ分割**
```
Slice 1: DB スキーマとマイグレーション
Slice 2: リポジトリ層
Slice 3: ユースケース層
Slice 4: ハンドラとルーティング
```
→ 各スライスが end-to-end で動かない。最後の結合まで「ケーキの味」がわからない。バーチカルスライスの目的を完全に裏切っている。

❌ **「確かめたいこと」が書けないスライス**
```
Slice 1: 共通バリデーション関数群を作る
**確かめたいこと**: ???
```
→ 「確かめたいこと」が書けないということは、それは独立したスライスではない。他のスライスに統合するか、分割をやり直す。

❌ **網羅的な仕様書化**
```
Slice 1 の参考メモに、関数シグネチャ・型定義・エラーパターン・テストケース全列挙を 200 行書く
```
→ 参考メモは引き継ぎパックであり、仕様書ではない。実装エージェントの再調査コストを下げる目的を超えて深掘りしない。詳細は実装エージェントに任せる。

❌ **前提ドキュメント不在のまま強行**
```
.contexts/ に何もない状態で「とりあえず計画を作ります」と進める
```
→ Step 0 の前提チェックで requirements-* / design-* への誘導を出す。設計なしで進めたい場合はユーザーの明示的な意思表示を必須とする。

## 計画書出力後のフロー

1. ユーザーが計画書を確認・調整
2. 実装は `/foo-skills:implement-feature` で **スライス単位** に進める
3. スライスを1つ完了するたびに、計画書の該当スライスにチェックを入れる
4. 全スライス完了後、`/foo-skills:development-report` で開発レポートに統合する
