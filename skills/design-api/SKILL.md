# API 設計

要件をもとに API のインターフェースを設計する。

review-api-design の観点を設計時に先取りし、一貫性・冪等性・エラー設計等の問題を事前に回避する。

## このスキルのスコープ

**扱う** (このスキルで決めること):
- API スタイルの選択 (REST / GraphQL / RPC) と既存規約との整合
- エンドポイントの設計 (URL 構造、HTTP メソッド、命名規則)
- リクエスト・レスポンス DTO の形状 (フィールド、ネスト構造、命名)
- エラーレスポンスの統一形式
- バージョニング戦略
- 認証ヘッダ・クッキーのプロトコル仕様 (どこに何を載せるか)
- ページネーション・フィルタリング・ソートの仕様
- 冪等性キー・ETag 等のプロトコル機能の適用判断

**扱わない** (他のスキルに任せる):
- DTO を domain の入力型に詰め替えるロジックの配置層 → `design-architecture`
- 認証ミドルウェアの実装層 → `design-architecture`
- DB スキーマ・テーブル構造 → `design-data-model`
- フロントエンドの API クライアント設計 → `design-component`
- ユーザーから見たセマンティクス → `requirements-specification`

スコープ外の項目は **出力しない**。OpenAPI 定義やリクエスト例を書くのは OK だが、その上でどの層が DTO を詰め替えるかや、どのライブラリで認証を実装するかは書かない。低品質な出力は存在自体が害悪なので、書かない方がマシ。

## 基礎原則 (必読)

設計判断のすべてに以下 2 つの基礎原則を適用する。詳細は `../design-architecture/resources/principles.md` を必ず読むこと。

### 1. 語彙定義義務

エンドポイントの URL セグメント・パラメータ名・DTO フィールド名は「動詞 + 目的語」または「対象 + 性質」で 1 文定義できるようにする。`data` `info` `result` `status` のような曖昧な名前を使わない。

### 2. 逃げの禁止と説明責任 (API 設計での典型逃げ)

- **数値構造の文字列化**: 日付・金額・期間を JSON で文字列フィールドにする際、ISO 8601 などの規格に従わずに自前フォーマットで返さない。クライアント側で parse できる規格に従う
- **エラー形式のばらつき**: エンドポイントごとに異なるエラー形状を返さない。統一形式を ADR で確定させる
- **直交次元の単一化**: 独立した属性を `type` フィールドに押し込めない
- **「サービス」「マネージャ」エンドポイント**: `/api/userService/getUser` のような語彙定義義務違反の URL を作らない。リソース指向 (`/users/{id}`) かアクション指向 (`POST /users:archive`) を統一して使う
- **バージョニング先送り**: 「将来必要になったら入れる」と言って入れず、後から破壊的変更で苦しむ

逃げざるを得ない場合は判断記録に Why を 3 点 (なぜ / 代替検討 / 見直しトリガー) 残す。

## Why: なぜ API 設計が重要か

API 設計の品質は、クライアントとの契約の明確さとシステムの長期的な保守性を決定する。不適切な API 設計は、クライアントの開発効率を低下させ、後から修正が極めて困難な技術的負債となる。

**根本的な理由**:
1. **契約の明確性**: API はクライアントとサーバーの契約であり、一度公開すると後方互換性の維持が必須となる。曖昧な API 設計（エラーレスポンスが不統一、バリデーションルールが不明確等）は、クライアント側の実装を複雑にし、予期しない動作を招く。明確な契約なら、クライアントは安心して実装できる
2. **クライアント開発効率**: 一貫性のある API 設計（命名規則の統一、レスポンス構造の標準化等）は、クライアント側のコードを単純化し、学習コストを削減する。不統一な API は、エンドポイントごとに異なるエラーハンドリングが必要となり、開発効率が著しく低下する
3. **後方互換性の維持コスト**: 破壊的変更を含む API 更新は、すべてのクライアントの修正を強制し、調整コストが膨大になる。適切なバージョニング戦略と非破壊的な変更の優先により、クライアントへの影響を最小化できる。バージョニングなしの破壊的変更は、クライアントを一斉に停止させるリスクがある
4. **パフォーマンスとスケーラビリティ**: Chatty API（1 画面に大量のリクエストが必要）は、ネットワーク遅延を増幅し、サーバー負荷を増大させる。適切な粒度の API 設計（必要なデータを 1〜2 リクエストで取得）により、ユーザー体験とシステム効率が向上する
5. **セキュリティとビジネスロジック保護**: 内部実装の詳細が API に露出すると、攻撃対象が増え、セキュリティリスクが増大する。適切な抽象化により、内部構造を隠蔽し、ビジネスロジックの変更を API に影響させずに実施できる

**API 設計の目的**:
- クライアントとの明確な契約を確立し、予測可能な動作を保証する
- 一貫性のある設計でクライアント開発効率を最大化する
- 後方互換性を維持し、クライアントへの影響を最小化する
- 適切な粒度でパフォーマンスとスケーラビリティを確保する
- 内部実装を隠蔽し、セキュリティとビジネスロジックを保護する

## 判断プロセス（決定ツリー）

API 設計時は、以下の順序で判断する。契約の明確性とクライアント開発効率に直結する問題から確認し、設計を進める。

### ステップ1: API スタイルと既存規約の確認（最優先）

**判断基準**:
既存 API と一貫性がない設計は、クライアント側のコードを複雑にし、学習コストを増大させる。

**チェック項目**:

1. **既存 API スタイルの確認**:
   - REST / GraphQL / RPC のいずれか
   - 既存の命名規則（kebab-case / snake_case）
   - レスポンス構造の共通形式（envelope pattern 等）
   - エラーレスポンスの形式

2. **一貫性の維持 vs 改善の判断**:
   - 既存設計に問題がある場合、ADR で方針変更を記録
   - 新旧 API の共存方法を計画（バージョニング、段階的移行）

**判定フロー**:
```
既存 API を確認
  ↓
既存 API がない？
  ↓ Yes → スタイル選択（REST / GraphQL / RPC）を ADR で決定
  ↓ No
  ↓
既存 API と一貫性を保つか？
  ↓ Yes → 既存規約に従う
  ↓ No → ADR で方針変更を記録し、共存方法を計画
```

**判定**: 既存 API がある場合、一貫性を保つことを優先

### ステップ2: リソースとエンドポイントの設計（高優先度）

**判断基準**:
リソースの粒度とエンドポイント設計が、クライアントの使いやすさとパフォーマンスを決定する。

**チェック項目**:

1. **REST の場合**:
   - リソース名は複数形（`/users`, `/orders`）
   - ネストは 2 階層まで（`/users/{id}/orders`）
   - URL にアクション動詞を含めない（`POST /users`）
   - CRUD 以外の操作はサブリソースで表現（`POST /orders/{id}/cancellation`）

2. **RPC スタイルの場合**:
   - サービス名 + メソッド名で構成（`UserService.CreateUser`）
   - REST の規則を機械的に当てはめない

**判定**: REST を選択した場合、REST の規則に従う

### ステップ3: リクエスト・レスポンスの設計（高優先度）

**判断基準**:
入出力の明確さが、クライアントの実装コストを決定する。

**チェック項目**:

1. **リクエスト設計**:
   - パスパラメータ: リソースの識別
   - クエリパラメータ: フィルタ・ソート・ページネーション
   - リクエストボディ: リソースの作成・更新データ
   - バリデーションルールを明記

2. **レスポンス設計**:
   - プロジェクト共通の構造に合わせる
   - 不要なフィールドを返さない（over-fetching の回避）
   - 日時フォーマット、null の扱い、空配列 vs null を統一
   - リスト系にはページネーションを付ける

**判定**: 共通構造がある場合、それに従う

### ステップ4: エラーレスポンスの設計（高優先度）

**判断基準**:
エラーレスポンスの一貫性が、クライアントのエラーハンドリングの複雑さを決定する。

**チェック項目**:

1. **エラー構造の統一**:
   - プロジェクト全体で統一された構造
   - プログラム的に判別可能なエラーコード体系

2. **エラーの分類**:
   - バリデーションエラー: どのフィールドがなぜ不正か
   - 認証・認可エラー: 401 / 403 の使い分け
   - リソース不存在: 404
   - ビジネスルール違反: 409 / 422
   - サーバーエラー: 500（詳細を漏らさない）

**判定**: エラー構造はプロジェクト全体で統一必須

### ステップ5: 冪等性の設計（中優先度）

**判断基準**:
副作用を持つ操作の冪等性が、二重実行時の安全性を決定する。

**チェック項目**:

1. **HTTP メソッドごとの冪等性**:
   - GET: 副作用なし（冪等）
   - PUT / DELETE: 冪等になるよう設計
   - POST: 二重実行で実害がある操作に Idempotency Key を設計

2. **冪等性キーの適用範囲**:
   - 決済、注文作成等に絞る
   - 全ての POST に求めるのは過剰

**判定**: 冪等性キーの適用範囲は ADR で記録

### ステップ6: バージョニング戦略（中優先度）

**判断基準**:
バージョニング戦略が、将来の API 進化の柔軟性を決定する。

**チェック項目**:

1. **バージョニング方式**:
   - URL path / header / query のいずれか
   - 破壊的変更の定義（フィールド追加は非破壊、削除・型変更は破壊）
   - 旧バージョンの廃止プロセス

2. **バージョニング不要の判断**:
   - 内部 API でクライアントとサーバーが同時デプロイされる場合

**判定**: バージョニング方式は ADR で記録

## プロセス

### 1. 既存 API の把握

既存の API がある場合、現在の設計規約を理解する。

**確認すべきこと**:
- API スタイル（REST / GraphQL / RPC）
- URL 命名規則（kebab-case / snake_case）
- レスポンス構造の共通形式（envelope pattern 等）
- エラーレスポンスの形式
- 認証・認可の方式
- バージョニング方針
- ページネーション方式

既存の API と一貫性を保つ。設計規約を変更する場合は ADR に記録し、既存 API との共存方法を計画する。

### 2. リソースとエンドポイントの設計

要件から API で公開するリソースを特定し、エンドポイントを設計する。

**REST の場合**:
- リソース名は複数形（`/users`, `/orders`）
- ネストは 2 階層まで（`/users/{id}/orders`）。3 階層以上は要検討
- URL にアクション動詞を含めない（`POST /users` であって `/users/create` ではない）
- ただし、CRUD に収まらない操作（承認、キャンセル等）はサブリソースとして表現する（`POST /orders/{id}/cancellation`）

**RPC スタイルの場合**:
- アクション名がエンドポイント名になる。REST の規則を機械的に当てはめない
- サービス名 + メソッド名で構成（`UserService.CreateUser`）

### 3. リクエスト・レスポンスの設計

各エンドポイントの入出力を定義する。

**リクエスト設計**:
- パスパラメータ: リソースの識別に使う（`/users/{userId}`）
- クエリパラメータ: フィルタ・ソート・ページネーションに使う
- リクエストボディ: リソースの作成・更新データ
- バリデーションルールを明記する

**レスポンス設計**:
- プロジェクト共通の構造に合わせる（envelope pattern `{ data, meta, error }` 等）
- 不要なフィールドを返さない（over-fetching の回避）
- 日時フォーマット、null の扱い、空配列 vs null を統一する
- リスト系にはページネーションを付ける

### 4. エラーレスポンスの設計

エラーの種類と返却形式を設計する。

**設計原則**:
- エラーレスポンスの構造をプロジェクト全体で統一する
- クライアントがプログラム的にエラー種別を判別できるコード体系を用意する
- エラーメッセージはユーザーに表示可能なものと、デバッグ用の詳細を分ける
- 内部実装の詳細（スタックトレース、SQL エラー等）を外部に漏らさない

**エラーの分類**:
- バリデーションエラー: どのフィールドがなぜ不正か
- 認証・認可エラー: 401 / 403 の使い分け
- リソース不存在: 404
- ビジネスルール違反: 409 / 422
- サーバーエラー: 500（詳細を漏らさない）

### 5. 冪等性の設計

副作用を持つ操作の冪等性を設計する。

- GET: 副作用なし（冪等）
- PUT / DELETE: 冪等になるよう設計する
- POST: 二重実行で実害がある操作には Idempotency Key を設計する
- 全ての POST に冪等性キーを求めるのは過剰。決済、注文作成等に絞る

冪等性キーの設計判断は ADR に記録する。

### 6. バージョニング戦略

API の進化方針を決める。

- バージョニング方式（URL path / header / query）
- 破壊的変更の定義（フィールド追加は非破壊、フィールド削除・型変更は破壊）
- 旧バージョンの廃止プロセス

内部 API でクライアントとサーバーが同時デプロイされる場合、バージョニングは不要。

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- API スタイルの選択（REST / GraphQL / RPC）
- ページネーション方式の選択（offset vs cursor）
- バージョニング方式の選択
- 冪等性キーの適用範囲の決定
- 認証方式の選択（JWT / OAuth / API Key 等）

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **Chatty API**: 1 画面の表示に大量のエンドポイントを叩く必要がある
- **RPC in REST Clothing**: REST を名乗りつつ `POST /api/getUsers` のようなエンドポイント
- **Inconsistent Error Shape**: エンドポイントごとにエラーレスポンスの構造が異なる
- **Leaking Internal Model**: 内部の DB 構造がそのままレスポンスに露出する
- **Breaking Without Versioning**: バージョニングなしに破壊的変更を入れる
