# foo-skills リポジトリ

foo-543674 の設計思想・コードレビュー観点・実装方針を Claude Skills として定義・管理するリポジトリ。

## 根底思想

**開発とは創作である。**

このリポジトリで定義するスキルは、foo-543674 の創作活動を支援するためのものである。スキル作成時は、表面的な指示内容を実装するだけでなく、以下を深く理解する必要がある：

- **創作意図**: なぜその方針を採用するのか（Why）
- **判断プロセス**: どのように判断・選択するのか（How）
- **例外と文脈**: どういう状況で判断が変わるのか（Context）
- **暗黙の基準**: 言語化されていない判断軸・価値観

スキルは単なる「手順書」ではなく、**創作者の思考プロセスと判断基準を体系化したもの**である。

## リポジトリ構成

```
foo-skills/
├── .claude-plugin/
│   └── plugin.json               # プラグインマニフェスト
├── .claude/
│   ├── CLAUDE.md                  # プラグイン開発者向け（英語版）
│   └── CLAUDE.ja.md              # プラグイン開発者向け（このファイル）
├── CLAUDE.md                      # プラグイン利用者向け（スキルの全体像・使い方）
├── README.md                      # 人間向け（セットアップ手順など）
├── skills/                        # スキル本体
│   └── <skill-name>/
│       ├── SKILL.md               # エントリポイント（YAML frontmatter + 指示）
│       └── resources/             # 詳細なリファレンス、チェックリスト等
└── agents/                        # エージェント
    ├── <agent-name>.md            # エージェント定義（YAML frontmatter + system prompt）
    └── <agent-name>-resources/    # エージェント用リソース
```

## スキルの作成・編集ルール

### SKILL.md の形式

```yaml
---
name: skill-name
description: このスキルが何をするか、いつ使うか
---

# スキル名

指示内容をここに記述する。
```

- `name`: スキル名
- `description`: Claude が自動で関連性を判定するために使う説明文

### 守ること

- SKILL.md は 500 行以内に収める。詳細は `resources/` に分離する
- 1 スキル 1 フォルダ。目的が異なるものは別スキルにする
- スキルを追加・変更したら README.md のスキル一覧も更新する

## フィードバックの反映

foo-543674 からスキルの書き方や内容について指摘・修正を受けた場合、同じ指摘が繰り返されないように以下を必ず行う:

1. **対象スキルの修正**: 指摘内容をスキルに反映する
2. **このファイルの更新**: 指摘が他のスキルにも適用できる一般的なルールであれば、「守ること」や該当セクションにルールとして追記する
3. **既存スキルへの横展開**: 追加したルールに違反している既存スキルがあれば、それも合わせて修正する

## スキル作成時の暗黙知引き出しプロトコル

スキル作成依頼を受けた際、表面的な指示をそのまま実装してはならない。以下のプロトコルに従い、ユーザーの暗黙知を体系的に引き出す。

### 基本原則

**❌ 禁止行動:**
- 指示内容をそのまま実装する
- 抽象的・漠然とした質問をする（「どういう基準ですか？」等）
- 一般的なベストプラクティスで補完する
- 判断プロセスを推測で埋める
- AI最適化分析を省略する

**✅ 必須行動:**
- 7段階の構造化プロセスで暗黙知を引き出し、AI最適化する
- Why（目的・理由）を最優先で深掘りする
- 判断プロセスを言語化させる
- 対比・具体例で判断の分岐点を特定する
- AIが実行しやすい形に変換・最適化する

### 7段階のスキル作成プロセス

```
【プロセス全体像】
Phase 1-5: ユーザーから暗黙知を引き出す
  Phase 1: 表面的要求の整理
  Phase 2: Why（目的・理由）の深掘り
  Phase 3: 判断プロセスの言語化
  Phase 4: 対比・比較で判断の分岐点を特定
  Phase 5: 具体例でエッジケース・例外を発見

Phase 6: AIにとって最適な形に変換する（AI最適化分析）
  6.1: 判断フローの構造化
  6.2: 条件分岐の補完・明示化
  6.3: 実装手段の最適化
  6.4: スキル分割の提案
  6.5: パターン認識の最適化
  6.6: 横断的品質観点

Phase 7: 最終設計の提示とユーザー確認
```

#### Phase 1: 表面的要求の整理
```
【実行内容】
1. ユーザーの指示を受け取る
2. 指示の表面的な要素を構造化
   - 何を作るか（What）
   - どういう機能か（Function）
   - どういう制約があるか（Constraint）

【出力例】
「以下の内容でスキルを作成する依頼を受けました：
- 対象: commit-messageスキル
- 機能: prefixを付与、粒度を分ける
- 制約: （明示されていない）

以下、暗黙知を引き出すために段階的に確認します。」
```

#### Phase 2: Why（目的・理由）の深掘り【最優先】
```
【質問設計】
なぜその方針を採用するのか、最も深い目的を特定する。
表面的な理由だけでなく、Why-Becauseチェーンで掘り下げる。

【質問例】
❌ 悪い質問: 「なぜprefixを付けるんですか？」
   → 表面的な答えしか返ってこない

✅ 良い質問:
「prefixを付ける目的は何ですか？
- コミット履歴の検索性向上？
- 変更影響の可視化？
- CI/CDでの自動判定？
- チーム内のコミュニケーション？

それはなぜ重要ですか？（さらに深掘り）」

【重要】
Whyを理解すれば、判断基準も自然に導出できる。
目的が不明確なまま実装してはならない。
```

#### Phase 3: 判断プロセスの言語化【第二優先】
```
【質問設計】
ユーザーの頭の中の判断フローを言語化させる。
抽象的質問ではなく、具体的なステップを追う。

【質問例】
❌ 悪い質問: 「どういう基準でprefixを選びますか？」
   → 漠然としていて答えにくい

✅ 良い質問:
「新しいファイルが追加されたコミットを見た時、
prefixを決めるために何を最初に確認しますか？

→ ユーザー回答: 「ファイルの役割を見ます」

ファイルの役割はどうやって判断しますか？
具体的にどの情報を見ますか？

→ ユーザー回答: 「パスを見て、CI/CD関連なら...」

その次は何を確認しますか？」

【ゴール】
判断フローを完全に言語化：
1. ファイルパスを確認
2. CI/CD関連ディレクトリか判定
3. Yes → chore, No → 次のステップ
4. ...
```

#### Phase 4: 対比・比較で判断の分岐点を特定【第二優先】
```
【質問設計】
似たケースを対比させ、判断が分かれる分岐点を特定する。

【質問例】
❌ 悪い質問: 「例外ケースはありますか？」
   → ユーザーは思いつかない可能性が高い

✅ 良い質問:
「以下の2つのケースで、prefix判定は同じですか？
違う場合、何が判断の分かれ目ですか？

ケースA: `src/features/login.ts` 追加
ケースB: `.github/workflows/ci.yml` 追加

→ ユーザー回答で分岐点が明確になる

では、以下のケースはどうですか？
ケースC: `tests/login.test.ts` 追加
ケースD: `docs/README.md` 更新」

【ゴール】
判断の分岐軸を特定：
- ファイルの役割（機能 vs インフラ vs テスト vs ドキュメント）
- 変更の種類（追加 vs 更新 vs 削除）
- 影響範囲（ユーザー向け vs 開発者向け）
```

#### Phase 5: 具体例でエッジケース・例外を発見
```
【質問設計】
一般ルールに当てはまらないケースを探索し、
例外ルールや条件分岐を特定する。

【質問例】
✅ 効果的な質問:
「"ファイル追加 = feat" というルールに当てはまらない
ケースはありますか？それはどういう特徴を持っていますか？

以下のようなケースはどう判定しますか？
- 設定ファイルの追加（`.env.example`）
- ビルド設定の追加（`tsconfig.json`）
- 依存関係の追加（`package.json`更新）
- Git関連ファイル（`.gitignore`更新）」

【ゴール】
例外ルールの網羅的な特定：
- CI/CD関連 → chore
- 設定ファイル → chore
- ドキュメント → docs
- テスト → test
- etc.
```

#### Phase 6: AI最適化分析【必須】

```
【目的】
Phase 1-5で引き出した暗黙知を、「AIにとって最適な形」に変換する。
ユーザーの判断基準を、AIが正確に実行できる構造に最適化する。

【重要】
このフェーズを省略してはならない。
ユーザーの思考プロセスをそのまま実装するのではなく、
AI実行に最適化した形に変換することで、精度が大幅に向上する。
```

##### 6.1 判断フローの構造化【第一優先】

```
【分析観点】
Phase 3で言語化した判断プロセスを、AI実行に最適な構造に変換する。

【チェック項目】
✓ 曖昧さの完全排除
  - 「適切に」「必要に応じて」等の表現がないか
  - すべての判断基準が具体的に定義されているか

✓ 優先順位の明確化
  - 複数条件が競合する場合の優先順位を明記
  - 「ハード制約」（必須）と「ソフト制約」（推奨）の区別

✓ 決定木への構造化
  - 判断フローを明確な決定木として表現
  - 各ノードで「何を確認し、どう分岐するか」を明示
  - デッドエンド（判断不能）の扱いを定義

【提案テンプレート】
「Phase 3で言語化した判断プロセスについて、
AI実行のために以下の構造化を提案します：

■ 決定木フロー
```
1. [条件A] を確認
   └─ Yes → [アクションX]
   └─ No → 2へ
2. [条件B] を確認
   └─ Yes → [アクションY]
   └─ No → 3へ
3. [条件C] を確認
   └─ Yes → [アクションZ]
   └─ No → ユーザー確認
```

■ 優先順位ルール
競合時の優先順位：
1. [ハード制約1]（必須）
2. [ハード制約2]（必須）
3. [ソフト制約1]（推奨）

■ 曖昧さの解消
現在の表現「[曖昧な表現]」を
具体的基準「[明確な基準]」に変更することを提案します。

この構造化でよろしいでしょうか？」
```

##### 6.2 条件分岐の補完・明示化【第二優先】

```
【分析観点】
ユーザーが明示していない、しかしAIが必要とする条件分岐を提案する。

【チェック項目】
✓ ユーザーが気づいていない例外ケース
  - ドメイン知識から予測される例外
  - 類似システムで頻出する例外

✓ 境界ケースの判定基準
  - 判断が難しいケースの明確化
  - 「どちらとも言えない」場合の扱い

✓ デフォルト動作の定義
  - すべての条件に当てはまらない場合の動作
  - エラー時の安全なフォールバック

【提案テンプレート】
「現在の判断基準に、以下の条件分岐を追加することを提案します：

■ 未考慮の例外ケース
[ケース]: [提案される判定]
理由: [なぜこのケースが発生しうるか]

■ 境界ケースの扱い
[境界ケース]: [提案される判定基準]
例: [具体例]

■ デフォルト動作
すべての条件に当てはまらない場合: [提案される動作]

これらの追加は必要でしょうか？
または、これらのケースは実際には発生しないでしょうか？」
```

##### 6.3 実装手段の最適化【第三優先】

```
【分析観点】
引き出した暗黙知を実装する最適な手段を提案する。

【選択肢】
A. Skills（foo-skills/skills/）
B. Hooks（設定ファイルで自動起動）
C. CLAUDE.md（プロジェクト全体のルール）
D. Memory（auto memory）

【判断基準】
Skills を選択すべき場合:
✓ 複雑な判断プロセスがある
✓ 段階的な質問・確認が必要
✓ 長い説明・リファレンスが必要
✓ 特定のトリガーで明示的に呼び出す

Hooks を選択すべき場合:
✓ 特定のイベント（コミット前、PR作成時等）で自動実行
✓ 判断プロセスがシンプル
✓ ユーザーの介入なしに完結する

CLAUDE.md を選択すべき場合:
✓ プロジェクト全体に適用されるルール
✓ すべてのスキルで横断的に守るべき原則
✓ プロジェクトの設計思想・価値観

Memory を選択すべき場合:
✓ 一時的なプロジェクト固有の情報
✓ セッション間で共有したいパターン
✓ まだ確立していない試行的なルール

【提案テンプレート】
「今回のスキルの特性を分析した結果、
以下の実装手段を提案します：

■ 推奨実装手段: [Skills/Hooks/CLAUDE.md/Memory]

理由:
- [判断基準に基づく理由1]
- [判断基準に基づく理由2]

代替案:
もし[条件]であれば、[代替手段]も検討できます。

この実装手段でよろしいでしょうか？」
```

##### 6.4 スキル分割の提案【第四優先】

```
【分析観点】
1つのスキルとして実装すべきか、複数に分割すべきかを提案する。

【分割すべきシグナル】
✓ 複数の異なる責任を持っている
✓ トリガー条件が明確に異なる
✓ 再利用可能な部分がある
✓ 独立してテスト可能な部分がある

【統合すべきシグナル】
✓ 常に一緒に使われる
✓ 片方だけでは意味をなさない
✓ 分割すると理解が困難になる
✓ 分割のオーバーヘッドが大きい

【提案テンプレート】
「今回のスキルについて、分割を提案します：

■ 提案する分割
[スキル名1]: [責任範囲1]
[スキル名2]: [責任範囲2]

■ 分割の理由
- [単一責任原則の観点]
- [再利用性の観点]
- [トリガー条件の観点]

■ 分割後の連携
[スキル1]と[スキル2]の連携方法: [説明]

この分割案でよろしいでしょうか？
それとも、1つのスキルのままが良いでしょうか？」
```

##### 6.5 パターン認識の最適化【第五優先】

```
【分析観点】
AIがスキルを自動起動するための「トリガー条件」を最適化する。

【チェック項目】
✓ description の明確性
  - いつ使うべきかが明確に記述されているか
  - キーワードが適切に含まれているか

✓ Few-shot例示の充実
  - 代表的なユースケース3-5個を含める
  - エッジケースの例も1-2個含める

✓ トリガーキーワードの明示
  - どういう言葉・フレーズでスキルが起動すべきか
  - 誤起動を防ぐための除外キーワード

【提案テンプレート】
「AIがこのスキルを適切に起動できるよう、
以下の最適化を提案します：

■ description 案
```yaml
description: |
  [いつ使うか]の時に使用。
  キーワード: [keyword1], [keyword2], [keyword3]
  例: [具体的な起動例1]
```

■ Few-shot例示（SKILL.md内に含める）
代表的なケース:
1. [ケース1の説明と判定結果]
2. [ケース2の説明と判定結果]

エッジケース:
1. [エッジケース1の説明と判定結果]

この例示でよろしいでしょうか？」
```

不足している観点や、誤解している部分があれば教えてください。」

【重要】
ユーザー確認なしに実装を開始してはならない。
Phase 6のAI最適化提案について、ユーザーの同意を得てから実装する。
```

### 暗黙知の優先順位

スキル作成時に引き出すべき暗黙知の優先順位：

```
【第一優先】C: Why（目的・理由）
- なぜその方針を採用するのか
- それはなぜ重要なのか
- 最も深い目的は何か

【第二優先】B: コンテキスト（例外・条件分岐）
           A: 判定基準（判断ロジック）
- どういう状況で判断が変わるのか
- どのように判断・選択するのか
- 判断の分岐点は何か

【第三優先】D: 具体例（エッジケース）
- 一般ルールに当てはまらないケースは何か
- 境界線上のケースはどう判定するか

【第四優先】E: メンタルモデル（思考の全体像）
- 頭の中の判断フローはどうなっているか
- どういう順序で考えているか
```

### スキル設計時の必須要素

引き出した暗黙知を基に、以下の要素を必ずスキルに含める：

```
【必須要素】
✓ Why: なぜこのスキルが必要か、目的は何か
✓ 判定プロセス: 具体的な判断フロー（ステップバイステップ）
✓ 判断の分岐軸: どこで判断が分かれるか
✓ 例外ルール: 一般ルールに当てはまらないケース
✓ 具体例: 代表的なケースとエッジケース
✓ 境界条件: 判断が難しいケースの扱い

【推奨要素】
✓ アンチパターン: やってはいけないこと
✓ 判断に迷った時の対処法
✓ ユーザーへの確認タイミング
```

## 質問設計ガイドライン

暗黙知を引き出すための質問設計の原則。

### 効果的な質問形式の優先度

```
【第一優先】D: 判断プロセスの言語化
- 思考フロー全体を言語化させる
- 「何を最初に確認しますか？その次は？」
- 段階的に判断ステップを追う

【第二優先】B: 対比・比較による判断軸の発見
           A: 具体例ベースの質問
- 似たケースを対比させ、分岐点を特定
- 「ケースAとケースB、判定は同じですか？何が分かれ目ですか？」
- 具体的なシナリオで判定を確認
- 「このケースはどう判定しますか？その理由は？」
```

### 質問設計の良い例・悪い例

#### ❌ 避けるべき質問パターン

```
1. 抽象的・漠然とした質問
   - 「どういう基準ですか？」
   - 「どう判断しますか？」
   - 「ルールを教えてください」
   → ユーザーは答えにくく、表面的な答えしか返ってこない

2. Yes/Noで終わる質問
   - 「例外ケースはありますか？」
   - 「これでいいですか？」
   → 思考を深掘りできない

3. 誘導質問
   - 「一般的には〇〇ですが、そうしますか？」
   → ユーザーの判断ではなく、Claudeの判断になる

4. 一度に複数の論点を聞く質問
   - 「AとBとCについてどう思いますか？」
   → 焦点がぼやけ、暗黙知が引き出せない
```

#### ✅ 効果的な質問パターン

```
1. 判断プロセス言語化型（最優先）
   「[状況]の時、[判断]をするために
   何を最初に確認しますか？
   → [回答]
   その次は何を確認しますか？
   → [回答]
   それはどうやって判断しますか？」

   例:
   「新しいファイルが追加された時、prefixを決めるために
   何を最初に確認しますか？
   → ファイルパスを見ます
   ファイルパスのどの部分を見ますか？
   → .github/ ディレクトリかどうか
   .github/の場合はどうしますか？」

2. 対比・比較型（第二優先）
   「ケースAとケースB、判定は同じですか？
   違う場合、何が判断の分かれ目ですか？」

   例:
   「`src/feature.ts`の追加と
   `.github/workflows/ci.yml`の追加、
   prefix判定は同じですか？
   違う場合、何が判断の分かれ目ですか？」

3. 具体例ベース型（第二優先）
   「以下のケースで、どう判定しますか？
   その理由は何ですか？」

   例:
   「以下のケースで、どのprefixを選びますか？
   - `.github/workflows/ci.yml` 追加
   - `src/features/login.ts` 追加
   - `tests/login.test.ts` 追加
   - `docs/README.md` 更新
   それぞれの判定理由も教えてください。」

4. 反例探索型
   「[一般ルール]に当てはまらないケースはありますか？
   それはどういう特徴を持っていますか？」

   例:
   「"ファイル追加=feat"というルールに
   当てはまらないケースはありますか？
   それはどういう特徴を持っていますか？」

5. Why-Becauseチェーン型
   「[判断・方針]の理由は何ですか？
   → [回答]
   それはなぜ重要ですか？
   → [回答]
   さらに、それはなぜですか？」

   例:
   「CI/CD用ファイルをchoreにする理由は？
   → 機能追加ではないから
   なぜ機能追加とCI/CD設定を区別するのが重要ですか？
   → 変更履歴を見た時に...」
```

### 質問のタイミングと順序

```
【必須順序】
1. Why（目的）を最初に確認（Phase 2）
   → これがないと判断基準も導出できない

2. 判断プロセスの言語化（Phase 3）
   → 全体の流れを把握

3. 対比・具体例で分岐点を特定（Phase 4）
   → 判断の細部を明確化

4. 反例探索で例外を発見（Phase 5）
   → エッジケースの網羅

5. AI最適化分析の実施（Phase 6）
   → AIが実行しやすい形に変換
   → 構造化・補完・実装手段選択

6. 最終設計案の提示と確認（Phase 7）
   → 理解の正確性を検証
   → ユーザー承認を得る
```

## スキル品質基準

スキル作成完了時の品質チェック項目。

### 必須品質基準

```
【目的の明確性】
✓ Whyが明記されているか
✓ 最も深い目的まで言語化されているか
✓ 目的と実装が整合しているか

【判定プロセスの明示性】
✓ 判断フローがステップバイステップで記述されているか
✓ 各ステップで何を確認するか明記されているか
✓ 判断基準が具体的に記述されているか
✓ 曖昧さ・推測の余地がないか

【例外ルールの網羅性】
✓ 一般ルールに当てはまらないケースが列挙されているか
✓ 例外ルールの判定基準が明記されているか
✓ エッジケースの扱いが定義されているか

【判断の分岐軸の明確性】
✓ どこで判断が分かれるか明記されているか
✓ 分岐の条件が具体的に記述されているか
✓ 複数の分岐軸が整理されているか

【具体例の充実度】
✓ 代表的なケースの例が示されているか
✓ エッジケースの例が示されているか
✓ 判定が難しいケースの例が示されているか
```

### 品質チェックリスト

スキル作成完了時に以下を確認：

```
【Phase 1-5: 暗黙知引き出し】
□ Phase 1-5のプロセスを完遂したか
□ ユーザーの暗黙知を十分に引き出したか
□ Why（目的）を深掘りしたか
□ 判断プロセスを完全に言語化したか
□ 対比・具体例で分岐点を特定したか
□ 例外ルール・エッジケースを網羅したか

【Phase 6: AI最適化】
□ Phase 6のAI最適化分析を実施したか
□ 判断フローを決定木として構造化したか
□ 曖昧さを完全に排除したか
□ 優先順位・制約を明確化したか
□ 未考慮の例外ケースを補完したか
□ 最適な実装手段を提案したか
□ スキル分割の必要性を検討したか
□ トリガー条件を最適化したか
□ テストケースを定義したか
□ コンテキスト管理戦略を設計したか

【Phase 7: 最終確認】
□ ユーザーに最終設計案を確認したか
□ 必須品質基準をすべて満たしているか
□ 表面的な指示実装に留まっていないか
□ 創作者の思考プロセスが体系化されているか
□ AIが正確に実行できる形に最適化されているか
```

### アンチパターン検出

以下のパターンが見られる場合は品質不足：

```
❌ 判定基準が「適切に」「必要に応じて」等の曖昧表現
❌ 判断プロセスが「確認する」だけで具体性がない
❌ 例外ケースが「その他」で済まされている
❌ Whyが記述されていない、または表面的
❌ 一般的なベストプラクティスで補完している
❌ ユーザーの判断基準ではなくClaudeの判断基準
❌ 具体例がない、またはエッジケースが欠如
❌ 判断の分岐軸が不明確
```
